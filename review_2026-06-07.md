# Production-Readiness Review: `fdb-record-layer-go`

**Date:** 2026-06-07
**Context:** Candidate datastore for a SaaS control plane.
**Reviewed at:** commit `57600002`, branch `fix/groupby-over-join`.

---

## Bottom line

This is genuinely impressive engineering with a production-grade core, wrapped in a
not-yet-production-grade project. The wire-compat story is real and CI-validated against
the actual Java server. But the crash surface, the absence of observability, a
from-scratch FDB client with no escape hatch, and — bluntly — a bus factor of one with
no license and no releases mean the recommendation is: **do not bet a control plane on
it today without specific mitigations.**

Production readiness is about the *weakest* load-bearing link, and several are weak.

**Overall: 4.5 / 10 for "drop-in production control-plane store."**

## What's genuinely strong (don't undersell this)

- **Wire compatibility is CI-enforced, not aspirational.** `conformance/BUILD.bazel`
  builds the *real* Java Record Layer 4.11.1.0 server from Maven and runs 439 Go↔Java
  cross-validation specs against one FDB instance on **every PR** (`conformance_test` is
  in `//...`, not stress-tagged). Most projects claiming "wire compatible" cannot say
  this. If the control plane needs Go and Java services sharing a cluster, this is the
  right foundation.
- **The dangerous FDB correctness paths are right.** `commit_unknown_result` is handled
  with a dummy-transaction sync barrier and read/write conflict copy-back
  (`pkg/fdbgo/client/commitpath.go:19-98`, `transaction.go:1298-1324`) — the exact
  libfdb_c mechanism. `IsRetryable` (`fdb/error.go:436`) matches the C++ predicate
  exactly. RYW is a careful port of `ReadYourWrites`/`WriteMap`, not a naive overlay.
- **Online index build and schema evolution are production-shaped.** Resumable via
  `RangeSet`, adaptive throttling, correct `WRITE_ONLY→READABLE` lifecycle
  (`online_indexer.go`); evolution validator rejects field drops/renames/unsafe type
  changes and allows only safe widenings (`metadata_evolution_validator.go`).
- **Test discipline is real.** ~10.7k test entry points, real FDB via testcontainers
  (no mocks), differential fuzzing of the pure-Go client against libfdb_c. The numbers
  in the README check out or are understated.

## Blockers (must-fix before a control plane)

### 1. No LICENSE — legal blocker
`git ls-files | grep -i licen` → empty, yet README:281 links `[LICENSE]`. You cannot
legally deploy unlicensed third-party code, and this is an Apache-2.0 Java port so
attribution obligations apply. (If the company *is* birdayz, this is a 5-minute fix; it
is still a hard gate for any external adopter and for internal legal review.)

### 2. Crash surface = DoS
There is **no top-level `recover()` at the SQL driver boundary**
(`pkg/relational/core/embedded/connection.go:305,336` — `QueryContext`/`ExecContext`
call plan+execute with nothing catching panics). ~100 planner panics and several eval
panics (`streaming_cursors.go:218,254`, `flat_map_cursor.go:213`) propagate through
`database/sql` to crash the host. Worse, the pure-Go client's **long-lived network
goroutines have no `recover()`** (`pkg/fdbgo/transport/conn.go:265-267` read/write
loops, `client/database.go:498` topology monitor) — a single malformed frame or
invariant trip takes down the whole process, uncatchable by any caller. For a shared
control plane this is the #1 fix.

### 3. No observability
`StoreTimer` is in-memory `atomic.Int64` only — **no Prometheus/OTel export, no logging
interface anywhere** (`store_timer.go`; grep for prometheus/otel/slog in non-test →
nothing). There are no commit-conflict or retry counters even in-memory. You would fly
blind: no way to answer "why is this query slow" or "how many 1020 conflicts are we
taking." A control plane without metrics/logs is not operable.

### 4. Default retry/connection footguns
`FDBDatabase.Run(ctx, …)`'s `ctx` **never reaches the retry loop or backoff** (runs on
`context.Background()`, `fdb/database.go:108,197`), and the default retry limit is
**unlimited** — a hot conflict retries forever. `OpenWithConnectionString` /
`OpenDatabaseFromConfig` **block forever** on an unavailable cluster (bootstrap uses
`context.Background()`), unlike `OpenDatabase` which caps at 60s. There are also two
divergent retry implementations (`runner.go` is bounded and ctx-aware; `Run` doesn't use
it).

### 5. No `-race` on the concurrency-critical path
Main CI runs `bazelisk test //...` with no race detector (`ci.yml:41`); nightly runs
`-race` on only 5 hand-picked targets — **excluding all of `//pkg/relational/...`**
(plan cache, driver, planner), which is exactly what `database/sql` hammers from many
goroutines. The code looks correctly locked, but for a control plane that is
"unverified," not "safe."

### 6. Pure-Go client risk with no escape hatch
The Record Layer hard-imports `pkg/fdbgo/fdb` in 53 files; the Apple CGo binding appears
**only in tests**. So "the project supports the CGo binding" is misleading — if you use
the Record Layer, you use the from-scratch client, period. It is well-tested but young
and churned 40+ fixes in the week before review, and it once crashed the FDB *server*
(found and fixed, `CRASH_BUG.md`). There is no `Database`/`Transaction` interface to
swap in libfdb_c if you hit a divergence in production.

## Should-fix (before relying on it at scale)

- **Bus factor of one.** 3328/3329 commits by one person, 0 external contributors,
  AI-persona review "gates" are the same author. For the company, this means total
  dependence on one engineer's knowledge of a ~525k-LOC port. Mitigate: a second human
  owner, real `CONTRIBUTING.md`, and treat the differential + conformance suites as
  *your* merge gates in *your* fork.
- **No releases / semver / CHANGELOG.** `git tag` is empty; consumers get a
  `v0.0.0-<pseudo-version>`. Pin a commit, vendor it, and run the suites yourself. Push
  for a tagged `v0.x` with a stability statement.
- **Known-open correctness gaps in the SQL engine** (TODO.md): join-enumeration
  *row-count nondeterminism* on 3-way/arithmetic joins (line 54), latent aggregate-`NULL`
  paths on INSERT…SELECT and union-remap (lines 70/73/74/79), 0-row type-coercion
  misses. These are edge cases, but "different row count across runs" in a database layer
  is the scariest class — keep control-plane queries simple and pin them with your own
  tests.
- **Add `govulncheck`** to CI (absent entirely) and a `SECURITY.md`.
- **Reconcile the docs.** README's "Not yet supported: LEFT/RIGHT OUTER JOIN, subqueries,
  LIMIT" directly contradicts TODO.md/DIVERGENCES.md where those are implemented. The
  project's own `PRODUCTION_READINESS.md` admits this drift. You cannot assess what is
  stable when the docs disagree.

## Recommendation

**Don't put it on the critical write path of a bet-the-company control plane today.**
But there is a credible staged path:

1. **Strongly prefer the embedded record-layer API (CRUD + secondary indexes +
   transactions) over the SQL/`database/sql` engine.** The SQL engine is where the crash
   surface, the plan cache, and the open correctness gaps live. The record-layer
   KV/index core is the mature, conformance-validated part and is what a control plane
   actually needs.
2. **Mitigations that would move the answer toward yes:** add the SQL-boundary +
   network-goroutine `recover()`; add metrics/log hooks; set bounded default retries +
   propagate `ctx`; run `-race` + the conformance + differential suites in *your* CI as
   merge gates; pin a vendored commit; sort the license. Most of these are days, not
   months.
3. **If you need bet-the-company write durability now,** the honest move is to build the
   `Database`/`Transaction` abstraction so the record layer can run on Apple's libfdb_c,
   and keep the pure-Go client for read-heavy/non-critical paths where its 2-4x speed
   shines.
