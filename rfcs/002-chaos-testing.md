# RFC 002: Chaos Testing Framework

**Status:** Draft
**Date:** 2026-03-14

## Problem

1525 tests verify correctness in controlled, single-transaction, single-goroutine conditions. None of them:
- Inject faults (commit failures, retries, timeouts)
- Verify that the store matches a known-correct model after sequences of operations
- Run long enough to surface drift bugs (COUNT off-by-one after 50K ops)
- Test concurrent mutation + validation under contention

We need a **framework** — not a one-off tool — for defining expected behavior, injecting faults (random or targeted), and verifying the store always matches a simple, obviously-correct model.

## Core Idea: Model-Based Chaos Testing

The framework maintains an **in-memory model** — a simple, obviously-correct shadow of what the FDB store should contain. Every operation is applied to both the real store and the model. After each operation (or batch), the framework verifies they match.

The model IS the specification. It's so simple it can't be wrong. The real store has 15 index maintainers, split records, versionstamped keys, atomic mutations, cursor pagination. The model has maps and counters. If they disagree, the store has a bug.

**Two modes:**
- **Targeted**: Write a specific scenario with a specific fault → test a specific code path
- **Random**: Seeded PRNG generates operation sequences + fault injection → brute-force for problems

Both use the same model, same verification, same fault injection mechanism.

```
  concurrent.go         # Concurrent stress runner

  report.go             # JSON violation reports
  chaos_test.go         # Entry point (Bazel target, "chaos" tag)
  BUILD.bazel           # Generated by gazelle
```

Lives inside `pkg/recordlayer/chaos/` (not a separate top-level package) because it needs access to store internals for verification. Tagged `chaos` — never runs with `just test`.

## The Model

```go
type StoreModel struct {
    // Ground truth: what should be in the store
    Records map[string]*ModelRecord  // tupleKey(PK) → record

    // Derived: what indexes should contain (recomputed from Records)
    // Not stored — recomputed on Verify() from Records + metadata
    metadata *RecordMetaData
}

type ModelRecord struct {
    PrimaryKey tuple.Tuple
    TypeName   string
    Message    proto.Message
    Version    *FDBRecordVersion  // nil if versioning disabled
    Fields     map[string]any     // flattened field values for index computation
}
```

Model operations are trivially simple:

```go
func (m *StoreModel) Save(typeName string, pk tuple.Tuple, msg proto.Message) {
    m.Records[tupleKey(pk)] = &ModelRecord{
        PrimaryKey: pk, TypeName: typeName, Message: msg,
    }
}

func (m *StoreModel) Delete(pk tuple.Tuple) {
    delete(m.Records, tupleKey(pk))
}

func (m *StoreModel) DeleteAll() {
    m.Records = make(map[string]*ModelRecord)
}

func (m *StoreModel) Count(typeName string) int64 {
    n := int64(0)
    for _, r := range m.Records {
        if typeName == "" || r.TypeName == typeName {
            n++
        }
    }
    return n
}
```

No indexes, no splits, no versionstamps, no atomic mutations. Just a map. If the model says there are 42 records and the store's COUNT index says 43, the store has a bug.

## Verification: Model vs Store

`Verify()` opens a read-only transaction, reads the actual store state, and diffs against the model.

```go
func Verify(store *FDBRecordStore, model *StoreModel) []Violation {
    var violations []Violation

    // 1. Record existence: every model record exists in store, every store record exists in model
    // 2. Record content: stored bytes deserialize to model's proto message
    // 3. Index entries: for each READABLE index, compute expected entries from model records,
    //    scan actual entries, diff
    // 4. Aggregate indexes: COUNT/SUM/MIN_EVER/MAX_EVER values match model computation
    // 5. RANK ordering: rank(record) matches sorted position in model
    // 6. Record count: store.GetRecordCount() == len(model.Records)
    // 7. Versions: each record's stored version matches model's version
    // 8. Split integrity: raw key scan finds no orphan chunks

    return violations
}
```

Each check produces structured `Violation` values, not panics. A run can accumulate violations and report them all.

```go
type Violation struct {
    Invariant   string      // "index_consistency", "aggregate_drift", "record_missing", etc.
    Index       string      // which index (if applicable)
    PrimaryKey  tuple.Tuple // which record (if applicable)
    Expected    any         // what the model says
    Actual      any         // what the store has
    Operation   string      // which operation triggered this (if known)
    OpIndex     int         // operation sequence number
}
```

## Fault Injection

### Mechanism: ChaosTransactor

`fdb.Transactor` is an interface. Our `FDBDatabase.transactor` field holds one. We wrap it:

```go
type ChaosTransactor struct {
    inner  fdb.Transactor
    faults *FaultConfig
    rng    *rand.Rand
    log    *FaultLog  // records what was injected for reproducibility
}

func (c *ChaosTransactor) Transact(fn func(fdb.Transaction) (any, error)) (any, error) {
    // Pre-transaction faults
    if c.shouldInject(FaultTransactionTooOld) {
        c.log.Record(FaultTransactionTooOld)
        return nil, fdb.Error{Code: 1007}
    }

    result, err := c.inner.Transact(fn)

    // Post-commit faults
    if err == nil && c.shouldInject(FaultCommitUnknown) {
        // Commit SUCCEEDED, but we pretend it didn't.
        // Re-execute fn in a new transaction — this is exactly what
        // fdb.Transactor does on error 1021. Tests retry idempotency.
        c.log.Record(FaultCommitUnknown)
        return c.inner.Transact(fn)
    }

    if err == nil && c.shouldInject(FaultArtificialConflict) {
        c.log.Record(FaultArtificialConflict)
        return nil, fdb.Error{Code: 1020}
    }

    return result, err
}
```

This requires **zero changes to production code**. `FDBDatabase` already accepts an `fdb.Transactor`. We just swap in `ChaosTransactor`:

```go
func NewChaosDatabase(db fdb.Database, faults *FaultConfig, seed int64) *FDBDatabase {
    return &FDBDatabase{
        transactor: &ChaosTransactor{
            inner:  db,
            faults: faults,
            rng:    rand.New(rand.NewPCG(uint64(seed), 0)),
        },
        db: db,
    }
}
```

### Fault Catalog

Every fault maps to a real failure mode in production:

| Fault | FDB Error | What it tests |
|---|---|---|
| `FaultCommitUnknown` | 1021 | Retry after ambiguous commit. **The most important fault.** Tests idempotency — does a retry double-count in COUNT/SUM indexes? |
| `FaultTransactionTooOld` | 1007 | Operations that take too long per transaction. |
| `FaultConflict` | 1020 | Retry after conflict. Tests that retry with fresh state is correct. |
| `FaultDatabaseLocked` | 1038 | Retry after transient lock. |
| `FaultThrottled` | 1051 | Retry after throttling. |

### Fault Configuration

```go
type FaultConfig struct {
    // Per-fault probability (0.0 = never, 1.0 = always)
    Rates map[FaultType]float64
}

// Named presets
var (
    FaultsNone       = &FaultConfig{}  // pure stress, no injection
    FaultsRetryHeavy = &FaultConfig{Rates: map[FaultType]float64{
        FaultCommitUnknown: 0.05,  // 5% of commits "fail"
        FaultConflict:      0.03,  // 3% get conflict
    }}
    FaultsTimeout = &FaultConfig{Rates: map[FaultType]float64{
        FaultTransactionTooOld: 0.02,
    }}
    FaultsAll = &FaultConfig{Rates: map[FaultType]float64{
        FaultCommitUnknown:     0.02,
        FaultConflict:          0.02,
        FaultTransactionTooOld: 0.01,
        FaultDatabaseLocked:    0.005,
        FaultThrottled:         0.005,
    }}
)
```

### Specific Fault Injection

For targeted tests, inject a specific fault at a specific point:

```go
func TestCommitUnknownDoesNotDoubleCount(t *testing.T) {
    s := chaos.NewScenario(t, metadata, chaos.WithSeed(42))

    // Normal save — establishes baseline
    s.SaveRecord(order1)
    s.Verify()  // model: 1 record, COUNT=1

    // Inject commit-unknown on the NEXT transaction
    s.InjectOnce(chaos.FaultCommitUnknown)
    s.SaveRecord(order2)
    // ↑ commit succeeds, ChaosTransactor re-executes, store handles retry
    s.Verify()  // model: 2 records, COUNT=2 — if COUNT=3, we found the bug
}
```

`InjectOnce` tells the ChaosTransactor to fire the fault exactly once on the next `Transact()` call, then deactivate. Deterministic, no randomness involved.

### Fault Log (for reproducibility)

Every injected fault is logged with its sequence number:

```go
type FaultLog struct {
    entries []FaultLogEntry
}

type FaultLogEntry struct {
    OpIndex   int       // which operation in the sequence
    Fault     FaultType // what was injected
    Timestamp time.Time
}
```

The fault log is included in violation reports so an agent can replay the exact fault sequence.

## Mode 1: Targeted Scenarios

Write specific test cases that exercise known-risky paths:

```go
func TestSaveDeleteSaveSamePK(t *testing.T) {
    s := chaos.NewScenario(t, metadata, chaos.WithSeed(1))
    s.SaveRecord(order1)       // insert
    s.DeleteRecord(order1PK)   // delete
    s.SaveRecord(order1v2)     // re-insert same PK, different data
    s.Verify()                 // model has order1v2, all indexes correct
}

func TestDeleteAllWithPendingVersionMutations(t *testing.T) {
    s := chaos.NewScenario(t, metadata, chaos.WithSeed(2))
    s.SaveRecord(order1)  // queues version mutation
    s.SaveRecord(order2)  // queues another
    s.DeleteAllRecords()  // should clear version mutations too
    s.Verify()            // model is empty, store is empty, no orphan versions
}

func TestOnlineIndexBuildWithConcurrentWrites(t *testing.T) {
    s := chaos.NewScenario(t, metadata, chaos.WithSeed(3))
    // Save 100 records
    for i := 0; i < 100; i++ {
        s.SaveRecord(randomOrder(s.Rng, int64(i)))
    }
    // Add a new index and build it online
    s.AddIndex(newValueIndex)
    s.BuildIndexOnline(newValueIndex)
    // Save more records while index exists
    for i := 100; i < 150; i++ {
        s.SaveRecord(randomOrder(s.Rng, int64(i)))
    }
    s.Verify()  // all 150 records indexed correctly in new index
}
```

Each scenario is a regular Go test. Runs in seconds. Deterministic. Uses the model to verify correctness.

## Mode 2: Random (Brute-Force)

Seeded PRNG generates operation sequences. Same seed = same sequence = same faults = same result.

```go
func TestRandomChaos(t *testing.T) {
    chaos.RunRandom(t, chaos.RandomConfig{
        Metadata:     metadata,
        Seed:         time.Now().UnixNano(),  // or fixed for repro
        NumOps:       10000,
        Faults:       chaos.FaultsRetryHeavy,
        VerifyEvery:  100,  // verify model↔store every 100 ops
    })
}
```

Internally:

```go
func RunRandom(t *testing.T, cfg RandomConfig) {
    rng := rand.New(rand.NewPCG(uint64(cfg.Seed), 0))
    model := NewStoreModel(cfg.Metadata)
    db := NewChaosDatabase(realDB, cfg.Faults, cfg.Seed)

    for i := 0; i < cfg.NumOps; i++ {
        op := pickWeightedOp(rng)

        err := db.Run(ctx, func(rtx *FDBRecordContext) (any, error) {
            store := openStore(rtx, cfg.Metadata)
            return op.Execute(store, model, rng)
        })

        // FDB retryable errors are fine (ChaosTransactor injects them)
        // Non-retryable errors might be bugs
        if err != nil && !isExpectedError(err) {
            t.Fatalf("op %d (%s): unexpected error: %v", i, op.Name, err)
        }

        if i % cfg.VerifyEvery == 0 {
            violations := Verify(store, model)
            if len(violations) > 0 {
                report := buildReport(cfg.Seed, i, violations, faultLog)
                writeReport(cfg.ReportDir, report)
                t.Fatalf("violation at op %d (seed=%d): %v", i, cfg.Seed, violations[0])
            }
        }
    }
}
```

**Reproducibility**: when a random run fails, the output is:

```
violation at op 7231 (seed=1710412800): index_consistency: orphan entry in
"order_price_idx" at key [42], no record with PK [42]
```

Re-run with `--seed=1710412800 --max-ops=7232` to reproduce instantly.

### Built-in Operations (weighted random)

| Operation | Weight | Model effect |
|---|---|---|
| SaveRecord (new PK) | 25 | `model.Save(type, pk, msg)` |
| SaveRecord (existing PK, overwrite) | 15 | `model.Save(type, pk, newMsg)` |
| DeleteRecord (existing PK) | 15 | `model.Delete(pk)` |
| DeleteRecord (non-existent PK) | 5 | no-op (model unchanged) |
| LoadRecord (existing) | 10 | verify deserialized == model |
| LoadRecord (non-existent) | 5 | verify nil |
| ScanRecords (full) | 5 | verify result set == model.Records |
| ScanIndex (random range) | 5 | verify entries match model computation |
| EvaluateAggregate (COUNT/SUM) | 5 | verify value matches model computation |
| DeleteAllRecords | 1 | `model.DeleteAll()` |
| RebuildIndex | 2 | model unchanged (just revalidates) |
| SaveRecord (large, >100KB) | 5 | `model.Save()` — tests split path |
| SaveRecord with ErrorIfExists | 2 | verify correct error or success |

The operations and weights are built into the framework. This isn't a generic tool — it knows what a record store does.

### Record Generation

```go
func randomOrder(rng *rand.Rand, pk int64) *demopb.Order {
    return &demopb.Order{
        OrderId:  proto.Int64(pk),
        Price:    proto.Int32(rng.Int32N(10000)),
        Quantity: proto.Int32(rng.Int32N(100)),
        Flower: &demopb.Flower{
            Type:  proto.String(flowerTypes[rng.IntN(len(flowerTypes))]),
            Color: randomColor(rng).Enum(),
        },
        Tags: randomTags(rng),
    }
}

// Large record — crosses 100KB split threshold
func randomLargeOrder(rng *rand.Rand, pk int64) *demopb.Order {
    o := randomOrder(rng, pk)
    // Add a big payload tag to push past 100KB
    o.Tags = append(o.Tags, string(randomBytes(rng, 100_000+rng.IntN(100_000))))
    return o
}
```

Uses the actual conformance proto types (Order, Customer). No synthetic protos needed.

## Mode 3: Concurrent Stress

Multiple goroutines hammering the same store. Model can't track expected state here (concurrent mutations make it non-deterministic), so this mode uses snapshot-consistent invariant validation instead.

```go
func TestConcurrentChaos(t *testing.T) {
    chaos.RunConcurrent(t, chaos.ConcurrentConfig{
        Metadata:      metadata,
        Seed:          time.Now().UnixNano(),
        Workers:       10,
        Duration:      5 * time.Minute,
        Faults:        chaos.FaultsAll,
        ValidateEvery: 30 * time.Second,
    })
}
```

Workers share a `PKTracker` (mutex-protected known PKs). Validators take a snapshot transaction:

```go
func ValidateSnapshot(store *FDBRecordStore) []Violation {
    // Build model from actual store state (full scan)
    model := buildModelFromStore(store)

    // Then verify derived state (indexes, counts, versions) against that model
    return verifyDerived(store, model)
}
```

This catches: index orphans, aggregate drift, split corruption, version mismatches — anything where the store's derived state disagrees with its own primary data.

## Reproducibility

### Everything is seeded

```go
chaos.NewScenario(t, metadata, chaos.WithSeed(42))
chaos.RunRandom(t, chaos.RandomConfig{Seed: 42, ...})
chaos.RunConcurrent(t, chaos.ConcurrentConfig{Seed: 42, ...})
```

The seed controls:
- Operation sequence (which op, which PK, which field values)
- Fault injection (which transactions get faults)
- Record generation (field values, sizes)

For concurrent mode, each worker derives its RNG from `seed + workerID`. Not fully deterministic (goroutine scheduling is non-deterministic), but the operation sequence per worker is reproducible.

### Replay

On failure, the report contains the seed + op index. Replay:

```go
// Original run found violation at op 7231 with seed 42
chaos.RunRandom(t, chaos.RandomConfig{
    Seed:   42,
    NumOps: 7232,  // stop right after the violation
    Faults: chaos.FaultsRetryHeavy,
})
```

Same seed → same operations → same faults → same violation.

### Minimization

Once a violation is reproducible, binary search for the minimal sequence:

```go
// Automated: find shortest prefix that still triggers the violation
chaos.Minimize(t, chaos.RandomConfig{Seed: 42, NumOps: 7232, ...})
// Output: "minimal reproduction: ops 0, 1, 2, 7230, 7231 (5 ops)"
```

The framework tries subsequences, keeping only ops that are needed to trigger the violation. Produces a minimal targeted scenario that can become a regression test.

## Concrete Bug This Catches

The commit-unknown + COUNT index bug:

1. Worker saves record PK=42 (new insert)
2. `addRecordCount(+1)` → FDB atomic ADD +1 → count is now 1
3. Index entries written for PK=42
4. Commit **succeeds** but ChaosTransactor returns error 1021
5. `fdb.Transactor` retries with a fresh transaction
6. Worker saves record PK=42 again — BUT PK=42 already exists now
7. `SaveRecord` loads existing record → finds PK=42 → this is an UPDATE, not INSERT
8. `addRecordCount` should NOT increment (update path)
9. If our code correctly detects this: COUNT stays at 1 ✓
10. If our code doesn't: COUNT becomes 2, model says 1 → **violation detected**

This bug is invisible in unit tests (no retries) and invisible in stress tests (real commit-unknown is extremely rare). With 5% commit-unknown injection, the framework catches it within seconds.

## What's NOT in This Framework

- **FDB-internal fault injection** (disk failures, network partitions) — FDB's own simulation handles this. We test our layer, not FDB.
- **Generic property-based testing** — use `rapid` directly for that. This framework is specifically about record stores.
- **Performance benchmarking** — separate concern. This is about correctness.
- **Multi-store testing** — future extension if needed.
