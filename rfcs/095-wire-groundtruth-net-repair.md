# RFC-095: Repair the wire-layer ground-truth conformance net

**Status:** ACCEPTED — FDB-C++-dev ACK; Torvalds ACK after revision (history correction,
reply-direction restoration, zero-churn regen gate). Implemented; see Outcome at the bottom.
**Scope:** test/codegen-tooling only — zero production-code behavior change
**Found while:** production-readiness audit of the pure-Go FDB client (TODO.md
"fdbgo/wire: `TestPrecomputeSize_GetReadVersionRequest` never runs in CI and fails when run")

## Problem

The wire layer's claim to byte-identity with C++ `ObjectWriter` rests on ground-truth
test vectors generated by the C++ extractor (`cmd/fdb-schema-extract`). That net is
**silently dead or red on every axis**:

1. **No bazel data dep → silent skip in CI.** `//pkg/fdbgo/wire/types:types_test`
   does not declare `testdata.json` as `data`, so `TestGroundTruthMarshal` and
   `TestManualMarshal_GetKeyServerLocationsRequest_basic` hit the
   `t.Skipf("testdata.json not found")` path on every CI run. A no-skip-rule
   violation hiding the next two findings.

2. **`TestGroundTruthMarshal` is RED when it actually runs** — on *every*
   reply-carrying message (all 11 vectors). Root cause: C++ serialization of a
   `ReplyPromise` writes the endpoint token, and `getEndpoint()` assigns a **random
   UID** at serialization time (`fdbrpc/include/fdbrpc/fdbrpc.h:197-198`,
   `save(Ar&, const ReplyPromise<T>&)` → `value.getEndpoint().token`). The vectors
   bake that run-specific random token in; the Go builders use a zero token → 16
   mismatching bytes per message. The extractor even has the intended fix half-built:
   `getReplyToken()` exists (`main.cpp:1224`) and a comment says "emit the token hex
   alongside so Go sets the same bytes" — but `emitTestVector` never emits it.
   **Everything else is byte-identical**: offsets, vtables, total size all match, so
   this is vector-generation noise, not a serializer bug.

3. **`TestPrecomputeSize_GetReadVersionRequest` (`wire/serializer_test.go`) is red by
   construction.** It is a hand-walked duplicate of pass-1 of the serializer that
   **omits the 8-byte fake-root object** which the real pass allocates between the
   message object and the vtables (generated `MarshalFDB`,
   `getreadversionrequest_generated.go:179-182`; C++ `save_helper` wraps the root in
   `FakeRoot`, `flow/include/flow/flat_buffers.h`). Hence Go=160 vs C++=168. The
   production path is already pinned byte-exactly (including size) by
   `TestGroundTruthMarshal`; the hand model is a stale second copy of generated logic.

4. **`gkslr_manual_test.go` passes via a tolerance gate** — "≤16 byte mismatches →
   LAYOUT CORRECT". That is exactly the "early-return tolerance gate" design
   principle #9 forbids; with the token pinned (fix 2) the gate can and must become
   byte-identity.

5. **`wire/conformance_test.go` is an orphan of deleted infrastructure.**
   `TestGroundTruth_AllMessages`/`TestGroundTruth_ReadFields` read per-message vectors
   from `pkg/fdbgo/wire/testdata/`. That suite **used to run green**: 322 vectors were
   committed (76 of them `*Reply.json`) and commit `9c4618ac` records "321 messages
   parsed by Go Reader". Commit `fb840ebe` (April 2026, "single codegen path:
   cmd/fdb-schema-extract") then deleted the generator `cmd/fdb-wire-schema-generator/`
   **and the committed vectors**, but left the test behind — it has silently skipped
   ever since. The deletion's real coverage loss is the **reply-direction parse ground
   truth** — replies are the only direction the Go client *parses* in production. The
   justfile still carries two recipes (`wire-schema`, `wire-testvecs`) invoking the
   deleted `cmd/`, and `wire/BUILD.bazel` globs the nonexistent directory.

## Fix

1. **Extractor: deterministic reply token, emitted per vector.** Each request is
   built with a **fixed endpoint** — `req.reply = ReplyPromise<T>(Endpoint({},
   UID(fixed, fixed)))` with the low bit of `first()` clear (non-stream tokens clear
   `TOKEN_STREAM_FLAG`, `FlowTransport.actor.cpp:1901-1909`); the
   `ReplyPromise(const Endpoint&)` ctor (`fdbrpc.h:158`) pre-binds the endpoint so
   `getEndpoint()` returns it untouched instead of lazily assigning
   `deterministicRandom()->randomUniqueID()`. This makes `testdata.json` byte-stable
   across regens. `emitTestVector` additionally writes the token as a
   `"reply_token"` field in **buffer order** — UID serializes via
   `scalar_traits<UID>` (`flow/include/flow/IRandom.h:111-126`): native-endian
   `first()` at offset 0, `second()` at offset 8, i.e. LE bytes of each word on
   FDB's platforms (the current `getReplyToken`'s `%016llx` word-BE rendering is
   wrong for this and gets fixed). Regenerate via
   `bazelisk build //cmd/fdb-schema-extract:generate_wire_types`.

2. **`TestGroundTruthMarshal`: pin the token, require zero mismatches, round-trip.**
   Builders take the vector's token and set `ReplyPromise{Token: tok}`. Any byte
   mismatch is a failure (no tolerance). Additionally assert the request **read
   direction**: C++ bytes → Go `UnmarshalFDB` → `MarshalFDB` → byte-identical to the
   C++ bytes.

2b. **Restore reply-direction parse ground truth.** New extractor vectors for every
   reply type the client parses in production (`GetValueReply`, `GetKeyReply`,
   `GetKeyValuesReply`, `GetReadVersionReply`, `CommitID`, `WatchValueReply`,
   `GetKeyServerLocationsReply`, `SplitRangeReply`), each constructed in C++ with
   deliberate non-default field values, serialized by the real `ObjectWriter`. A new
   Go test parses each vector and asserts the deliberately-set **field values** (not
   just parse-without-crash, which is all the old 322-vector suite checked). Reply
   vectors need no token pinning — parse direction doesn't care that embedded
   endpoint tokens are random; they're opaque bytes the assertions don't touch.
   Honest scope statement: this does NOT restore the ~300 server-internal types the
   old suite smoke-parsed; the client never touches those. It restores — and
   strengthens, via field assertions — ground truth for everything the client
   actually parses.

3. **Delete `TestPrecomputeSize_GetReadVersionRequest`** (the whole
   `wire/serializer_test.go`; the file contains only this test). Justification: it
   tests a hand-transcribed, stale copy of generated logic, not production code; the
   production serializer (both passes) is pinned byte-exactly by ground truth. Fixing
   the hand model would just create a third copy of the same logic to keep in sync.

4. **`gkslr_manual_test.go`: write the pinned token, assert byte-identity.** The
   manual two-pass walkthrough stays (it exercises the `wire` primitives —
   `PrecomputeSize`/`WriteToBuffer` — independently of generated code, and unlike the
   deleted test it models the fake-root correctly); only the tolerance gate dies.

5. **Bazel + no-skip:** `types_test` gets `data = ["testdata.json"]`; the
   `t.Skipf("testdata.json not found")` paths become `t.Fatalf` — the file is a
   committed artifact with a declared dep; absence is a build bug, not a skip reason.
   Same for the second skip path: a vector with no Go builder / no field-assertion
   entry is `t.Fatalf`, not skip — vectors are emitted with a `"kind"`
   (`request`/`reply`) so each test dispatches its kind exhaustively and an
   unmatched name fails loudly.

6. **Delete the orphans:** `wire/conformance_test.go`, the `wire-schema` +
   `wire-testvecs` justfile recipes, the `testdata/**` glob in `wire/BUILD.bazel`.

## Wire-compat statement

No production bytes change. This is test + codegen-tooling work. Its *outcome* is the
proof, now actually running in CI, that the Go serializer's bytes are identical to
C++ `ObjectWriter` output for every vector-covered message — the property the client's
wire layer has been claiming without a live gate.

## Test plan / teeth

- Red→green: `TestGroundTruthMarshal` fails on the old vectors (random token) and on
  un-pinned builders; green only with token-pinned vectors + builders.
- Revert-proof: dropping the `reply_token` plumbing (builders back to zero token)
  reddens every reply-carrying subtest; re-introducing the ≤16 tolerance gate is
  detectable by mutation (set one non-token byte wrong → must fail).
- CI: `bazelisk test //pkg/fdbgo/wire/... --test_output=all` shows the ground-truth
  tests RUN (not SKIP) — verified in the PR by test log, and structurally guaranteed
  by `t.Fatalf` on missing data.
- Regen hygiene gate: re-running `generate_wire_types` must produce **zero
  `*_generated.go` churn** — only `testdata.json` changes. Any generated-code drift
  is a separate finding to disposition, not something to slip into this PR.
- Reply-vector teeth: each field assertion uses a literal expected value mirrored
  from the C++ constructor site (never re-derived from the Go parse), so a parser
  regression (e.g. the GetReadVersionReply slot-miscount class fixed in `fb840ebe`)
  reads the wrong slot and fails loudly.

## Outcome (implementation)

The repaired net caught **three real bugs** on its first run (plus one hygiene
refactor originally misclaimed as a fourth bug — see item 3, corrected after
Torvalds' mutation probe):

1. **Wire divergence (write path):** the generated marshal omitted the
   RelativeOffset for EMPTY vector-of-struct fields (`if len > 0` guard), where
   C++ writes the shared-empty-vector offset unconditionally
   (`flat_buffers.h:964` visitor write; `:1212/:1230` shared empty blob). Go
   commit-request bytes differed from libfdb_c whenever read/write-conflict or
   mutation vectors were empty. Fixed in the generator; pinned by the
   `CommitTransactionRequest_{empty,three_sets,3_system_keys}` vectors (red
   pre-fix).
2. **`GetRangeSplitPoints` returned zero points from every real reply:**
   `parseSplitRangeReply` treated `splitPoints` (a FlatBuffers offset-vector,
   `VectorRef<KeyRef>` default strategy) as a `VecSerStrategy::String` inline
   blob — `ReadBytes` landed on the count word. The only e2e test tolerated an
   empty result ("may be nil/empty, which is fine"), so this shipped green.
   Fixed with `wire.Reader.ReadVectorBytes` + an envelope-positioned walk;
   pinned by `SplitRangeReply_two_points` (red pre-fix) and the e2e now
   requires ≥1 in-range split point.
3. **NOT a bug — hygiene refactor (claim corrected).** `parseWaitMetricsReply`
   ran `UnmarshalFDB` on the full ErrorOr envelope, which *looks* like the
   same wrong-root class as bug 2 but is correct by layout: the ErrorOr union
   stores its value RelativeOffset at object offset 4, exactly where FakeRoot
   keeps field 0, so the root walk lands on the inner table. Torvalds proved
   it by mutation — reverting the change keeps `StorageMetrics_basic` green.
   The rewrite to the `ReadErrorOrInto`-positioned reader stands as hygiene
   (one canonical envelope walk instead of a layout coincidence), and the
   vector covers the canonical path, but it pins no bug.
4. **`parseCommitReply` read a conflict-shaped CommitID as success:** with
   `report_conflicting_keys` the proxy replies
   `CommitID{version: invalidVersion, conflictingKRIndices}` instead of an
   ErrorOr not_committed (`CommitProxyServer.actor.cpp:2448-2466`); C++ maps it
   to not_committed (`NativeAPI.actor.cpp:6653/:6726`), Go set
   `committedVersion = -1` and returned nil. Latent today (Go never sets the
   flag) but a wrong-shape-equals-success hazard. Fixed + revert-proven via
   `CommitID_conflict`.

Also found and fixed in the extractor: an out-of-bounds `std::string(lit, 24)`
read (literal is 22 chars) that baked a nondeterministic byte into the
`CommitTransactionRequest_3_system_keys` vector, and the
`VecSerStrategy::String` `push_back_deep` stale-`_cached_size` trap
(`Arena.h:889-893`) — the reply constructors use shallow `append`, which
invalidates the cache. The zero-churn regen gate surfaced the two hand-fixes
living in DO-NOT-EDIT generated files (KeyRangeRef swap-inversion, OOM cap);
both are now emitted by the generator, and `just generate-wire-types` gofmt's
its output so regens stay churn-free.
