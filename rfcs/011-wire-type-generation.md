# RFC 011: Wire Type Generation

## Status: Proposed

## Problem

Wire types in `pkg/fdbgo/wire/types/` are hand-written Go files that port C++ `serialize()` methods. This has caused bugs:

- **Wrong slot numbers**: `GetReadVersionReply` used slots 5,6,7,10 instead of 6,7,8,11 because the developer forgot that `Optional<Value>` consumes two vtable slots (type tag + value).
- **Wrong reader method**: `parseGetKeyServerLocationsReply` called `ReadBytes(0)` on a nested struct slot (should be `ReadNestedReader(0)`), misinterpreting a RelativeOffset as a length-prefixed blob.
- **Wrong wire type**: `Error.MarshalInto` used `WriteInt32` but the wire type is `uint16_t` (confirmed by ground-truth test vector).
- **Hardcoded slot numbers**: Most hand-written types use numeric literals instead of generated slot constants, making the code fragile and unauditable.

The C++ extractor (`cmd/fdb-schema-extract/`) already extracts vtables, slot constants, and field metadata from real FDB headers. But it only generates `vtables_generated.go` — the struct definitions and marshal/unmarshal methods are hand-written, disconnected from the generated constants.

## Design Principles

1. **C++ source is the spec.** FDB's wire format is defined by C++ `serialize()` methods and the `flat_buffers.h` type trait system. It is NOT an IDL.

2. **The C++ extractor is the single source of truth for wire layout.** Slot assignments, field types, nesting structure, reader/writer methods — all derived from C++, all generated. No guessing.

3. **Top performance.** All slot numbers, types, and nesting baked in at generation time. Generated code is a flat sequence of `r.ReadT(N)` calls — no runtime dispatch, no intermediate allocations, no helper functions (Go doesn't inline them past cost 80). The generated `UnmarshalFDB` must be a single function with all reads inlined.

4. **AI ports logic, not structure.** For the ~5% of types with conditional `serialize()` (branches, protocol version checks, pre/post-processing), the AI fills in a generated stub. The struct definition and slot constants are always generated — the AI can't get them wrong.

5. **Zero-allocation unmarshal.** Pair types flatten nesting at generation time: `ReadNestedReader(0)` then `ReadBytes(0)`/`ReadBytes(1)` directly into parent struct fields. No intermediate struct allocation for nested types.

## Architecture

### Two-file split per type

Every wire type produces two files:

```
wire/types/
  commitid_generated.go       # ALWAYS regenerated. DO NOT EDIT.
  commitid_custom.go          # Generated ONCE as stub. Developer fills in body.
                              # Only exists for types with custom serialize() logic.
```

**`_generated.go`** contains:
- Go struct definition with field names, types, and slot comments
- VTable constant
- VTable closure (if available)
- Slot constants
- File identifier constant
- For **simple types** (no branches in C++ serialize): full `UnmarshalFDB` and `MarshalFDB`
- For **custom types** (branches in C++ serialize): struct + constants only, NO methods

**`_custom.go`** contains (only for custom types):
- `UnmarshalFDB` and `MarshalFDB` method stubs with `panic("not implemented")`
- The actual C++ `serialize()` source inlined as a Go comment block
- Source file path and line number for the C++ type

`_generated.go` is always overwritten on re-run. `_custom.go` is generated once — the extractor checks if it exists and skips it.

### What the extractor generates

#### Simple type (no branches) — fully generated

```go
// commitid_generated.go — Code generated by fdb-schema-extract. DO NOT EDIT.
// Source: fdbclient/include/fdbclient/CommitProxyInterface.h:188
// C++ serialize: serializer(ar, version, txnBatchId, metadataVersion, conflictingKRIndices)
// FDB version: 7.3.75

package types

import "github.com/birdayz/fdb-record-layer-go/pkg/fdbgo/wire"

const CommitID_FileID uint32 = 14254927

var CommitID_VTable = wire.VTable{16, 24, 4, 20, 22, 12, 23, 16}

const (
    CommitIDSlotVersion              = 0  // int64, ReadInt64/WriteInt64
    CommitIDSlotTxnBatchId           = 1  // uint16, ReadUint16/WriteUint16
    CommitIDSlotMetadataVersion      = 2  // Optional<Value> tag, ReadUint8; value at slot 3, ReadBytes
    CommitIDSlotConflictingKRIndices = 4  // Optional<VectorRef<int>> tag, ReadUint8; value at slot 5, ReadBytes
)

type CommitID struct {
    Version              int64  // slot 0, Version (int64), ReadInt64
    TxnBatchId           uint16 // slot 1, uint16_t, ReadUint16
    HasMetadataVersion   bool   // slot 2, Optional<Value> tag
    MetadataVersion      []byte // slot 3, Optional<Value> value
    HasConflictingKR     bool   // slot 4, Optional<Standalone<VectorRef<int>>> tag
    ConflictingKRIndices []byte // slot 5, Optional<Standalone<VectorRef<int>>> value
}

func (m *CommitID) UnmarshalFDB(data []byte) error {
    r, err := wire.NewReader(data)
    if err != nil {
        return err
    }
    if r.FieldPresent(CommitIDSlotVersion) {
        m.Version = r.ReadInt64(CommitIDSlotVersion)
    }
    if r.FieldPresent(CommitIDSlotTxnBatchId) {
        m.TxnBatchId = r.ReadUint16(CommitIDSlotTxnBatchId)
    }
    if r.FieldPresent(CommitIDSlotMetadataVersion) && r.ReadUint8(CommitIDSlotMetadataVersion) > 0 {
        m.MetadataVersion = r.ReadBytes(CommitIDSlotMetadataVersion + 1)
        m.HasMetadataVersion = true
    }
    if r.FieldPresent(CommitIDSlotConflictingKRIndices) && r.ReadUint8(CommitIDSlotConflictingKRIndices) > 0 {
        m.ConflictingKRIndices = r.ReadBytes(CommitIDSlotConflictingKRIndices + 1)
        m.HasConflictingKR = true
    }
    return nil
}

func (m *CommitID) MarshalFDB() []byte {
    w := wire.NewWriter(nil)
    return w.WriteMessage(CommitID_FileID, CommitID_VTable, 8, func(obj *wire.ObjectWriter) {
        obj.WriteInt64(int(CommitID_VTable[CommitIDSlotVersion+2]), m.Version)
        obj.WriteUint16(int(CommitID_VTable[CommitIDSlotTxnBatchId+2]), m.TxnBatchId)
        // Optional fields: skip in generated marshal (needs conditional write logic)
    })
}
```

#### Pair/compound types — concrete flattened struct per instantiation

```go
// pair_keyrangeref_vectorss_generated.go — Code generated by fdb-schema-extract. DO NOT EDIT.
// Source: derived from GetKeyServerLocationsReply.results element type
// C++ type: std::pair<KeyRangeRef, std::vector<StorageServerInterface>>
// C++ serialize (via serializable_traits<std::pair>):
//   serializer(ar, p.first, p.second)
//     p.first = KeyRangeRef (serialize_member → nested struct)
//       KeyRangeRef has: begin (bytes, slot 0), end (bytes, slot 1)
//     p.second = std::vector<StorageServerInterface> (dynamic_size → vector)

package types

import "github.com/birdayz/fdb-record-layer-go/pkg/fdbgo/wire"

const (
    PairKRVSSSlotKeyRange = 0 // KeyRangeRef (nested struct), ReadNestedReader
    PairKRVSSSlotServers  = 1 // vector<StorageServerInterface>, ReadVectorCount

    // KeyRangeRef inner slots (inside the nested struct at slot 0)
    KeyRangeRefSlotBegin = 0 // bytes (KeyRef/StringRef), ReadBytes
    KeyRangeRefSlotEnd   = 1 // bytes (KeyRef/StringRef), ReadBytes
)

// PairKeyRangeVectorSS is one element of GetKeyServerLocationsReply.results.
// Fields are flattened from the nested pair<KeyRangeRef, vector<SS>> structure
// to avoid intermediate allocations.
type PairKeyRangeVectorSS struct {
    Begin   []byte // KeyRangeRef.begin — pair slot 0 (nested), inner slot 0
    End     []byte // KeyRangeRef.end — pair slot 0 (nested), inner slot 1
    Servers []byte // vector<StorageServerInterface> — pair slot 1 (raw bytes)
}

// UnmarshalFrom reads from a Reader positioned at one pair element.
// Flattens the nested KeyRangeRef into Begin/End fields directly — no
// intermediate allocation.
func (m *PairKeyRangeVectorSS) UnmarshalFrom(r *wire.Reader) error {
    // Slot 0: KeyRangeRef (nested struct)
    if krR, err := r.ReadNestedReader(PairKRVSSSlotKeyRange); err == nil {
        m.Begin = krR.ReadBytes(KeyRangeRefSlotBegin)
        m.End = krR.ReadBytes(KeyRangeRefSlotEnd)
    }
    // Slot 1: vector<StorageServerInterface> (raw bytes for now)
    if r.FieldPresent(PairKRVSSSlotServers) {
        m.Servers = r.ReadBytes(PairKRVSSSlotServers)
    }
    return nil
}
```

### Extractor configuration

The extractor needs to know which types are "custom" (have branches in C++ serialize). This is a static list maintained alongside the extractor:

```cpp
// In cmd/fdb-schema-extract/main.cpp or a config file
std::set<std::string> customTypes = {
    "CommitProxyInterface",    // isDeserializing post-processing
    "GrvProxyInterface",       // isDeserializing post-processing
    "StorageServerInterface",  // isDeserializing post-processing
    "GetReadVersionRequest",   // isDeserializing flag processing
    "ProtocolInfoReply",       // isSerializing/isDeserializing version conversion
    "DatabaseConfiguration",   // isDeserializing config parsing
};
```

For all other types: generate full struct + UnmarshalFDB + MarshalFDB.

The extractor can also auto-detect custom types by parsing the C++ serialize() method body for `isSerializing`, `isDeserializing`, `protocolVersion()` — these are the markers for conditional logic.

### Pair type discovery

The extractor discovers pair instantiations by examining field types in serialize() calls. When it encounters `std::vector<std::pair<A, B>>`, it:

1. Resolves the pair's serialize: `serializer(ar, p.first, p.second)`
2. Determines each member's trait:
   - `A` has `serialize()` → nested struct (ReadNestedReader)
   - `A` is `StringRef`/`KeyRef` → bytes (ReadBytes)
   - `A` is scalar → inline scalar (ReadInt32/etc.)
   - `B` is `std::vector<T>` → vector (ReadVectorCount)
3. For nested members, recursively resolves their inner fields
4. Emits a flattened Go struct with all nested reads inlined

### Generated file naming

```
{lowercase_typename}_generated.go    # always regenerated
{lowercase_typename}_custom.go       # generated once, never overwritten
```

For pair types:
```
pair_{first}_{second}_generated.go
```

### What changes from today

| Component | Before | After |
|-----------|--------|-------|
| Struct definition | Hand-written in `wire/types/*.go` | Generated in `*_generated.go` |
| Slot constants | Generated in `vtables_generated.go` | Generated in `*_generated.go` (per-type) |
| VTable constants | Generated in `vtables_generated.go` | Generated in `*_generated.go` (per-type) |
| UnmarshalFDB | Hand-written, slot numbers hardcoded | Generated with slot constants inlined |
| MarshalFDB | Hand-written | Generated for simple types, stubbed for custom |
| Pair types | Brute-force slot scanning in client code | Generated flattened structs in `wire/types/` |
| `vtables_generated.go` | Monolithic file with all types | Replaced by per-type `_generated.go` files |

### Performance characteristics

- **UnmarshalFDB**: All reads inlined in one function. No helper function calls (Go cost budget = 80, typical unmarshal cost = 150+, would NOT be inlined if split). Compiler can optimize register allocation and bounds checks across the entire function body.
- **MarshalFDB**: Same — all writes inlined in one function.
- **Pair types**: Nesting flattened at generation time. `ReadNestedReader` + `ReadBytes` calls directly into parent struct fields. No intermediate struct allocation.
- **Zero runtime metadata**: No `[]FieldDef` slices, no type switches, no interface dispatch. Every slot number is a compile-time constant.

### Migration path

1. **Phase 1**: Extend C++ extractor to emit `_generated.go` files for reply types we use: `CommitID`, `GetReadVersionReply`, `GetValueReply`, `GetKeyValuesReply`, `GetKeyReply`, `GetKeyServerLocationsReply`.
2. **Phase 2**: Emit pair types: `pair<KeyRangeRef, vector<SS>>`, `pair<UID, SS>`, `pair<UID, Tag>`.
3. **Phase 3**: Emit request types (move from hand-written to generated).
4. **Phase 4**: Emit `_custom.go` stubs for custom types. AI fills in method bodies.
5. **Phase 5**: Delete `vtables_generated.go` (replaced by per-type files). Delete hand-written type files.

Each phase is independently testable — the generated types must pass all existing integration tests against real FDB.

### C++ serialize patterns and how they map

| Pattern | C++ example | Generated? | Override mechanism |
|---------|-------------|------------|-------------------|
| Pure mechanical | `serializer(ar, f1, f2, f3)` | Full — struct + UnmarshalFDB + MarshalFDB | N/A |
| Optional fields | `Optional<Value> metadataVersion` | Full — two-slot pattern (tag + value) | N/A |
| Nested struct | `pair<KeyRangeRef, ...>` | Full — flattened, ReadNestedReader inlined | N/A |
| isDeserializing post-processing | Derive endpoints from base token | Struct + constants only | `_custom.go` stub with C++ source |
| isSerializing pre-processing | Convert ProtocolVersion to uint64 | Struct + constants only | `_custom.go` stub with C++ source |
| protocolVersion-gated fields | Different field sets per version | Struct + constants only | `_custom.go` stub with C++ source |

### Open questions

1. **Should `vtables_generated.go` be kept for backward compatibility during migration, or replaced immediately?** Replacing immediately is cleaner but requires updating all imports in one step.

2. **Should the extractor auto-detect custom types by parsing C++ serialize() for `isSerializing`/`isDeserializing`/`protocolVersion()`, or use a static list?** Auto-detection is more maintainable but requires a simple C++ parser in the extractor.

3. **For pair types, should the extractor generate a generic `UnmarshalFrom(*Reader)` or a type-specific `UnmarshalFromGetKeyServerLocationsResult(*Reader)`?** Generic is cleaner; type-specific avoids any ambiguity about which context the pair appears in.
