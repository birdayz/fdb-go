# RFC 012: Pure Go Client Performance

## Status: In Progress

## Baseline (2026-04-01, Ryzen 9 3900X, FDB 7.3.75)

| Metric | Pure Go | CGo (libfdb_c) | Ratio |
|---|---|---|---|
| GetValue ns/op | 1,347,000 | 212,000 | 6.4x slower |
| GetValue B/op | 4,200 | 383 | 11x more |
| GetValue allocs/op | 57 | 13 | 4.4x more |
| Marshal (GetValueRequest) ns/op | 321 | N/A (C++) | — |
| Marshal allocs/op | 2 | 0 (C++) | — |
| Unmarshal (GetValueReply) ns/op | 58 | N/A (C++) | — |
| Unmarshal allocs/op | 1 | 0 (C++) | — |

## CPU profile breakdown

- 36% — syscalls (TCP read/write)
- 32% — goroutine scheduling (park/unpark/findRunnable)
- 92% of wall time is WAITING, not computing

## Where the time goes

Each `Transact + Get` does two sequential TCP round trips:
1. **GRV**: send GetReadVersionRequest → wait → parse GetReadVersionReply
2. **GetValue**: send GetValueRequest → wait → parse GetValueReply

Each round trip: goroutine writes frame → syscall → TCP → FDB processes → TCP → syscall → readLoop goroutine wakes → channel send → calling goroutine wakes. Multiple goroutine context switches per RPC.

The CGo client uses `libfdb_c`'s C++ event loop: single-threaded, epoll-based, no goroutine scheduling overhead. Both GRV and GetValue go through the same optimized event loop.

## Optimization results

### Marshal path: 1,418 → 321 ns/op (4.4x faster, 26 → 2 allocs)

| Optimization | ns/op | allocs | Technique |
|---|---|---|---|
| Baseline (old) | 1,418 | 26 | Rebuild vtable set per message |
| + MessageTemplate | 824 | 18 | Pre-pack vtable closure at init, O(1) offset lookup |
| + ObjectWriter pool | 479 | 8 | sync.Pool for root ObjectWriter + backing slices |
| + Writer arena | **321** | **2** | Inline arena in pooled struct for nested ObjectWriters/nestedStructs |

**MessageTemplate**: vtable closure is static per message type. `NewMessageTemplate` pre-packs vtable bytes and builds an O(1) offset lookup map at init. `WriteMessagePacked` uses the template — zero vtable allocation, zero sorting, zero packing at runtime.

**ObjectWriter pool**: `sync.Pool` recycles the root ObjectWriter with pre-allocated backing slices (outOfLine, patches, nestedStructs). `clear()` + slice truncation on reuse.

**Writer arena**: Fixed-size inline arrays (`[8]ObjectWriter`, `[8]nestedStruct`, `[512]byte`) embedded in the pooled struct. `WriteStruct` slices from the arena instead of heap-allocating. Falls back to heap if capacity exceeded (never happens for current FDB message types — max 4 nested structs).

**Remaining 2 allocs**: output buffer (`w.buf` — could be eliminated by pooling the Writer) and one OOL slice grow (could be eliminated by pre-sizing the OOL buffer based on input data length).

### Unmarshal path: 58 ns/op, 1 alloc — already fast

Wire unmarshal is 0.004% of the total Get latency. Generated by C++ extractor, all reads inlined in one function, uses pre-computed slot constants. Not a bottleneck.

### End-to-end: still 6.4x slower (marshal optimization doesn't move the needle)

Marshal optimization saved ~1µs per request. Total Get is ~1,347µs. The 1.1ms gap is entirely network latency + goroutine scheduling — not serialization.

## Trade-offs analyzed

### Write path: arena lifecycle

The arena is safe because the lifecycle is fully contained:

```
Pool.Get → writeFields fills arena → assemble into w.buf → w.buf to net.Conn → Pool.Put
```

No reference to arena memory escapes. Output buffer is a standalone copy. By the time `Put` recycles the arena, bytes are on the wire.

### Read path: frame buffer ownership

```
ReadFrame: make([]byte, payloadLen)    ← one allocation
UnmarshalFDB: reply.Value = data[x:y] ← sub-slice, zero-copy
return reply.Value                     ← user holds reference into frame buffer
```

The frame buffer CAN'T be pooled — the user's `[]byte` aliases it. This is the one unavoidable allocation per read. The CGo client has the same cost (copies value from C memory to Go slice — one alloc).

**Alternative: pool frame buffers, force user to copy**. A `ReadInto` API where the frame buffer is pooled and the user must copy values out before the buffer is recycled. Saves the frame buffer alloc but adds a copy alloc — net zero. More complex API for no gain. Not recommended unless we add a streaming/scan API where the user processes values in-place without retaining them.

**Reply structs don't escape**: `var reply GetValueReply` is stack-allocated. `UnmarshalFDB` fills its fields. Only the `Value` sub-slice escapes. No pool needed for reply structs.

**Reader struct (96 bytes)**: allocated by `NewReader`, escapes to heap. Could be eliminated with a `Reset` pattern (`var r Reader; r.Reset(data)`). Saves 96 bytes per parse on a 1.3ms path. Not worth the API change.

## Remaining optimization opportunities (by impact)

### CRITICAL — Reduce round trips (the actual bottleneck)

| Optimization | Expected impact | Complexity |
|---|---|---|
| **GRV caching** | ~2x faster (1 RT instead of 2) | Medium — cache version for N ms, invalidate on conflict |
| **Pipelined GRV + read** | ~2x faster | High — deferred read version, architectural change |
| **Batched writes** | Reduce syscalls | Medium — bufio.Writer for frame coalescing |

**GRV caching**: The CGo client caches the read version across transactions within a batch window. Each sequential `Transact` currently does a fresh GRV round trip. Caching the version from the previous transaction (with a short TTL, e.g., 5ms) would eliminate the GRV round trip for sequential workloads. This is the single biggest win available — potentially halving latency from ~1.3ms to ~0.7ms.

**Pipelined GRV + read**: Send the GRV request, don't wait. Send the GetValue request with a deferred read version that resolves when GRV completes. Both requests fly in parallel. Requires changing `Transaction` to support lazy read version resolution. The C++ client does this internally.

### HIGH — Reduce per-RPC overhead

| Optimization | Expected impact | Complexity |
|---|---|---|
| **Eliminate context.WithDeadline per RPC** | -211 allocs/s, -4µs | Low — use parent context |
| **Pool reply channels** | -119 allocs/s | Low — sync.Pool for chan Response |
| **Pool frame read buffers** | -112 allocs/s (if user copies) | Medium — requires API consideration |
| **Pool timers** | -80 allocs/s | Low — timer.Reset reuse |
| **Batch TCP writes** | Fewer syscalls | Medium — frame coalescing |

### MEDIUM — Reduce goroutine scheduling

The readLoop goroutine + channel dispatch adds context switches per RPC. The C++ client avoids this with a single-threaded event loop.

Options:
- **io_uring** — Linux async I/O, avoids goroutine park/unpark for reads. Experimental Go support.
- **Batched channel operations** — reduce the number of goroutine wake/park cycles.
- **Lock-free response dispatch** — replace channel with atomic CAS on a response slot.

These are diminishing returns compared to reducing round trips.

### LOW — Marshal path (already optimized)

| Optimization | Expected impact | Complexity |
|---|---|---|
| Pool Writer (reuse w.buf) | -1 alloc | Low |
| Pre-size OOL buffer from key length | -1 alloc | Low |
| Convert remaining request types to templates | -5 allocs per GRV/location/commit | Low |

## Architecture comparison: Go vs C++ event loop

The fundamental performance gap comes from the execution model:

**C++ (libfdb_c)**: Single-threaded actor model. All I/O through epoll. Request → queue → event loop sends → epoll waits → response arrives → callback fires. Zero context switches. One thread handles thousands of concurrent operations.

**Go (our client)**: Goroutine-per-request model. Request → goroutine writes to conn → goroutine parks → readLoop goroutine reads response → channel send → original goroutine wakes. Two goroutine context switches per RPC. Go's scheduler adds ~1-5µs per switch.

For a single sequential Get, this means ~4 goroutine switches × ~2µs = ~8µs of scheduling overhead, on top of the actual network latency. Not huge, but it adds up — and it compounds with the two-round-trip problem (GRV + GetValue = 8 switches).

## Priority order

1. **GRV caching** — biggest single win, potentially 2x improvement
2. **Convert all request types to MessageTemplate** — low effort, saves ~20 allocs per Get
3. **Pool reply channels + eliminate context.WithDeadline** — low effort, ~300 fewer allocs/s
4. **Batched TCP writes** — medium effort, reduces syscall overhead
5. **Pipelined GRV + read** — high effort, another potential 2x on top of GRV caching
