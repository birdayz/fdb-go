## Active investigation: boolean expressions in projections

`SELECT b = true FROM lb ORDER BY a` fails with `42703: column "b=true" does not exist`. Root cause traced:

1. Parser sees `b = true` in the SELECT list. `columnNameFromExpr` correctly rejects it (has a predicate). Falls into computed-expression path.
2. `projExprs[i]` is set to the ANTLR expression (non-nil). Scope resolution correctly SKIPS it.
3. `canonicalTextOf(e.Expression())` falls through to `GetText()` for this expression — produces `"b=true"` without spaces. **This is the canonicalTextOf bug**: the start/stop token interval should work but doesn't for this specific expression shape. Need to debug why `GetInputStream()` returns nil.
4. `isComputedExpression("b=true")` returns false (no spaces, no operators detected) — so the Cascades translator treats it as a plain column name → `FieldValue{Field: "B=TRUE"}`.
5. Executor looks up `"B=TRUE"` in the row map → not found → `42703`.

**Two fixes needed:**
- **Fix canonicalTextOf** for this expression shape (why does the input stream fallback fire?)
- **upgradeProjectionValues** should handle comparison expressions in projections (`b = true` → a boolean Value). Java's ExpressionVisitor resolves `b = true` as a `ComparisonValue` (returns boolean). Go's `expr.Resolver.WalkExpression` currently only handles this in predicates (WHERE), not in projections. Need to extend `WalkExpression` to produce a boolean Value from comparison expressions.

The Java-aligned fix: the expression walker should handle comparison expressions as Values (not just Predicates). Java's `ExpressionVisitor.visitExpression` returns `Expression` for everything — comparisons, arithmetic, function calls. Go's walker currently bifurcates: `WalkPredicate` for comparisons, `WalkExpression` for values. The proper fix is to make `WalkExpression` handle comparisons too (returning a boolean-typed Value), which is what Java does.
