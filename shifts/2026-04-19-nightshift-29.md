## Known issues / tech debt

- `execSelect`, `execInsert`, `execDelete`, `execUpdate` all load the schema on every call — no caching. Fine for now.
- WHERE/SET only supports `col = constant` (simple equality, no `>`, `<`, `AND`, `OR`, subqueries). Anything more complex returns `ErrCodeUnsupportedOperation`.
- UPDATE/DELETE do full-table scan + post-filter — no index pushdown. Phase 3 (semantic analysis) will address this.
- `execInsert` does not validate that PRIMARY KEY columns are provided — proto2 optional semantics means missing PK field = absent field, which may cause weird behavior on scan.
