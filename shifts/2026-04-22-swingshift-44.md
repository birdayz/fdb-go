## What landed (chronological)

### Covering-index pushdown — the HIGH item (b2260177 + review fixes f6317901)

When every column referenced by projection / WHERE / ORDER BY is
derivable from the index key + PK, synthesise a
`*FDBStoredRecord[proto.Message]` from the `IndexEntry` alone
(dynamicpb message) and skip the per-row `LoadRecord` fetch. One
FDB round-trip per row instead of two. Applies to every
secondary-index pushdown branch (equality / IN-list / range /
composite range / LIKE prefix).

- `canCoverIndex(sq, idx, rt) bool` in `covering_index.go`: walks
  `sq.projCols` / `projExprs` / `whereExpr` / `orderBy` via
  `collectColumnRefs` (finds every `FullColumnNameExpressionAtomContext`
  node) and compares against `(secondaryIndexColumns(idx),
  rt.PrimaryKey.FieldNames())`.
- Bails on `SELECT *` / `qualifier.*` / joins / aggregates /
  `GROUP BY` / `HAVING` / non-scalar kinds (enum / message / repeated).
- `synthesizeCoveredRecord` populates the dynamicpb fields from
  `entry.IndexValues()` (indexed cols) + `entry.PrimaryKey()` (PK
  cols, skipping the record-type-key prefix via the new exported
  `recordlayer.KeyExpressionHasRecordTypePrefix`). Conversion errors
  surface (round-1 review fix — previously silently dropped the
  field, producing NULL-vs-row false-negatives).
- `coveringIndexRangeScanCursor` takes `*Index` directly (round-1
  fix: previously took a name + re-looked-up, with a `return nil`
  latent-panic branch).
- Range-building helpers extracted: `buildSecondaryIndex{Equality,
  Range,CompositeRange}TupleRange` are shared between covering and
  non-covering cursor builders (round-1 review — removed duplication).
- Pinned by `covering_index_pushdown.yaml` (24 cases) + the
  secondary-IN-list's covering path.

### IN-list pushdown — 4 variants

- **Single-col PK** (2323bb93): `WHERE id IN (v1..vN)` decomposes
  into N point scans via `pkInListCursor` (lazy chain).
- **Single-col secondary index** (2323bb93, covering wired in
  008c4950): each sub-scan uses `secondaryIndexPushdownCursor`
  (or `coveringIndexRangeScanCursor` when `canCoverIndex` holds).
- **Composite PK** with leading-eq relaxation (55277d19):
  `WHERE a = lit AND b IN (v1..vN)` on PK (a, b) — N sub-scans
  anchored at `[a_lit, v_i]`. IN-col can sit anywhere, not just
  last; earlier cols must be equated; later cols unconstrained
  (post-filtered).
- **Composite secondary index** (4fafe11a): same shape for
  `(region, plan)` etc. Covering dispatch in the sub-scan.

Bail cases (everywhere): NOT IN, subquery IN, NULL-only lists,
type mismatch per element, composite-PK with missing leading-eq.

Also fixed a **nil panic** (008c4950) on `WHERE v IN v` /
`WHERE x IN ?`: the inList grammar admits a `preparedStatement
Parameter` / `fullColumnName` alternative where `Expressions()` is
nil. Both the pushdown extractor and `evalInPredicateTri` now
nil-check before `AllExpression()`. Pre-existing latent bug — the
panic was reachable by any user query with `IN ?`.

Pinned by `in_list_pushdown.yaml` (41 cases including UPDATE /
DELETE via composite PK IN).

### LIKE prefix pushdown — 4 variants + interior wildcards + ESCAPE

- **Single-col** (0730e9fa): `WHERE col LIKE 'foo%'` on STRING
  PK or VALUE index → `[prefix, strinc(prefix))` via
  `extractColLikePrefixLiteral` + `likePrefixToPKRangeBounds`.
- **Composite PK range + LIKE** (bcf10093): wired into
  `tryPKCompositeRangeFromWhere` — `WHERE a = 'us' AND name LIKE
  'foo%'` on composite PK (a, name) narrows at the index level.
- **Composite secondary-index range + LIKE** (bcf10093): same
  for composite indexes.
- **ESCAPE clause** (4f0a0184): `LIKE 'foo\_%' ESCAPE '\'` extracts
  literal prefix `foo_`. Escape char consumes next rune verbatim;
  malformed ESCAPE (zero or multi-char) or dangling escape at EOL
  bails.
- **Interior wildcards** (f0dc406f): relaxed the extractor — any
  first unescaped wildcard terminates the literal prefix. `LIKE
  'foo%bar'` / `LIKE 'f_o'` / `LIKE 'a_ple'` now narrow correctly.
  Scan path's `likeMatch` handles the rest as post-filter.

Byte-level successor handled by `likePrefixStrinc` (strips trailing
0xFF, increments; all-0xFF falls back to open-high bound —
unreachable for valid UTF-8 anyway). Fuzz target
`FuzzLikePrefixStrinc` pins the byte-level invariant (17M
iters clean); `FuzzLikePatternToPrefix` pins no-panic + non-empty
prefix (the stronger "prefix alone matches pattern" invariant
doesn't hold for mandatory-tail patterns like `foo%bar`, relaxed
in round-6).

Pinned by `like_prefix_pushdown.yaml` (41 cases).

### ORDER BY elimination + LIMIT early-termination (c9f32b0d + 29cae655)

- Every pushdown branch sets `naturalOrder []string` — the column
  sequence the cursor emits rows in, always ASC.
- `naturalOrderSatisfies(sq.orderBy, naturalOrder,
  naturalOrderAliases)` (extracted in round-5, b6e52255) checks
  prefix match: all clauses ASC, all clauses are a prefix of
  `naturalOrder`, aliases resolved via the captured map.
- **Sort elimination**: `if !sortSkippable { sort.SliceStable(...) }`
  in the sort block — stable sort on already-sorted data is a
  no-op, skipping it is pure perf.
- **LIMIT early-termination**: when the same conditions hold AND
  `sq.limit >= 0` AND no DISTINCT, the scan loop stops reading
  once OFFSET+LIMIT rows accumulated.
- SELECT-list alias resolution (ca0604c9): `SELECT id AS pk ...
  ORDER BY pk` resolves through the alias map before the prefix
  check.

Natural orders per branch:
- PK equality / PK range / composite PK range / full scan → `pkCols`
- Secondary equality → `pkCols` (index key fixed, PK suffix varies)
- Secondary range / composite range → `idxCols ++ pkCols`
- All IN-list variants → nil (lazy chain over list order, not sorted)

Pinned by `order_by_elimination.yaml` (24 cases including DESC /
NULLS LAST / DISTINCT bail cases + alias resolution).

  `ScanProperties` argument through every pushdown cursor builder —

- [ ] **IN-list via subquery decomposition** — currently bails.
  If the subquery is small (pre-evaluated to a literal list via

- [ ] **Fuzz corpus with string dimension** — add `s string` to
  `FuzzLikePatternToPrefix`, assert `likeMatch(pattern, s, esc)
  && ok ⇒ strings.HasPrefix(s, prefix)`. Directly pins "no
  matching row excluded by the range" instead of the current
  no-panic + non-empty weak invariants.
- [ ] **Mutual recursion** (from dayshift-43 handover) — `WITH
  RECURSIVE a AS (..., b ...), b AS (..., a ...)`. Today CTEs
  are evaluated in declaration order; mutual refs bail.
- [ ] **Continuation resumption of partial recursive results**
  (from dayshift-43) — for `maxRows: 1` pagination. Java's
  `RecursiveStateManager` uses TempTable cursors.

### LOW

- [ ] **ORDER BY dedup edge cases** — `ORDER BY b, B` (case-folded)
  and `ORDER BY a.b, b` (qualified vs bare). Niche; rides on a
  future identifier-folding pass.
- [ ] **`GROUP BY (a+b) AS alias`** — expression group keys can't
  be referenced via alias from the SELECT list.
- [ ] **Benchmark comparing pushdown vs full scan** — no direct
  micro-benchmark. Existing `just bench` bench suite doesn't
  isolate the cursor-pick cost. Would make the perf win
  observable.
- [ ] **One-element IN-list → equality** — `WHERE pk IN (5)` could
  fire the equality pushdown (tighter narrowing); currently goes
  through IN-list sub-scan-of-one. Tiny perf.
- [ ] **Pure-prefix composite-PK pushdown** — `WHERE a = 1 AND b
  = 10` on PK (a, b, c) currently bails to scan (no range col).
  A pure-prefix extractor could narrow to `[rtk, 1, 10]` and
  trailing-PK unconstrained. Moderate effort.
