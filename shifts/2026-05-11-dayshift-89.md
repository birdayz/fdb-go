## What was done

### Java ExceptionUtil error mapping (1:1 mechanical port)

Complete port of Java's `ExceptionUtil.recordCoreToRelationalException`:
- `*wire.FDBError{Code: 1031}` → 53F00 (TRANSACTION_TIMEOUT)
- `*wire.FDBError{Code: 1020}` → 40001 (SERIALIZATION_FAILURE)
- `*wire.FDBError{Code: 1007}` → 40001 (SERIALIZATION_FAILURE)
- `*wire.FDBError{Code: 2017}` → 25F01 (TRANSACTION_INACTIVE)
- `*MetaDataError` → 42000 (SYNTAX_OR_ACCESS_VIOLATION)
- `*RecordAlreadyExistsError` → 23505 (UNIQUE_CONSTRAINT_VIOLATION)
- `*RecordDeserializationError` → XXF01 (DESERIALIZATION_FAILURE)

Primary detection via typed `*wire.FDBError` (numeric codes), string fallback for wrapped errors. Translation covers both Plan() and Execute() error paths. 12 unit tests pin all mappings.

### SQL benchmarks (#34 progress)

3 new benchmarks with allocation data:
- `BenchmarkFDB_JoinQuery` — NLJ, 5 result rows: 4.65ms, 38K allocs
- `BenchmarkFDB_AggregateGroupBy` — 5 groups: 8.05ms, 9K allocs
- `BenchmarkFDB_IndexScanRange` — ~50 rows: 2.18ms, 12K allocs

Key finding: Cascades planner overhead is negligible (<0.05ms) vs FDB I/O (~1.1ms). Plan cache value is in parsing avoidance, not planning time.

### fdb.Error value type handling

Discovered that `DB.Run()` converts `*wire.FDBError` to `fdb.Error` (value type) via `convertError`. The original typed detection in `translateFDBError` only matched `*wire.FDBError` — the actual error type surfacing from production code (`fdb.Error`) was caught only by the string fallback. Fixed by adding `fdb.Error` detection and extracting `translateFDBCode` helper.
