## What landed

### Correlated subqueries (headline feature)

**Commit 0571e2cc** ‚Äî outer-row scope stack. `WHERE EXISTS` / `NOT
EXISTS` / `IN` / `NOT IN` subqueries can now reference outer-row
columns. Implementation:

- `outerScopes []outerScope` field on `EmbeddedConnection`.
- `pushOuterScope()` push + pop helper, called at each subquery
  entry point via `defer conn.pushOuterScope(...)()`.
- `resolveOuterColumn()` walks the stack innermost-first for both
  proto-path (`evalExprAtom`) and map-path (`evalExprAtomOnMap`)
  column resolution.
- `outerScopeFromMsg` (proto) and `outerScopeFromMapRow` (map)
  build an `outerScope` with qualifier set from the source.

5 subquery call sites plumbed: `evalExprPredicateTri` (proto
EXISTS), `evalInPredicateTri` (proto IN subquery), `evalHavingTri`
(HAVING EXISTS), `evalPredicateOnMapExprTri` (map EXISTS),
`evalPredicateOnMapTri` (map IN subquery). All use `defer` for
panic safety (addressed in review).

**Commit 98f9cc34** ‚Äî user-alias support. The proto scan loop now
populates `conn.currentSourceAliases` via `pushSourceAliases()`, so
`outerScopeFromMsg` picks up the SQL-level alias (`FROM emp AS e`
‚Üí `{E, EMP}`) in addition to the proto descriptor name. Without
this, correlated refs via an explicit alias errored 42703.

**Commit 81082d5**, **315393a1** ‚Äî UPDATE / DELETE scan loops also
push `currentSourceAliases` so correlated EXISTS / NOT EXISTS works
in WHERE of `UPDATE`/`DELETE` too.

**Commit 8cf41205** ‚Äî inner-scan user alias recognized in qualified
col refs. Inner's `FROM project AS p WHERE p.emp_id = e.id` was
erroring 42703 because the inner qualifier check only accepted the
proto descriptor name (PROJECT); extended to also consult
`conn.currentSourceAliases` which the inner scan loop already
populates.

**Pinned by `correlated_subquery_probes.yaml` (16 cases)**:
EXISTS / NOT EXISTS / IN / NOT IN / additional inner filter /
self-referential uncorrelated / self-correlated with alias /
outer-scope-from-JOIN-map / doubly-nested (inner refs outermost
row) / UPDATE with correlated EXISTS / DELETE with correlated NOT
EXISTS / both outer and inner aliased on different tables / unknown
qualifier ‚Üí 42703.

**Scalar subquery correlation stays unsupported**: scalar
subqueries pre-evaluate before the outer scan starts (to avoid
nested FDB txns), which precludes per-row outer binding. Pinned as
42703 in the probes.

### Derived-table GROUP BY / ORDER BY / LIMIT / HAVING restoration (bug fix)

**Commit ce7d8a15** ‚Äî root cause: `extractFromSimpleTable`'s
derived-table branch returned early, bypassing the shared post-
construction pipeline that parses ORDER BY / LIMIT / OFFSET /
GROUP BY / HAVING and runs GR1 validation. So a query like
`SELECT MAX(x.col1) FROM (SELECT col1 FROM t) AS x GROUP BY x.col1`
aggregated without grouping, returning a single row.

Refactor: build `sq` in both the derived-table and atom-table
branches, then fall through to the shared pipeline. Plus three
follow-on fixes in `aggregateMapRows`:

1. **Seed group-by values into rowMap** ‚Äî `SELECT x.col1 + 10 FROM
   ... GROUP BY x.col1` errored 42703 at emit time because only
   aggCols-outName entries were in rowMap. Populate both qualified
   and bare forms of each group value.
2. **Aggregate arg validation** ‚Äî `MAX(x.col2)` absent from source
   now errors 42703 instead of silently returning NULL.
3. **outExpr column-ref validation** ‚Äî ungrouped col ref in a
   projection expression now errors 42803 (grouping error) instead
   of silently failing at emit with 42703. New
   `harvestColumnRefs` helper walks the expression tree, excluding
   refs inside aggregate calls. Matches Java's distinction between
   "exists but ungrouped" (42803) and "doesn't exist" (42703).

**Pinned by `derived_table_group_by.yaml` (10 cases)**: all 7 cases
from Java's groupby-tests.yamsql lines 136‚Äì185 (MAX grouped / col2
not in derived ‚Üí 42703 / both cols / `col1 + 10` group-key /
ungrouped `x.col1 + x.col2` ‚Üí 42803 / `col1 + col1` /
nested-aggregate-derived + outer WHERE) plus ORDER BY / LIMIT /
HAVING regression guards on derived-table outer.

### Review-feedback fixes (all review items addressed)

**Commit 4a0d9b21** ‚Äî switched all 5 `pushOuterScope` call sites
from explicit `pop()` to `defer conn.pushOuterScope(...)()` for
panic safety. Reviewer flagged the inconsistency with the
docstring and with `pushCTEScope` / `pushValidQualifiersScope` /
`pushSourceAliases`. Plus clarifying comments on intentional
asymmetries:

- `resolveOuterColumn`: qualified refs stop at first matching
  qualifier scope (SQL semantics: one source per qualified ref).
- `harvestColumnRefs`: 42803 only raised when ref is defined in
  source; undefined refs fall through to 42703 at emit. Matches
  Java.
- `groupByNames` case-sensitivity caveat.

**Commits 315393a1, 52ecb304, 8cf41205**: the reviewer's three
minor observations (execDelete consistency, fragile INSERT-Carol,
inner alias recognition) all addressed.
