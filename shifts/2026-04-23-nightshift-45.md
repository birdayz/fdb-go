### Pushdown perf follow-ups

**One-element IN-list → equality (84fab4e1)**
- PK and secondary single-col IN-list, at the dispatch site:
  when `len(pkVals) == 1` (or `len(sil.values) == 1` for
  secondary), promote to the equality cursor instead of building
  a one-element lazy chain.
- Observable win: `naturalOrder = pkCols` for the len==1 case
  enables ORDER BY elimination + LIMIT early-termination that
  was previously blocked.
- Review caught: same promotion was missing from the DML shared
  dispatcher `pkPushdownCursor`. Fixed in round 1 (de0b47c1).
- yamsql: 4 new cases in `in_list_pushdown.yaml` covering PK and
  secondary, with and without LIMIT, NULL-drop edge case.

**Composite-PK pure-prefix pushdown (8cc61e49)**
- `WHERE a = 1 AND b = 5` on PK (a, b, c) was falling through to
  full type scan. Added `tryPKCompositePrefixFromWhere` picking
  the longest leading prefix of equated PK cols; narrows to
  `[rtk, eq_vals...]` as a tuple prefix, residual cols stay
  post-filter via evalPredicate.
- Slots into the dispatcher chain as the LAST PK branch (tighter
  composite forms still win first). SELECT + DML both route
  through it.
- yamsql: 13 new cases in `composite_pk_prefix_pushdown.yaml`
  covering 2-col + 3-col PKs, residual on trailing cols, bail
  cases, type-mismatch, UPDATE/DELETE through the path.

**FuzzLikePatternToPrefix strengthened with string dimension
(811a8698)**
- Extended the fuzz target from `(pattern, escape)` to
  `(pattern, escape, s)`. Now pins the correctness invariant
  for the pushdown: `likeMatch(pattern, s, escape) && ok ⇒
  HasPrefix(s, prefix)`.
- Restricted to valid UTF-8 inputs — `likePatternToPrefix`
  rune-decodes (invalid bytes fold to U+FFFD), `HasPrefix` is
  byte-level, so the two disagree on non-UTF8 sequences.
  Protobuf STRING columns are UTF-8 by contract, so the fuzz
  skips invalid sequences.
- The fuzzer immediately caught one false-negative (pattern
  `"a\xff%"` vs s `"a\x98"`), which drove the UTF-8 scoping.
  40s × ~1.8M execs/s now clean.

**ORDER BY DESC via reverse scan (fb2ee26f)**
- Added `naturalOrderSatisfiesReverse` (DESC-prefix counterpart
  of the ASC checker; NULL-ordering flips to NULLS LAST as the
  native reverse-tuple order) and `scanPropsForOrder` (direction
  selector).
- Plumbed `ScanProperties` through `pkPushdownScanCursor`,
  `pkPushdownRangeScanCursor`, `pkPushdownCompositeRangeScanCursor`,
  and the `ScanRecordsByType` full-scan fallback. UPDATE / DELETE
  paths pass `ForwardScan()` explicitly (no naturalOrder consumer
  there).
- `reverseScanApplied` flag tracks the direction decision so the
  post-scan sort eliminator + LIMIT early-term accept
  DESC-prefix matches.
- Secondary-index reverse scan stays on the follow-up list —
  scope-controlled. Not urgent: secondary-index scans fall back
  to in-memory sort which is correct, just slower.
- yamsql: 9 new cases in `order_by_elimination.yaml` covering
  single-col PK DESC, PK range DESC, composite PK range/prefix
  DESC, DESC + LIMIT, alias-resolved DESC, mixed ASC/DESC bail.

### RFC-022 Cascades conformance staging (c12f3136)

User asked to review TODO.md Phase 4 vs Java conformance. The
TODO conflates two conformance goals of very different difficulty:
semantic equivalence (yamsql result-set identical) and plan
equivalence (plan tree + cache-key identical). RFC-022 splits
them and re-orders Phase 4 so the hardest questions land first:

- **4.-1 Plan-equivalence harness** built against the naive
  generator FIRST — gives a baseline of "what Go and Java already
  agree on" before any rule porting.
- **4.-0.5 Generics-vs-interfaces spike** resolves RFC-021 risk
  #3 up-front (Cascades is heavy on `Value<T>` /
  `BindingMatcher<? extends T>`; Go's lack of wildcard bounds
  needs a decision).
- **4.-0.25 Plan-cache-key compatibility sub-RFC** decides
  whether we chase Java-matching cache-key hashes (requires pinning
  to Java 4.10.6.0 and forever catching up) or downgrade to
  best-effort.
- **4.5 Rule porting re-ordered**: Batch A = six rules covering
  swingshift-44's 11-branch pushdown chain first, so the harness
  gets end-to-end coverage of the existing 94 yamsql scenarios
  before the 60+ less-common rules land.
- Round-2 review flagged open-question #3 as "load-bearing";
  closed in d7c332dd: naive generator has NO stable plan-cache-key
  format today (no cache at all — parses + plans + executes
  inline per call). Phase 2 / 4.-0.25 is therefore strictly a
  future concern with no pre-existing wire format to preserve.

### MEDIUM — complete reverse-scan DESC story

- [ ] **Secondary-index reverse scan** — `secondaryIndexRangeScanCursor`,
  `secondaryIndexCompositeRangeScanCursor`, and the covering-index
  variants still hardcode `ForwardScan()`. Plumb ScanProperties
  through the same way PK cursors now do, so `WHERE idx_col > 5
  ORDER BY idx_col DESC` takes reverse scan instead of in-memory
  sort.

- [ ] **ORDER BY with equality-prefix relaxation** — `WHERE a = 1
  ORDER BY b, c` on PK (a, b, c) currently bails because
  `naturalOrderSatisfiesDir` requires the ORDER BY's first clause
  to match naturalOrder[0] (= a). But a is equated, so every row
  has the same a; the emission order by (a, b, c) IS just by
  (b, c). Relaxation: treat equated leading cols as zero-width.

### LOW — follow-up ideas from swingshift-44

Still pending:

- [ ] **IN-list via subquery decomposition** — currently bails.
  If the subquery is already pre-evaluated (existing
  `preEvaluateScalarSubqueries` path) and small, decompose into
  N point scans like literal IN-list.
- [ ] **Mutual recursion** — `WITH RECURSIVE a AS (..., b ...), b
  AS (..., a ...)`. Today CTEs are evaluated in declaration order.
- [ ] **Continuation resumption of partial recursive results** —
  for `maxRows: 1` pagination. Java's `RecursiveStateManager` uses
  TempTable cursors.
- [ ] **Benchmark comparing pushdown vs full scan** — the existing
  bench suite doesn't isolate the cursor-pick cost.
- [ ] **Pure-prefix composite-secondary-index pushdown** — this
  shift did PK. Same relaxation for composite secondary indexes
  would let `WHERE region = 'us' AND plan = 'pro'` narrow on
  `idx_region_plan` even without a range.
