## TL;DR

**Architectural Java-conformance fix landed** for the embedded engine's ORDER BY rejection path: removed the in-memory `sort.SliceStable` fallback at `select_query_full.go:1152`, gated every PK / secondary-index pushdown branch on whether its natural emission order satisfies the user's ORDER BY, sorted IN-list values internally so sub-scan concatenation emits in PK order, added `tryIndexScanForOrdering` (full secondary-index scan as the last branch before the full-PK fallback), tracked at-most-1-row scans (PK equality, single-value IN-list) for trivial-sort exemption, exempted aggregate / DISTINCT (the latter is a Go-only feature with its own pipeline). Rejection is now structural ("no scan satisfies"), mirroring fdb-relational's Cascades planner emergent rejection ("no rule fires" — `RemoveSortRule` exists but no `ImplementSortRule` does, so when no inner plan satisfies, `CascadesPlanner.resultOrFail` throws `UnableToPlanException`). Single-table SELECT path is now Java-conformant; JOIN / CTE / UNION sort sites still have in-memory fallbacks (tracked in TODO.md, deferred to C2 QueryExecutor port).

**Plus 24 new A3 cross-engine scenarios** (54 → 78 total) covering null comparisons, boolean precedence, self-join, string comparison, NULL arithmetic, ORDER BY indexed col, compound arithmetic, multi-statement DML setup, complex WHERE 3VL combinations, PK equality + ORDER BY non-PK col with satisfying index, projection-alias forms (column- and table-aliased), JOIN with ORDER BY on right-side PK (documenting Java's cost-driven JOIN-side reordering), PK-DESC ordering via reverse-scan, BIGINT MAX/MIN/zero numeric boundaries, extended COALESCE shapes (2/3/4-arg, fallback chains, COALESCE-in-WHERE), LIKE with ESCAPE clause, UTF-8 string round-trip, pure-constant SELECT-list projection, indexed IN-list with ORDER BY, type-promoted comparisons (`<`, `<=`, `>`, `>=`, `=`, `<>` BIGINT vs DOUBLE), advanced DML (multi-col UPDATE / computed RHS / compound DELETE / no-match), composite-index ORDER BY (single-col prefix, leading-col DESC, equated-leading + trailing-col), strict-order NULL position (ASC NULLs FIRST + DESC full-tuple-reverse), and `WHERE col IS NULL`+`ORDER BY` interaction with a satisfying index. Surfaced **5 new fdb-relational gotchas** documented in CLAUDE.md: `OR/AND` left-to-right same-precedence, `+/-/*/%/...` left-to-right same-precedence, unary minus on a column reference rejected, bare-paren `(scalar)` in projection treated as record-constructor, UNION inner-SELECT ORDER BY rejected (Cascades-emergent). Also added **CLAUDE.md design principle 10** ("Emergent behaviour over special-case checks") capturing the architectural insight from the ORDER BY fix.

**Refinement note (last commit)**: dropped the initial `atMostOneRow` exemption that made PK equality / single-value IN-list scans bypass the rejection check. Java's `RemoveSortRule` checks the Ordering property explicitly — an equality match has `Ordering=()` which doesn't satisfy a non-empty requested ordering. Java rejects `WHERE id = X ORDER BY non_pk_col` even though the result is at-most-1-row. Refined to gate ALL PK / secondary-index branches uniformly on `pkOrderingSatisfiesOrderBy` / `indexBranchSatisfiesOrderBy`, letting them decline when ORDER BY can't be satisfied so the chain falls through to `tryIndexScanForOrdering` (or rejection if no index satisfies). Both engines now align: with a satisfying index, both succeed; without, both reject.

### CLAUDE.md gotchas added (4 new behavioural-divergence entries)

1. **`OR`/`AND` operator-precedence diverges from SQL standard.** fdb-relational 4.11.1.0 binds them at same precedence left-to-right, so `a OR b AND c` parses as `(a OR b) AND c` instead of standard `a OR (b AND c)`. Surfaced by `boolean_precedence` test #0 returning [2,4] (Java) vs [1,2,4] (Go / SQL spec). Cross-engine corpus must use explicit parens for any OR/AND mix.
2. **Arithmetic operator-precedence diverges from SQL standard.** Same shape: `a + b * 2` parses as `(a + b) * 2` instead of SQL-standard `a + (b * 2)`. Surfaced by `arithmetic_compound` test #0 returning 26 (Java) vs 16 (SQL spec / Go) for `(a, b) = (10, 3)`. Same workaround.
3. **Unary minus on a column reference rejected by parser.** `SELECT -col FROM ...` fails with `syntax error` at the `-`; only numeric-literal unary minus parses. Use `0 - col` for negation.
4. **Bare-paren `(expr)` around a single scalar in projection** is parsed as a single-element record/tuple constructor, returning `ImmutableRowStruct`. Same parser quirk as the existing `WHERE (b = true)` rejection — extends to projection context. Multi-operator parens (`(a + b) * 2`) work because the outer operator forces scalar context.

### Architectural — ORDER BY rejection mirroring Java's emergent path

**Root cause story**: The Go embedded engine has had a hard-coded `sort.SliceStable` fallback since April 19 (commit `32545be2 feat(relational/embedded): ORDER BY + LIMIT for SELECT`). The implementation was modelled after SQL standard semantics (scan→filter→sort→limit), not after fdb-relational's Cascades architecture. The Cascades port (Track B) is in progress but not yet wired into the embedded engine. The cross-engine harness (Track A3) is the first mechanism that could surface this divergence; surfaced this shift by `string_compare` test #11 (ORDER BY non-PK string col).

**Java's structural rejection**: fdb-relational's Cascades planner has only `RemoveSortRule` + `PushRequestedOrderingThroughSortRule` in `cascades/rules/`. There is **no `ImplementSortRule`** — the only call site of `RecordQuerySortPlan` is the legacy heuristic `RecordQueryPlanner.java:315`, not the SQL stack's Cascades. With no rule generating a `SortPlan` in the SQL path, when no inner plan satisfies the requested ordering, the Cascades planner produces no physical plan and `CascadesPlanner.resultOrFail` throws `UnableToPlanException("Cascades planner could not plan query")`. The rejection is **emergent from rule-set incompleteness**, not from an explicit "throw if no satisfying scan" check.

**Go's mirror**: Replaced the `sort.SliceStable` fallback with a structural check that emerges from "no scan-strategy produces rows in the requested order":

- **Helpers** (`order_by.go`):
  - `scanSatisfiesOrderBy(orderBy, naturalOrder, equatedCols, aliases)` — wraps `naturalOrderSatisfies` ∨ `naturalOrderSatisfiesReverse`.
  - `pkOrderingSatisfiesOrderBy` — empty / all-equated / scan-by-PK-cols satisfies.
  - `indexBranchSatisfiesOrderBy` — secondary-index branch flavour, builds `(idxCols, pkCols)` candidate.
- **Gates** (`select_query_full.go`):
  - PK range / composite-range / composite-prefix / single-equality-on-secondary: gated on `pkOrderingSatisfiesOrderBy`.
  - Secondary-index range / composite-range / composite-prefix: gated on `indexBranchSatisfiesOrderBy`.
  - PK IN-list / composite-PK IN-list / single-value secondary-index IN-list: gated, plus IN-list values pre-sorted internally so the lazy chain emits in PK order (matches Java's IN-list-emit-in-key-order via the IndexEntry sort).
  - Multi-value secondary-index IN-list / composite-IN-list lazy chain: gated on `len(orderBy)==0 || allOrderByEquated` (no usable natural order across sub-scans).
- **New branch** (`secondary_index_pushdown.go`): `tryIndexScanForOrdering` — full secondary-index scan (no WHERE pushdown) inserted as the last branch before the full-PK fallback, picks an index whose `(idxCols, pkCols)` satisfies the ORDER BY. Mirrors Java's `RemoveSortRule` firing when an index scan's Ordering property satisfies the requested order.
- **At-most-1-row exemption**: PK equality / single-value IN-list set `atMostOneRow = true`. Exempted from the rejection because the in-memory sort over a 1-row result is trivially correct (Java's RemoveSortRule treats 1-row Ordering as "any" satisfying any requested ordering).
- **Aggregate / DISTINCT exemption**: Aggregate produces a small post-aggregation result set; DISTINCT is a Go-only feature with its own pipeline (Java rejects DISTINCT entirely per existing CLAUDE.md gotcha). Both keep the `sort.SliceStable` fallback.
- **Final rejection**: `if !satisfiable && !isAggregate && !sq.distinct && !atMostOneRow { return nil, api.NewErrorf(api.ErrCodeUnsupportedSort, "ORDER BY clause cannot be satisfied by any scan strategy; no index produces rows in the requested order") }`.
