### Production gap (P4 still 3s)

The unit test produces the correct plan. The production stress test still takes 3s because:

1. **PullFilterAboveSortRule** copies the IN-filter above the Sort, creating TWO SelectExpressions (one below Sort, one above). Both get InJoin plans.

2. **Root Reference has 17 members** including Projection wrappers, InJoin wrappers, InMemorySort wrappers, and Filter wrappers at the same level. The cost model correctly identifies InJoin(IndexScan) as cheaper, but...

3. **The extraction follows Projection wrappers**, not bare InJoin wrappers. `promoteInJoinWinners` promotes InJoin at the Reference, but the extraction picks a Projection wrapper that wraps InMemorySort(Filter(Scan)) — not the Projection wrapping InMemorySort(InJoin(IndexScan)).

4. **Root cause**: the Projection wrapper that wraps InMemorySort(InJoin(IndexScan)) IS among the 10 Projection members. But `reoptimizeAll` picks the first physical as winner (the one wrapping Filter(Scan)), and the extraction uses the winner. The promoteInJoinWinners pass only promotes bare InJoin/InUnion, not Projection-wrapped InJoin.

### Fix path (clear, one change needed)

Extend `promoteInJoinWinners` to also consider Projection wrappers whose INNER plan is an InJoin/InUnion. Walk the plan tree to check if InJoin/InUnion is nested inside. If a Projection(InMemorySort(InJoin(IndexScan))) is cheaper than the current winner Projection(InMemorySort(Filter(Scan))), promote it.

Alternatively: fix `reoptimizeRecursive` to unconditionally re-evaluate using `GetBest(costModel)` — but this was tried and causes regressions with sort-elimination tests due to cost model criterion #12 (unmatchedFieldCount penalizes ordered index scans). The criterion #7 sign bug (documented in this shift's investigation) should be fixed first.

## Key findings

- **All queries go through Cascades** (line 311 of connection.go). The plan visitor's pushdown chain (`select_query_full.go`) is dead code for `database/sql` driver queries.
- **EXPLAIN uses naive path**, not Cascades. EXPLAIN output doesn't reflect the Cascades physical plan.
- **PlanningCostModelLess criterion #7 has swapped return values** — `comparePrimaryScanVsIndexScan` returns 1 (index wins) when it should return -1 (primary wins), and vice versa. Hidden by criterion #2's `.Total()` vs `.Cardinality` asymmetry. Not fixed this shift to avoid cascading regressions.
- **Cost model criterion #12** (unmatchedFieldCount) penalizes full-range index scans that provide ordering. This prevents `reoptimizeAll` from picking reverse index scans when InMemorySort is also available.
