## What was done

### Two CLAUDE.md gotchas RESOLVED (Go-vs-Java conformance)

1. **Bare-bool-column-as-operand-in-projection** — `SELECT b AND TRUE`, `SELECT NOT b`, `SELECT b OR FALSE` etc. over a BOOLEAN column now match Java's Kleene three-valued semantics. Fix threaded `allowBareField bool` through `evalExprPredicateTri` / `evalComparisonPredicateTri` (`pkg/relational/core/embedded/eval_predicate.go`): operands of AND/OR/NOT/XOR (any context) and projection-level `evalExpr` (`eval_proto.go`) pass `true` so a bare `FullColumnNameExpressionAtomContext` evaluates as a value via IsTruthy. Top-level WHERE / HAVING entry still passes `false`, preserving Java's `WHERE flag` rejection. Five corpus entries re-enabled in `booleanScenario`. CLAUDE.md gotcha entry marked RESOLVED. Go-side regression test in `TestFDB_BareBoolProjection` (`pkg/relational/sqldriver/embedded_fdb_test.go`).

2. **`SUM(BIGINT)` int-preserving accumulator** — `SELECT SUM(qty) / COUNT(*) FROM t` over a BIGINT `qty` now integer-divides (10/3=3) on both engines, matching Java's `SUM(BIGINT)→BIGINT` typing. `mapGroupState` / `groupState` (`pkg/relational/core/embedded/aggregate.go`, `select_query_full.go`) carry parallel `sumsI []int64` and `sumNonInt []bool` per slot. `sumNonInt` starts as the zero value (false = "still int-only") and only flips to true the first time a non-int64 runtime value is observed — `make([]bool, n)` gives the correct initial state, so the original `sumIntOnly` flag was inverted and the `newAllTrue` helper dropped per /simplify review. Overflow on the int64 accumulator wraps silently, same as Java's `long`. CLAUDE.md gotcha entry marked RESOLVED. Go-side regression test in `TestFDB_SumIntegerDivision`.

### SQLState cross-engine wiring (Skip'd at harness gate)

Java side (`conformance/conformance_server.java`): catches the root-cause exception and extracts SQLSTATE via two paths — `SQLException.getSQLState()` (JDBC standard) and `RelationalException.getErrorCode().getErrorCode()` (the planner / executor surface, via reflection to avoid hard imports). Emits the SQLSTATE in the structured error response as `sqlState` alongside the existing `error` / `exceptionClass` / `exceptionFullClass` fields.

Go side (`pkg/relational/conformance/plandiff/httpclient.go`): `*JavaError{Message, ExceptionClass, ExceptionFullClass, SQLState}` typed struct replaces the previous `fmt.Errorf` format-only error. Existing string-matching callers continue to work via the `Error()` method's `"plandiff: java <ExceptionClass>: <Message>"` format. `TestJavaError_SQLStateExtraction` + `TestJavaError_NoSQLState` pin the wiring on mocked conformance-server responses.

Cross-engine harness (`conformance/yamsql_cross_engine_conformance_test.go`): `assertCrossEngineErrorCode` helper added (currently dead — harness Skips at the per-test gate). Lifting the skip surfaced fdb-relational planner stalls on certain error paths under load (type-mismatch IN-list, GREATEST mixed types, comma-join + bare-col-with-aggregate); under load these stalls compound and Java's HTTP server thread pool starves, dragging unrelated `error_conformance_test.go` tests into 120s timeouts. ### CLAUDE.md gotchas added (2 new)

1. **`col IN (SELECT ...)` parser-NPEs in fdb-relational 4.11.1.0** — both `WHERE v IN (SELECT v FROM b)` and `WHERE v NOT IN (SELECT v FROM b)` raise NullPointerException from the IN-predicate visitor. Joins the existing parser-NPE bug list (CROSS JOIN, scalar subqueries' expressionAtom, COUNT(DISTINCT)). Cross-engine corpus must skip `subquery_in.yaml` entirely.

2. **WHERE-bare-bool-operand rejection (one-sided gap)** — `WHERE b AND TRUE` / `WHERE b OR FALSE` / `WHERE NOT b` over a BOOLEAN column raise VerifyException in fdb-relational 4.11.1.0, even though the same shapes in projection are accepted. Java treats the operand of AND/OR/NOT inside WHERE as needing a strictly-typed BooleanValue, not a coerced FieldValue. Go's embedded engine accepts both contexts uniformly (SQL-correct); the asymmetry is fdb-relational-side.
