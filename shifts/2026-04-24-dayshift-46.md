### RFC-022 Phase 4 staging

**RFC-023 generics-vs-interfaces decision (952807c7)**
- Two working implementations of the Cascades Value + BindingMatcher
  hierarchy (shape (a): interface + any; shape (b): generic structs
  + constraints) with a shared 10-line predicate matcher running in
  both. Measured API friction and composition ergonomics against
  the real use case.
- Decision: shape (a) — non-generic `BindingMatcher` + `any` +
  free-function `Get[T]` retrieval helper.
- Three findings:
  1. Zero-size matcher structs collapse as map keys under Go's
     zero-size-type optimisation. Both shapes need nonce fields
     + factory constructors. Java doesn't hit this.
  2. Shape (b)'s composition-time type safety collapses under
     heterogeneous children — `ArithmeticMatcher.Left:
     BindingMatcher[Value]` is shape (a) with generic sugar.
  3. Generic combinators don't compose past 2 levels. Real
     Cascades rule patterns are 4-7 levels deep.
- Shape (b) deleted mid-shift; shape (a) promoted to its Java-
  aligned production home at `pkg/recordlayer/query/plan/cascades/`
  (commit a7a02b4d).

**RFC-024 plan-cache-key compatibility sub-RFC (fb070d9d)**
- Java's `RelationalPlanCache` is Caffeine, per-process,
  in-memory. No serialiser, no persister, no replicator.
- `QueryCacheKey.hashCode()` is JVM-local via `Objects.hash(...)`.
- Conclusion: hash-identical Java compat is NOT achievable without
  building Java-tag-pinned emulators for `AstNormalizer` +
  `PlannerConfiguration` + `Objects.hash`. No consumer exists.
- Phase 4.4 cost model free to ship Go-native cost.
- Go-internal invariants still apply: deterministic hash,
  schema-version-sensitive keys, point-release stability. Folded
  into Phase 4.0 QueryCacheKey work.

### Plan.Explain() seam (8de7caf7)

- `Plan.Explain() string` on the `query.Plan` interface.
- `PlanFunc.Explain()` — nil-safe; returns `""` when `ExplainFn` is
  unset.
- `MultiPlan.Explain()` joins children with `";\n"`.
- Naive Generator returns `"<kind>: <canonical SQL text>"` —
  Phase 1a placeholder. Cascades (Phase 4+) will replace with
  structural plan-tree rendering for the RFC-022 §4.-1 plan-
  equivalence harness.

### IN-subquery decomposition + IN set semantics (ddff61ea)

- Pre-evaluate uncorrelated `col IN (SELECT ...)` before the outer
  `runInTx` opens. Same pattern as `scalarSubqueryCache` — key by
  parse-tree pointer, cleared on `ResetSession`.
- `extractColInList` consults the cache; cache-miss falls through
  to the runtime IN-subquery evaluator (preserves correlated
  semantics).
- Java-aligned dedup: `dedupeAny` in both `extractColInList` and
  `runInSubqueryOnce` collapses duplicate values (IN is a SET per
  SQL standard; Java's `PredicateSimplification` does the same).
- Fixes a pre-existing bug: `WHERE id IN (1, 1, 2)` point-scanned
  id=1 twice and emitted the matched record twice.
- 9 new yamsql scenarios in `in_subquery_decomposition.yaml`
  (added two after round-2 review gap on explicit dup-result
  coverage); 2 new in `in_list_advanced.yaml`; 8 new unit tests for
  `dedupeAny`.
