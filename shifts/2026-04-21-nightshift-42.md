## What landed (chronological)

### Recursive CTEs

**2c0bc4a7** — semi-naive / level-order evaluator. `WITH RECURSIVE c
AS (seed UNION [ALL] recursive) ...` materialises via iterative
semi-naive: each iteration binds `c.ctes[name]` to the previous
round's new rows, re-evaluates the recursive branch, appends until
convergence. UNION ALL on acyclic data shrinks the working set
naturally; UNION DISTINCT filters dups via a seen set. Cyclic UNION
ALL is bounded by a 10000-iteration cap → 54F01. RECURSIVE keyword
without self-reference routes to the non-recursive path
(Postgres-compatible). Column rename `WITH RECURSIVE c(col1, col2)
AS ...` applied to the seed BEFORE the recursion loop so the
recursive branch sees the renamed columns.

Self-reference detection: `containsTableRef` walks the parse subtree
for `ITableNameContext` nodes matching the CTE name; ignores alias
tokens.

**7c47834b** — DFS pre/post-order. Matches Java 4.7.1.0+
`RecursiveUnionCursor` DFS emission. Split into
`recursiveCTELevelOrder` and `recursiveCTEDFS`; dispatch in
`materializeRecursiveCTE` after the shared seed / rename / shape
validation.

**19ad819c, d18f776e, b664c7a7** — review fixes and hardening:
removed fragile `defer delete`, unified error wording, clarified
54F01 message, added multi-CTE and alias-collision tests,
`stripCTEColumnQualifiers` so `WITH x AS (SELECT d.id FROM ...)
SELECT id FROM x` resolves.

### PK pushdown (SELECT / UPDATE / DELETE × equality / range × single / composite PK)

**2a8be4df** — single-column equality MVP. `WHERE pk = literal`
narrows `ScanRecordsByType` to `ScanRecordsInRange` on the exact
PK.

**855ee85f** — composite PK equality. AND chain covering every PK
column pushes down; tuple built in declared PK order.

**2011f3d4** — extended to UPDATE / DELETE via shared
`pkPushdownCursor` helper. Factored `tryPKEqualityFromWhere` as the
shared core; `tryPKEqualityPushdown` as the SELECT-gated wrapper.

**31796785** — round-1 fix: `FieldKeyExpression` PK shape
(intermingled tables) bails out — no type filter on its range,
would return wrong-typed records. Only `CompositeKeyExpression`
(SQL DDL's default, with RecordTypeKey prefix) pushes down.

**338cdcd2** — extra non-PK predicates in the AND chain. `WHERE pk
= X AND status = 'active'` pushes down; scan's `evalPredicate`
applies the extras to the one loaded row. Safe: narrowed range is a
SUPERSET of matching rows.

**5d105adc** — single-col PK range predicates (`>`, `>=`, `<`,
`<=`) + bounded pairs. `pkRangeBounds` struct, `tryPKRangeFromWhere`,
`pkPushdownRangeScanCursor` with appropriate
`EndpointTypeRangeInclusive` / `RangeExclusive`. Symmetric
`lit OP col` flips via `flipComparisonOp`.

**ea25b61f** — pinned UPDATE / DELETE with range predicates.

**d73a38dd** — composite PK range pushdown. Leading equalities on
PK cols 0..n-2 plus equality or range on the LAST PK col.
`pkCompositeRange` + `tryPKCompositeRangeFromWhere` +
`pkPushdownCompositeRangeScanCursor`. Call-site order:
equality → single-col range → composite range → scan fallback.

**fe128ffb, 67d0e394** — removed unreachable `=` case from both
range extractors, unstaled the composite-range doc comment.

### GROUP BY alias as SELECT-list reference

**8e2b8733** — `sq.groupByAliases` map populated during GROUP BY
parsing (UPPERCASE alias → groupBy index). After GROUP BY is
parsed, walk `projCols` and `aggCols` rewriting bare refs matching
an alias to the underlying group-by column, preserving the alias as
output name. Mirror of the existing "GROUP BY references a
SELECT-list alias" path.

**5919e170** — alias-of-alias pin: `SELECT YY AS XX FROM t GROUP
BY col1 AS YY`.

**526b20ed** — case-insensitive dedup: `GROUP BY col1 AS x, col2 AS
X` now errors 42702 (was silently accepted; uppercase the key in
`seenAliases` to match `groupByAliases`).

**b2576f50, 0ab6a1bc** — alias resolves inside aggregate args:
`SELECT MAX(z) FROM ... GROUP BY x.col1 AS z`. Extended the rewrite
to touch `aggCols.aggArg` when `aggExpr == nil`. Pins for MAX with
qualified arg and COUNT(DISTINCT alias).

### Secondary-index pushdown

**34b7589c, e6630e77** — single-col MVP. `SELECT ... WHERE col =
literal` on a single-column VALUE index (non-PK) uses
`ScanIndexRecords` (index range + record fetch by PK) instead of
the full type scan. `trySecondaryIndexPushdown` walks the AND
chain for bare-col equalities and matches them against
`md.GetIndexesForRecordType`. `secondaryIndexPushdownCursor`
adapts `*FDBIndexedRecord` to `*FDBStoredRecord` via `MapCursor`.
PK pushdown still takes precedence. Guarded by
`store.IsIndexScannable` to safely fall through to full scan if
the index is WRITE_ONLY / DISABLED.

**ea9c2c79** — UPDATE / DELETE path. Shared
`trySecondaryIndexFromWhere` core; both mutation cursor sites
run it after the three PK pushdown variants via the extended
`pkPushdownCursor` helper (now takes `md` too).

**f7d5254c** — composite VALUE indexes (`ON t(a, b, …)`).
`secondaryIndexColumns` returns the ordered field list for both
`FieldKeyExpression` and `CompositeKeyExpression` shapes;
`trySecondaryIndexFromWhere` requires every index column equated
with a type-compatible literal. Tuple built in declared index-
column order. Partial composite matches fall through to scan.

**3d4f5298** — NULL-RHS pin: `WHERE col = NULL` bails on NULL at
`evalConstantAtom`, falls through to scan, which produces
UNKNOWN → empty. Same gap as PK pushdown, now closed for
secondary index too.

Conservative scope — range predicates on indexed columns and
covering-index optimisation (skip the PK fetch when all projected
columns are in the index entry) are the remaining follow-ups.

### Alias-qualified projection on single-source SELECT

**d6eab83f** — `SELECT d.id FROM t AS d` on the proto path
(`execSelectQueryFull`) previously errored 42703 because
`allFields.ByName("d.id")` doesn't match (proto fields are
unqualified). Now: if the projection is qualified and the
qualifier matches the source's `tableName` / `tableAlias`, strip
before the lookup. Enables clean end-to-end CTE composition with
`stripCTEColumnQualifiers`.

**e5dc2ee3, 3524c4a9** — pins in `wrong_qualifier.yaml` (single-
source alias-qualified projection + ON-clause RHS coverage that
corrected a stale "known gap" note — it already works).

### Qualified vs bare GROUP BY / SELECT reconciliation

**11b3a50a, 6df14a1b** — discovered while probing Java
groupby-tests.yamsql line 117: `SELECT col1 FROM (SELECT col1 FROM
t1) AS x GROUP BY x.col1` produced 2 groups (correct) but emitted
NULL for each. The grouped-row emit looked up `col1` in a
`groupColIdx` keyed only by the qualified form. Mirror shape also
broken. Fix applied in three sites (aggregateMapRows + proto-path
`groupColIdx` build; emit lookup; GR1 validation) — register the
bare last-segment alongside the qualified key, and lookups fall
back to the bare form. First-wins on bare collision.

## Next-shift follow-ups

### HIGH — worth picking up first

- [ ] **Secondary-index pushdown extensions** — MVP landed this
  shift (single-col VALUE index equality, SELECT only). Natural
  extensions: composite index keys (e.g. `ON t(a, b)` matching
  `WHERE a = X AND b = Y`), range predicates on indexed columns,
  UPDATE / DELETE sites, covering-index optimisation (skip the
  by-PK record fetch when all projected columns are in the index
  entry).

- [ ] **Intermingled schema type-filter wrapper** — to re-enable
  PK pushdown on `FieldKeyExpression` PKs, wrap the narrowed
  cursor with a type-filtering predicate (the `filterCursor`
  type in `pkg/recordlayer/cursor_combinators.go` is the right
  shape, just needs to be exported). Currently tagged as
  bail-out in `extractPKUserFields`.

### MEDIUM

- [ ] **Mutual recursion** — `WITH RECURSIVE a AS (..., b ...), b
  AS (..., a ...)`. Today CTEs are evaluated in order; `b` isn't
  bound when `a`'s recursive branch tries to read it. Fix:
  pre-register all CTE names with empty bindings, then iterate
  until all converge. Uncommon in practice.

- [ ] **Continuation resumption of partial recursive results** —
  for `maxRows: 1` style paginated retrieval. Java's
  `RecursiveStateManager` uses TempTable cursors.

- [ ] **Composite PK range on first N-1 components** — today only
  the LAST composite-PK column may carry a range; the leading
  columns must be equated. Relaxing to "all leading columns
  equated up to and including the first range" still yields a
  contiguous FDB key range. Probably 50–100 LOC.

### LOW — follow-ups from tonight's work

- [ ] ORDER BY dedup edge cases (pre-existing): `ORDER BY b, B`
  (case-folded identifiers) and `ORDER BY a.b, b` (qualified-vs-
  bare resolving to same column). Both niche; ride on a future
  identifier-folding pass.

- [ ] Aggregate expression alias in GROUP BY (Java
  groupby-tests.yamsql line 178 style): `SELECT G + 4 FROM
  (SELECT MIN(...) AS G FROM ...) AS Y WHERE G > 5`. Didn't probe
  tonight; unknown if it works.

- [ ] `GROUP BY (a+b) AS alias` — expression group keys can't be
  referenced via alias from the SELECT list because the rewrite
  path only handles bare-column group-by entries. Not a common
  pattern.
