## What to work on next

1. **#85/#86 Alias threading** — highest impact for JOIN correctness.

   **Root cause:** `mergeRows` in `pkg/recordlayer/query/executor/executor.go:700` qualifies row keys with `recordTypeName(row)` (e.g., `EMPLOYEE.ID`). For self-joins, both sides have the same type name → collision. For CTE+JOIN, the predicate uses the CTE alias (`BIG_SALES.CUSTOMER_ID`) but the map has the underlying type (`SALES.CUSTOMER_ID`).

   **Fix approach:**
   - Add `OuterAlias string, InnerAlias string` fields to `RecordQueryNestedLoopJoinPlan` (`pkg/recordlayer/query/plan/plans/nested_loop_join.go`)
   - Thread aliases from `LogicalJoin` → `translateJoin` (`cascades_translator.go:321`) → `SelectExpression` → `ImplementNestedLoopJoinRule` (`rule_implement_nested_loop_join.go`) → plan constructor
   - In `mergeRows`, use alias (when set) instead of record type name for qualified keys
   - Parser already stores aliases: `selectQuery.joins[i].alias` in `select_parser.go`
   - Tests: `TestFDB_SelfJoin` (expects 2 rows), `TestFDB_JoinOnCTE` (expects 1 row)

2. **#87 Streaming agg ordering** — enables GROUP BY + ORDER BY on group keys without index.

   **Root cause:** `ImplementSortRule` calls `ToPlanPartitions(innerRef)` → `computePartitionOrdering` → `computeWrapperRichOrdering`. The streaming agg wrapper's `HintOrdering()` DOES return group keys. The issue might be that streaming agg isn't chosen (hash agg is preferred) or that the `RichOrdering.Satisfies` comparison fails on Value equality (FieldValue pointer vs content comparison).

   **Debug:** Add a test `SELECT region FROM T GROUP BY region ORDER BY region ASC` with data. If it rejects (0AF00), add debug prints in `ImplementSortRule.OnMatch` to see partition count and ordering.

3. **#88 Reverse index scan** — enables ORDER BY DESC on indexed columns.

   **Fix:** `RecordQueryScanPlan` already has `IsReverse()`. The index scan wrapper's `HintOrdering()` should report BOTH ascending and descending orderings (one per scan direction). Then `ImplementSortRule` matches DESC against the reverse ordering and yields the reverse-scan plan. Or: create a second scan plan variant with `reverse=true` during EXPLORE.

4. **#90 strictlySorted** — correctness for DISTINCT + ORDER BY.
