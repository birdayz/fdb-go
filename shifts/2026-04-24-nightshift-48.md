## What landed (chronological)

### 1. cascades: `Comparison.Operand any → Value`

**Why:** Before, `Comparison.Operand` carried a Go-native literal. `a = b`, `a < b + 1`, `a = CAST(col AS INT)` couldn't compose — the resolver rejected non-constant RHS with an error. Constant-RHS was the only shape the seed could express.

**Shape now:**
- `Comparison.Operand` is a `cascades.Value`. Constant RHS wraps via `LiteralValue` (nil → `NullValue`, bool → `BooleanValue`, otherwise `ConstantValue`); row-context RHS is a `FieldValue` / `ArithmeticValue` / whatever the expression produces.
- `ComparisonPredicate.Eval(evalCtx)` evaluates BOTH sides against the row context. Pure dispatch is exposed as `EvalAgainst(left, right any) TriBool`.
- `Comparison.Eval(left)` stays as the plan-time entry point but evaluates RHS with a nil context (constant RHS returns its literal; non-constant RHS degrades to nil → UNKNOWN, which is correct when no row is in scope).
- `ComparisonConstantSimplifyRule` gates constant-fold on `IsConstantValue(RHS)`. `col = field` can't decide at plan time.
- `PredicateEquals` uses `valueNamesEqual` (via `ExplainValue`) instead of `reflect.DeepEqual`. Extended `valueLiteralString` to render `[]any` / `[]byte` / `int{8,16,32,int}` so the rendered form is injective over the seed types — required for IN-list distinctness.
- `Resolver.ResolveComparison` drops the constant-RHS gate. `Like`/`StartsWith` still require a string-literal pattern but wrap it in a `Value`. `ResolveIn` builds a `ConstantValue` over `[]any`.

**API additions:**
- `LiteralValue(any) Value` — wraps Go-native literal.
- `NewLiteralComparison(typ, lit)` — Comparison constructor for the common literal-RHS case.
- `Comparison.EvalAgainst(left, right any) TriBool` — pure dispatch.

**Tests pinned:**
- Non-constant RHS row-context evaluation (`age = cutoff`, `a = b + 1`) with TriUnknown NULL propagation.
- Constant-fold declined for non-constant RHS.
- Constant-fold fires when both sides are `ConstantValue`s.

### 2. logical: `buildLogicalPlanForSelectWithCatalog`

**Why:** Logical-builder ran on parse-tree text only — `LogicalFilter.PredicateText` carried canonical SQL, no `QueryPredicate` tree. Future planner work needs a real predicate tree.

**Shape:**
- `LogicalFilter` grows `Predicate cascades.QueryPredicate` alongside `PredicateText`. `Explain()` prefers the predicate tree when non-nil.
- `buildLogicalPlanForSelectWithCatalog(sq, md)` runs the text builder, then calls `buildWherePredicate` to walk the WHERE through `expr.WalkPredicate` and attach the result.
- `buildWherePredicateForTable(md, tableName, alias, whereExpr)` is the shared walker entry — takes RecordMetaData + table info, returns `(QueryPredicate, ok)`. Best-effort: any walker error / catalog miss / nil metadata returns `(nil, false)` for clean text fallback.
- DELETE + UPDATE variants land too: `buildLogicalPlanForDeleteWithCatalog`, `buildLogicalPlanForUpdateWithCatalog`. Same pattern, same fallback.
- JOINs and derived tables decline to the text path (need multi-source scope; walker has a single-source shim).

**`upgradeFirstFilter` returns bool** for observability per round-1 review. Callers explicitly `_ = upgradeFirstFilter(op, pred)` with a comment naming the invariant. `TestUpgradeFirstFilter_Invariants` pins the invariant across SELECT / SELECT+ORDER+LIMIT / SELECT+GROUP / DELETE / UPDATE shapes — future builder change that drops the Filter fires this test instead of silently swallowing predicates.

**Tests pinned:**
- Happy path: `price > 5` becomes `PRICE > 5` on the predicate tree.
- AND composition with two leaves resolved.
- `md=nil` identical to text builder.
- Catalog miss (table not registered) → text fallback.
- Walker decline (`UPPER(...)` scalar function) → text fallback.
- JOIN shape → text fallback.
- DELETE WHERE / UPDATE WHERE attach predicates to the nested LogicalFilter.

**Not wired:** `naive_generator.ExplainFn` keeps calling the text-only builder. Plumbing metadata into Explain has design tradeoffs (cold-cache non-determinism, transaction cost) — left as a future-shift item. The catalog-aware builder is the forward-path seam consumers can call where metadata is already loaded.

### 3. expr/walk: CAST / CONVERT to INT / STRING / BOOL

`walkSpecificFunction` dispatches `SpecificFunctionCallContext` (which wraps `DataTypeFunctionCallContext`). CAST and CONVERT share one path — both have `Expression()` and `ConvertedDataType()`.

`primitiveTypeToValueType` maps `INTEGER`/`BIGINT` → `TypeInt`, `STRING` → `TypeString`, `BOOLEAN` → `TypeBool`. `FLOAT`/`DOUBLE`/`BYTES`/`UUID`/`VECTOR` aren't in the seed `ValueType` enum — they surface as `UnsupportedExpressionShapeError`, fallback handles them.

The grammar's `PrimitiveType` rule has no `INT` alias — `CAST(x AS INT)` fails at the parser before reaching the walker (verified manually). Documented in commit message.

Tests: CAST(name AS INTEGER/STRING/BOOLEAN/BIGINT), CAST FLOAT/DOUBLE/BYTES decline, CAST in WHERE composition, CONVERT(name, INTEGER) shares the path.

### 4. cascades: composite-constant LHS folds

Round-2 reviewer flagged that `constantLiteral` only recognised the leaf-constant Value types. `CAST(5 AS INT) = 5` had a constant LHS but the rule didn't fire. Fix: fall through to `EvaluateConstant` for any non-leaf shape. `IsConstantValue` recurses through composites so `FieldValue` etc. still decline cleanly.

Test pins `CAST(5 AS INT) = 5` folds to `TriTrue`.

### 5. Microbenchmark

`BenchmarkComparisonPredicate_Eval_NonConstantRHS` — 42ns/op (vs 32ns/op for constant RHS). The +10ns is the second `Operand.Evaluate(evalCtx)` call. CI bench surfaces a regression if a future change adds an alloc on this path.

## Current state

**Catalog-aware builder wired into `naive_generator.ExplainFn`.** When the session schema cache holds the active schema, EXPLAIN output renders predicate trees (`PRICE > 5`) instead of canonical SQL text (`Filter(price > 5)`). Cold cache → text-builder fallback (deterministic, never blocks on a catalog fetch). Coverage: SELECT (incl. CTE / UNION / WITH), DELETE, UPDATE, INSERT … SELECT.

**Non-constant-RHS comparisons compose end-to-end.** WHERE clauses like `id = some_field`, `price < cost + tax` walk through `expr.WalkPredicate` → `cascades.ComparisonPredicate` with both sides as Values. Row evaluation reads both via `evalCtx`.

**Constant-fold covers composite LHS.** `CAST(5 AS INT) = 5`, `(1 + 2) > 0`, `CAST(1+2 AS BOOL) = TRUE` all fold at plan time via the `constantLiteral` → `EvaluateConstant` fall-through.

**Walker covers CAST + CONVERT to INT/STRING/BOOL.** Float/double/bytes/uuid/vector targets decline cleanly to the text fallback.

**LIKE … ESCAPE end-to-end.** `Comparison.Escape` rune field (zero = no escape). likeMatch handles trailing-escape rejection and adversarial escape-equals-wildcard cases (`escape == '%'` / `'_'`). Walker dispatches the ESCAPE clause; multi-char escape rejected; Explain renders `ESCAPE '\'` (or `ESCAPE ''''` for the single-quote escape). FuzzLikeMatchEscape varies the escape rune across 6 chars, ~30M execs clean.

**Multi-source JOIN scope.** `buildWherePredicateForJoins` populates the walker's scope with one ScopeSource per (primary + JOIN). Qualified refs (`Order.price`) resolve via alias; unique bare refs resolve via single-source match; ambiguous refs trip AmbiguousColumnError → text fallback. Self-join without alias (DuplicateAlias) → text fallback. 3-way JOINs work end-to-end. Found and fixed a pre-existing alias-extraction bug along the way (extractor required AS keyword; now uses GetAlias() consistently).

**MOD operator.** `cascades.OpMod` wired through walker (both `%` and `MOD` syntactic forms) and Eval (truncated-toward-zero, mod-by-zero → nil).

## Next-shift follow-ups

### HIGH

- [ ] **Derived-table WHERE.** `sq.derivedQuery != nil` still falls back to text. Needs a nested-plan source rather than a Table — different shape from JOIN. Sized 1 shift.
- [ ] **Phase 4.0 Type hierarchy port.** Replace `cascades.ValueType` enum with structured `Type` / `TypeRepository` / `Typed`. Unblocks: `CAST AS FLOAT` / `BYTES` / `UUID` in the walker, `Column` types end-to-end, real Record / List types. Sized 3–5 shifts.

### MEDIUM

- [ ] **Parameter-bound Comparison.** Support `x = ?` at plan time by letting Comparison carry a `Placeholder` on its RHS (or as the `Operand` Value when it's a `ParameterValue`). Plan-cache key needs this for query-by-value reuse.
- [ ] **Scalar function calls in walker.** `UPPER`, `LOWER`, `LENGTH`, etc. Needs a scalar function catalog (Phase 4.0). `walkSpecificFunction` already exists; just needs more dispatch arms.
- [ ] **Standalone Value simplification.** `Simplify()` only visits QueryPredicates; standalone `ArithmeticValue`s (e.g. on a SELECT-list expression with no surrounding comparison) don't fold. ComparisonConstantSimplifyRule already folds composite-constant operands inside comparisons via `EvaluateConstant`, so `(1+2) > 0` → `TRUE` works end-to-end — the gap is only Values-not-inside-predicates.

### LOW

- [ ] **Integer DIV operator.** Distinct from `OpDiv` (which is true division) — `DIV` keyword in MySQL semantics is integer-truncated. Could overlap with our existing OpDiv on int64 inputs since Go's `/` on integers is already truncating; would need investigation. Not a blocker.
- [ ] **Arithmetic over floats.** TypeFloat exists now and float literals walk through; comparisons over floats work via cmpAny's promotion. But `ArithmeticValue.Evaluate` is still int-only — mixed int/float arith returns nil (NULL-at-Value-layer). True coercion needs the Type hierarchy port.
- [ ] **`LogicalOperator → RelationalExpression` translator.** Phase 5 work — converts the logical tree built here into the cascades-side `RelationalExpression` tree the planner consumes. Today the cascades package has no `RelationalExpression` hierarchy yet (Phase 4.1).

### Inherited

- [ ] Continuation resumption of partial recursive results.
