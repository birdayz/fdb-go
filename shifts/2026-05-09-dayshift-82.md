Ported Java's `RemoveSortRule` lines 112-157:
1. **Distinct partition + ordering coverage → strictlySorted:** When a partition is distinct and all ordering values are covered by sort keys or equality-bound keys, yield plans marked as `strictlySorted`.

## Known issues / tech debt

- `GetEqualityBoundValues` uses stricter semantics than Java (all-fixed vs any-fixed). No practical impact with current index scan bindings but worth aligning if multi-binding scenarios emerge.
- `strictlySorted` flag is set by ImplementSortRule but not yet consumed downstream (union/intersection plans don't check it). Will become active when those plans add dedup optimization.
- In simple cases (single index satisfying ORDER BY), the exploration-phase `SortOverOrderedElimRule` eliminates the sort before `ImplementSortRule` runs, so strictlySorted is never set. The flag only fires in complex multi-candidate scenarios where the sort survives to the PLANNING phase. This matches Java's architecture (RemoveSortRule is an ImplementationCascadesRule, not an exploration rule).
