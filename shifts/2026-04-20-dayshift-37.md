### 1. Multi-source `SELECT <qualifier>.*` — commit `635a2810`

Pre-fix the parser accepted `SELECT a.*` but the evaluator coerced it to unqualified `SELECT *`. On a JOIN or comma-join that leaked all columns from all sources instead of restricting to the aliased source, and a wrong qualifier (`SELECT nope.* FROM a`) silently succeeded — both noted as known gaps by nightshift-36.

- Added `projQualifier string` field to `selectQuery`. The parser captures the qualifier from `SelectQualifierStarElementContext.Uid()` and stores it verbatim; schema-free at parse time, resolved at execution.
- `execSelectJoin` (the path that handles both explicit JOINs and comma-joins) now looks up the qualifier against `tableAlias` and each `joins[i].alias` via a new `resolveQualifierColumns` helper. The helper pulls the matching source's column list from the record type descriptor — or, when the source names a CTE, from the CTE's stored column list. Row-value lookup in the projection loop prefers the `"alias.col"` key so two sources with overlapping column names don't collide on whichever wrote the unqualified key last.
- `execSelectQueryFull` (single-source, proto-backed) and `execSelectFromCTE` (single-source CTE) validate the qualifier matches the table's alias or name and otherwise raise SQLSTATE 42F01 (`ErrCodeUndefinedTable`). The no-FROM path rejects any qualifier with the same code.
- `qualified_star.yaml` — new scenario with 6 assertions: single-source `a.*`, comma-join `a.*`, comma-join `b.*`, aliased source `x.*` (where `x` = alias of `a`), wrong qualifier on single-source (42F01), wrong qualifier on multi-source (42F01).

### 2. RIGHT JOIN NULL-pad works on empty left table — commit `564b1e5e`

nightshift-36's RIGHT JOIN NULL-padding fix sampled a runtime left row to learn the column keys that need to be NULLed on unmatched-right rows. That works when the left side has any rows at all, but fails when the left table is entirely empty — no runtime row means the qualified `a.id` key is never discovered, so `SELECT a.id` on an unmatched right row falls through to the unqualified `id` key populated from the right and returns b.id instead of NULL.

- Added `collectLeftJoinKeys(md, leftSources)`: for every source merged into the nested-loop accumulator so far, iterate its record type descriptor (or CTE column list) to collect both the unqualified column name and the `alias.col` variant. De-duplicated with a `seen` set.
- Seeded `leftSources` with the top-level FROM source `(sq.tableName, sq.tableAlias)` and appended `(jc.tableName, jc.alias)` after each JOIN iteration completes. The RIGHT JOIN step reads the current `leftSources` slice (which reflects every source that contributed to `joined` at that point, including the chained `a RIGHT JOIN b RIGHT JOIN c` case) and builds keys from metadata.
- Extended `outer_join.yaml` with a new `a_empty` table and a `RIGHT OUTER JOIN` against populated right-table rows; pre-fix all three unmatched rows returned b's id instead of NULL.

### 3. TODO.md — commit `dd782214`

Marked both follow-ups `[x]` with links to the pinning scenarios. The qualified-star entry now also notes the one remaining sub-gap (mixed `SELECT a.*, b.label`) that's out of scope and tracks the work it would need (expanding qualified-star to its resolved columns before the projection loop).

### 4. CONCAT NULL-skip comment — commit `b0f11a62`

While auditing the handover's "CONCAT / REPLACE NULL semantics" follow-up I noticed the inline comment in `execFunction` claimed NULL-skip was "per SQL standard". That's wrong — the SQL standard / Oracle / SQL Server propagate NULL through concatenation; only MySQL and Postgres's `CONCAT()` skip. Fixed the comment to accurately describe the behaviour and cross-reference `trim_concat.yaml`. No behaviour change — Java-side conformance question stays open pending the Phase-B harness.

### 5. INFORMATION_SCHEMA SELECT-list projection — commit `6b82096f`

Opportunistic fix while auditing the handover's INFORMATION_SCHEMA follow-up. The system-tables route dropped the SELECT list entirely — `SELECT TABLE_NAME FROM "INFORMATION_SCHEMA"."TABLES"` returned all 10 TABLES columns instead of just TABLE_NAME. Handlers always emit their full column set; fine with `SELECT *`, silently wrong for any named projection.

- Added `projectSystemRows(driver.Rows, *selectQuery)` at the dispatcher level: picks up after the handler returns a `*staticRows` and applies SELECT-list projection, aliases, ORDER BY, LIMIT/OFFSET. Column-name matching case-insensitive (matches filterSysRows). Unknown column → 42703 (UndefinedColumn); computed expressions rejected cleanly (system tables are a data view, not a query plan target).
- New `information_schema.yaml` (5 tests): narrowed TABLE_NAME projection, exact-match WHERE, COLUMNS filter via TABLE_NAME, aliased projection, unknown-column 42703.
- Also found during this audit: CREATE TABLE preserves identifier case. `CREATE TABLE emp` stores as `emp`; `WHERE TABLE_NAME = 'emp'` matches correctly. Nightshift-36's "0 rows for 'EMP'" observation was likely a scenario-specific stale claim rather than a broader bug.
- Still open: the parser rejects unquoted `FROM INFORMATION_SCHEMA.TABLES` because `TABLES` is reserved. Workaround is quoting (`"TABLES"`). Proper fix is grammar work and out of scope for this shift.

### 6. EXISTS / NOT EXISTS harness — commit `7b5485a1`

EXISTS and NOT EXISTS are supported in all four evaluator paths (proto WHERE, map WHERE/JOIN, CTE WHERE, HAVING) for uncorrelated subqueries, but no conformance scenario pinned the behaviour. Seven assertions in `exists.yaml` cover: empty inner table → EXISTS FALSE / NOT EXISTS TRUE, state change after INSERT flips both, uncorrelated self-reference on the same table, and HAVING EXISTS against an aggregate.

### 7. Derived-table harness — commit `e5ebd25e`

FROM subquery AS alias works (via `execSelectQuery` materialising the inner into an in-memory CTE) but was unpinned. Five assertions in `derived_table.yaml` cover: basic projection, aggregation inside the inner query with outer-query WHERE filter, missing-alias 42601 at the grammar level, outer-column alias, and the known limitation that a derived table as the *extra* FROM source (after a comma) errors with 0A000 — derived tables as the *first* source work fine.

### 10. UNION column-count arity check — commit `bd0fc3f3`

Harness surfaced a real bug: `execUnion` never checked that the two sides had matching column counts. `SELECT id, v FROM a UNION SELECT id FROM b` (mismatched) silently accepted and stitched rows of inconsistent widths — the driver's scan path or downstream aggregation would panic or return wrong answers. Added `len(leftCols) != len(rightCols)` check rejecting with SQLSTATE 42601. New `union_columns.yaml` scenario (4 assertions): positional name matching, mismatched-arity error, and a pin on the separate known bug that `ORDER BY col` on a UNION plumbs through to each side's SELECT and errors 22023 when names differ — TODO'd.

### 11. DISTINCT on JOIN path — commit `5e5f25f0`

Second real bug the harness surfaced: `execSelectJoin` was silently dropping the SELECT DISTINCT clause. `SELECT DISTINCT a.cust_id FROM a, b WHERE a.cust_id = b.cust_id` returned the full cross-join with duplicates. Pre-fix DISTINCT only worked on single-source FROM (`execSelectQueryFull` at line 1785). Added the same dedup pass before ORDER BY in `execSelectJoin`, guarded against aggregate queries and COUNT(*). New `distinct_join.yaml` (3 assertions) + `count_distinct_join.yaml` (3 assertions — COUNT(DISTINCT) over JOIN was already correct, pinned as a regression guard).

### 12. RIGHT JOIN double ON-evaluation — commit `a7120385`

Reviewer's observation #7: the RIGHT JOIN block re-iterated every (left, right) pair solely to populate `matchedRight[]`, re-running the ON predicate that the main INNER/LEFT matching pass had already evaluated. For a RIGHT OUTER JOIN on a populated left side this doubled the ON-evaluation cost. Fix: allocate `matchedRight[]` once at the top of the jc iteration (only when joinType is RIGHT), set `matchedRight[ri] = true` during the existing matching pass whenever ON passes, drop the second nested loop. O(n*m) evaluations now, down from 2*O(n*m). No behaviour change; pinned by existing RIGHT OUTER JOIN scenarios in `outer_join.yaml`.

### 9. Review-feedback refactor — commit `05d4ad35`

Review turned up 6 actionable items (2 of 8 were positive observations). Addressed every one:

- Single-source `execSelectQueryFull` qualified-star check now delegates to `resolveQualifierColumns` (ignoring the returned columns; for a single-source query `a.*` resolves to the same set as `*`). Collapses two copies of the alias-matching rule into one so the paths can't silently diverge. Moved into the `runInTx` closure where `md` is in scope.
- `execSelectFromCTE` keeps its inline check (no `RecordMetaData` in that path — the CTE-only route never consults the schema) but now documents *why* it doesn't share the helper.
- Added a comment on the projection loop's unqualified-key fallback: `scanTableToMaps` always populates both `alias.col` and `col`, so the fallback is dead on real rows today and present only as a safety net for future map producers.
- `qualified_star.yaml` gained an explicit `FROM a INNER JOIN b ON ...` scenario. The comma-join path and INNER JOIN path share projection code today, but a dedicated case pins the contract against refactors that might split them.
- `outer_join.yaml` gained the intersection of both fixes: `SELECT a.*` on an empty-left RIGHT OUTER JOIN. Both the qualifier resolver and the NULL-padder work from metadata, so no runtime row is ever inspected.
- `TODO.md` tracks the pre-existing O(n²) double ON-evaluation in `execSelectJoin`'s RIGHT JOIN block (reviewer flagged it — memoising the first pass would fix it).

## Known issues / tech debt (surfaced this shift)

- **Mixed qualified-star + named columns** (LOW) — `SELECT a.*, b.label FROM a, b` still errors with "cannot mix qualifier.* with named columns in SELECT list". The parser short-circuits any qualifier-star element when `len(elems) > 1`. Supporting this requires expanding `a.*` to its resolved column list during parse (needs schema access) or during execution (replace the qualifier entry with its columns before the projection loop runs). Not blocking typical queries; flagged in TODO.md.
- **Wrong-qualifier path for INFORMATION_SCHEMA tables** (LOW) — `SELECT X.* FROM INFORMATION_SCHEMA.TABLES AS X` falls through the system-tables route and ends up in `execSystemTable` which ignores `projQualifier` entirely (it returns a fixed column list). Not tested; probably silently projects everything. INFORMATION_SCHEMA query gaps are already tracked as a separate follow-up.
- **CONCAT / REPLACE NULL semantics** (LOW) — still unresolved. Reading Java's fdb-relational-core grammar shows `CONCAT` / `CONCAT_WS` are reserved keywords but no dedicated evaluator exists anywhere in the tree; Java likely doesn't implement SQL CONCAT at all. That makes our behaviour a pure Go design choice — the Phase-B round-trip harness is still the right way to settle it definitively.

## Next priorities (for swingshift-38)

In descending ROI order:

1. **Aggregate over arithmetic expression**. `SUM(qty * price)` / `AVG(col / 2)` / `MIN(CASE ...)` currently error with 42703 because `extractAggFunc` resolves the inner expression via `selectExprToColumnName` (bare column refs only). Java and Postgres both accept arbitrary expressions in aggregates — this is one of the most common SQL idioms the Go layer still rejects. Fix: capture the raw expression context in `aggSelectCol`, evaluate per-row before folding into the aggregate state. Pinned as known-broken in `aggregate_expr.yaml`. Medium — ~80 lines incl. tests.

2. **Aggregate + expression ORDER BY on UNION / GROUP BY / DISTINCT**. SQL-standard says ORDER BY on a UNION refers to the result schema (left side's column names). Today the parser plumbs ORDER BY through to each side's SELECT, so `SELECT id, v FROM a UNION SELECT id, w FROM b ORDER BY id, v` errors 22023. Fix: sort the combined row set *after* `execUnion` returns. Pinned in `union_columns.yaml`.

3. **Mixed qualified-star + named columns**. `SELECT a.*, b.label FROM a, b` — expand qualifier to its resolved columns before the projection loop. The `resolveQualifierColumns` helper added this shift gives you the column list; the remaining work is re-threading `projCols`/`projAliases`/`projExprs` building to allow mixed elements in the SELECT list (currently any other element alongside a qualifier-star is rejected by the parser). Medium — touches the parser's element loop and potentially `execSelectJoin` / `execSelectQueryFull` projection. ~100-150 lines.

4. **GROUP BY on expressions**. `GROUP BY amt / 100` / `GROUP BY col + 1` currently rejected with 0A000 because `extractFromSimpleTable` uses `columnNameFromExpr` (column names only). Java and Postgres both accept expression GROUP BY. Fix analogous to #1 but on the GROUP BY path. Pinned in `group_by_multi.yaml`.

5. **INFORMATION_SCHEMA parser gap (unquoted identifiers)**. The SELECT-list projection is fixed (commit `6b82096f`). What's left is grammar-side: unquoted `FROM INFORMATION_SCHEMA.TABLES` is rejected because `TABLES` / `COLUMNS` / `INDEXES` / `SCHEMATA` are reserved keywords. Workaround (double-quoting) works today. Proper fix: extend the grammar to allow these identifiers in qualified-table position after `INFORMATION_SCHEMA.`. Non-trivial parser work (~100-150 lines incl. tests).

6. **Java-side round-trip harness (Phase B)**. Unchanged from nightshift-36's priority #3. Wire `fdb-relational-core` Maven deps into MODULE.bazel, extend `conformance_server.java` with a `SqlSteps` class, let the Go runner route each yamsql test through both engines. This is the piece that would catch wire-format drift and settle the CONCAT NULL semantics question without guessing. ~4-6 h — good swingshift or dayshift scope since the Maven wiring is mostly Bazel chore.

7. **Continue extending the harness**. 48 scenarios today, up 10 from start-of-shift. Still-unported yamsql categories: arrays, function DDL (`CREATE FUNCTION`), case-sensitivity options, bitmap-aggregate indexes. Every scenario added this shift either pinned a working contract or surfaced a real bug (UNION arity, DISTINCT-on-JOIN, aggregate-over-expression, GROUP-BY-on-expression).

8. **RIGHT OUTER JOIN + WHERE / GROUP BY combinations**. `outer_join.yaml` pins the bare JOIN cases plus empty-left-table and a.* intersection this shift; deeper combinations (OUTER + WHERE + GROUP BY + HAVING) remain unpinned. Cheap to extend once someone is in that area.

9. **ORDER BY NULL-direction on aggregate queries with the proto path**. Low priority — nightshift-36 centralised `orderByLess` through a helper, so the behaviour is consistent across map / CTE / JOIN paths; the proto-backed aggregate path still has its own `sort.Slice` call that could fall out of sync. Worth a shared helper pass next time someone is there.
