## Key decisions

- **Batch insert is Go-only API** — Java has no equivalent. Wire-compatible (same FDB mutations). Equivalence test proves identical output to N×SaveRecord.
- **`sync.Map` for `maintainerCache`** — attempted `map[string]` optimization, reverted after concurrent map panic in chaos tests. `sync.Map` overhead (~ns) negligible vs FDB round trips.
- **hyperpb-go not needed** — codebase already uses vtprotobuf (`MarshalVT`/`UnmarshalVT`/`SizeVT`) on all hot paths. Standard `proto.Marshal` is fallback only.

## Benchmark accuracy improvements

Current benchmarks share the Ginkgo testcontainer and use `-benchtime=5s`. To get more accurate results:

- **Alloc counts are already deterministic** — same code path = same allocs. The variance (e.g., 100 vs 109) is iteration-count rounding / GC timing, not real variation. Allocs are the primary optimization metric.
- **ns/op is noisy** — dominated by FDB round trip to testcontainer, not by our code. To reduce noise:
  - Increase `-benchtime=30s` (more iterations average out variance)
  - Use `-count=10` (multiple runs show distribution)
  - Use `benchstat old.txt new.txt` for statistically valid comparison with confidence intervals
  - Pin `GOMAXPROCS=1` to reduce goroutine scheduling noise
- **Fresh container per run is NOT needed** — the noise is from network/Docker scheduling, not FDB server state. A fresh container adds ~4s startup for no accuracy gain.
- **Best practice for before/after comparison:**
  ```sh
  # On master (before):
  bazelisk run //pkg/recordlayer:recordlayer_test -- \
    -test.bench='^BenchmarkSaveRecord$' -test.run='^$' \
    -test.benchtime=30s -test.count=10 | tee /tmp/old.txt

  # On branch (after):
  bazelisk run //pkg/recordlayer:recordlayer_test -- \
    -test.bench='^BenchmarkSaveRecord$' -test.run='^$' \
    -test.benchtime=30s -test.count=10 | tee /tmp/new.txt

  # Compare:
  go install golang.org/x/perf/cmd/benchstat@latest
  benchstat /tmp/old.txt /tmp/new.txt
  ```

## SaveRecord profiling results (remaining alloc breakdown)

Profiled with `-memprofile` on `BenchmarkSaveRecord` (100 allocs). The remaining allocations are structural — they're the FDB client doing its job, not avoidable without architectural changes:

| Source | % of allocs | What it is | Can optimize? |
|---|---|---|---|
| `transport.ReadFrame` | 14% | FDB wire protocol framing | Pool frame buffers |
| `loadRecordStoreState` | 27% (cumulative) | Store Open — pipelined range reads | Cached across txns already |
| `ryw.set` (defensive copy) | ~5% | RYW cache copies serialized record value | No — needed for read-your-writes correctness |
| `addReadConflict` / `addWriteConflict` | ~7% | Conflict range keys per Get/Set | Could pool read conflict buf (write already pooled) |
| `newPacker` | ~2.5% | Tuple packer pool misses | Pool is already in place |
| `snapshotCache.insert` | ~3% | Server read cache for repeated reads | Structural |
| `FDBStoredRecord` allocation | ~1.5% | Result struct returned to caller | Can't avoid |

**Actionable items:**
- Pool read conflict buffers (same pattern as write conflicts in PR #59) — saves ~2-3 allocs per Get
- Pool transport frame read buffers — saves ~1 alloc per FDB round trip
- Everything else is structural floor
