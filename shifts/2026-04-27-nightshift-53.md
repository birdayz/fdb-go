### Bug fix found via fuzz extension

While extending `FuzzRecordMetaDataFromProto`'s seed corpus with the metadata shapes A2 sends across the wire (VALUE / COUNT / SUM indexes), the fuzz target found a real panic in `RecordMetaDataBuilder.Build`:

```
panic: hash of unhashable type: []uint8
  ...metadata.go:578 (RecordMetaDataBuilder.Build)
  ...metadata_proto.go:211 (RecordMetaDataFromProto)
```

**Root cause**: `normalizeSubspaceKey` returned its `any` argument unchanged for unknown types. A `[]byte` subspace key reached `indexSubspaceKeySeen[normalizeSubspaceKey(...)]` and Go panicked because slices aren't hashable.

**Fix**: added `case []byte: return string(k)` to `normalizeSubspaceKey`. Reviewer's non-blocking observation also added `case tuple.Tuple` for nested-tuple keys (preemptive — Java doesn't currently emit those).

**Regression seed** `6b2b0fc91474e6b9` checked into `pkg/recordlayer/testdata/fuzz/FuzzRecordMetaDataFromProto/`.

### Documentation

CLAUDE.md gained three new entries under the "Cross-language metadata wire-format gotchas" section:

1. **`parseFrom(bytes)` MUST use `EXTENSION_REGISTRY`** registering `RecordMetaDataOptionsProto.registerAllExtensions(...)`. Without it, `RecordMetaData.build(proto)` raises "Union descriptor is required" — the `[record].usage = UNION` extension drops, Java can't identify the union message by name.
2. **Java-loaded `RecordMetaData` produces `DynamicMessage`**. `Order.newBuilder().mergeFrom(rec.getRecord())` fails with "can only merge messages of the same type". Workaround: `Order.parseFrom(rec.getRecord().toByteArray())`.
3. **Java's `IndexTypes.COUNT` requires `GroupingKeyExpression(field, 0)`, NOT `field.groupBy(empty())`**. The intuitive form puts the field in the GROUPED slot, which COUNT rejects with `InvalidExpressionException`. Symmetric for SUM: `GroupingKeyExpression(field, 1)` (1 means the field is the aggregated part).

### A2 closure (CRITICAL, OPTIONAL — depends on appetite for keyspace refactor)

The remaining A2 scope is the SchemaTemplateCatalog wire format. To do it:

1. Migrate Go sqldriver from `RelationalKeyspace.CatalogSubspace()` (three-string) to `DefaultCatalogSubspace()` (Java-compat tuple). This touches `pkg/relational/sqldriver/driver.go:147` and possibly downstream code.
2. Or: add a separate Java step that reads Go-written templates from the three-string subspace AND reads Java-written templates from the tuple subspace, and compare wire formats at the byte level (sidesteps the keyspace divergence by acknowledging it).

The first option is the proper fix; the second is a workaround that pins the divergence.
