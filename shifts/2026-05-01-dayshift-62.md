### 3. Root cause: Java conformance server stderr pipe blocking

**The bug**: Cross-engine harness sometimes hung at exactly 120s on negative-path entries, in unpredictable patterns. Per-entry isolation (fresh Java server per entry) was the existing workaround, but only fired for `ExpectErrorContains`, not `ExpectErrorMessage`.

User pushed back on the "Java is poisoned" hypothesis — "find the root cause." Investigation:
- Probed direct: 3 consecutive negatives on one server → 1.2s, no hang.
- Probed volume: 50 positives + 1 negative on one server → 130ms, no hang.
- Probed full sequence: 186 corpus entries on one server → 5 entries hung at exactly 120s each (each at the FIRST negative after a long run of positives).
- Captured `jstack` during the hang: Java JVM fully idle (zero active workers, 22 threads all waiting). The HTTP request never reached Java's worker pool.
- Tried `DisableKeepAlives=true`: same hang. So it's not connection-state.

Real cause (found by reading code): `e.printStackTrace()` in `conformance_server.java#handleInvoke` writes 5–10 KB to stderr per error. `JavaInvoker.startJavaServer()` reads stderr ONCE at startup, then never drains it. Linux pipe buffer is 64 KB — after ~6–7 errors accumulate, Java's NEXT `printStackTrace` BLOCKS on the stderr write. The request handler thread is stuck mid-stack-trace; the HTTP response never sends; Go's POST hangs at the 120s response timeout.

**Fix**: Drain Java's stderr to `os.Stderr` in a background goroutine for the process lifetime. Java's stack traces stay visible during debugging. Per-entry isolation is no longer needed — full corpus completes in 28s with positives and negatives interleaved on one shared server.

The pre-existing per-entry isolation got removed too (it was a hack that masked the real issue without fixing it).

### 4. Java conformance server JSON encoder for IEEE-754 specials

After aligning float-divide-by-zero to Java's IEEE-754 (return ±Infinity / NaN, no throw), the cross-engine harness still failed: Java's response `{"rows":[[Infinity]]}` isn't valid JSON. Gson's `JsonPrimitive((Number))` emits bare `Infinity`/`-Infinity`/`NaN` tokens that Go's HTTP client rejects at unmarshal.

Fixed in `sql_plan_steps.java#encodeValue`: detect IEEE-754 specials and encode as strings (`"Infinity"`, `"-Infinity"`, `"NaN"`). Mirror in Go's `coerceForComparison` (`go_runner.go`): `math.Inf(±1)` and `math.NaN()` → same strings. Symmetric encoding keeps cross-engine row comparison byte-equal.

Two new corpus entries pin the path: `double_divide_by_zero_returns_infinity` (5.0/0.0 = +Infinity), `double_zero_div_zero_returns_nan` (0.0/0.0 = NaN).

### 6. WITH RECURSIVE non-self-ref alignment
