## What was done

### DecorrelateValuesRule: 3 real bugs fixed

Ported 3 missing Java negative tests that exposed real Go bugs:

1. **DoNotUseValuesBoxWithPredicates** — values box with predicates (even tautologies like TRUE) was incorrectly accepted as a values box. Fixed: reject when `len(childSel.GetPredicates()) > 0`.

2. **DoNotTreatRangeTwoAsValues** — values box over `range(2)` (cardinality 2, not 1) was incorrectly accepted. Fixed: added `isRangeOneQuantifier()` that evaluates RangeValue constant arguments and verifies cardinality == 1.

3. **DoNotTreatRangeWithConstantObjectValueAsValueBox** — values box over `range(ConstantObjectValue)` was incorrectly accepted because the COV doesn't correlate to the child quantifier. Fixed: `isRangeOneQuantifier` requires all-constant range arguments.

Test infrastructure updated: `makeValuesBox`/`makeRangeOneQ` now use `TableFunctionExpression(range(0,1,1))` instead of `FullUnorderedScanExpression`, matching Java's actual `range(1)` source.

### NormalizePredicatesRule existential guard investigation

Investigated removing the existential quantifier guard. Root cause: removing it causes **planner non-convergence** (MaxTasks cap hit at 10k). The NLJ rule successfully fires on normalized expressions and yields physical plans, but cascading rule interactions (NormalizePredicatesRule → PredicatePushDownRule → re-normalization) create an infinite exploration loop. The NLJ plans ARE yielded correctly but the planner never converges to extract them.

Fixed: requires rule deduplication infrastructure (tracking which normalizations have been applied). Updated DIVERGENCES.md with precise root cause. Separate improvement: NormalizePredicatesRule now uses `NewSelectExpressionWithAliases` to preserve source aliases through CNF normalization.
