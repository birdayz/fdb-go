### Aggregate engine (the biggest theme — 11 commits)

- **Aggregate-over-expression (`SUM(qty*price)`)** — `9d1cb73c`. extractAggFunc classifies the FunctionArg as bare-column ref vs arbitrary expression; aggExpr field on aggSelectCol; both proto and map evaluation paths widened symmetrically. Pinned by 11 new assertions in `aggregate_expr.yaml`. (P1.)
- **GROUP BY on arbitrary expressions** — `124337c5`. groupByExprs parallel slice; per-row evaluation. Pinned by 6 scenarios in `group_by_multi.yaml`. (P4.)
- **GROUP BY N positional reference** — `4bd6d415`. Reuses resolveSelectListPosition helper from ORDER BY N.
- **GROUP BY expression repeated in SELECT list** — `e24bf770`. Proto aggArgFDs resolver tripped over the synthetic CASE-text name; new groupExprByName map skips FD lookup for expression groups.
- **HAVING accepts math / bit atoms over aggregates** (`SUM(v) * 2 > 50`) — `637bfbba`.
- **HAVING-only aggregates** — `0dda30cf`. `SELECT grp FROM t GROUP BY grp HAVING SUM(v) > 0` previously dropped GROUP BY entirely. Now harvestAggregates walks HAVING and adds matching aggCols entries with the new `hidden` flag.
- **HAVING on bare COUNT(*) (fast path)** — `b814fdaf`. The countStar fast path in both aggregation paths skipped HAVING entirely.
- **SELECT expressions wrapping aggregates** (`SUM(a)+SUM(b)`, `COALESCE(SUM(v),0)`, `CASE WHEN COUNT(*)>0 THEN ...`) — `28705198`. New aggSelectCol.outExpr field for post-aggregation evaluation; harvested aggregates added as hidden accumulators; evalExprAtomOnMap now resolves AggregateFunctionCall via rowMap lookup. Round-7 reviewer caught a novel-aggregate harvest bug; fixed in `cff81d84`.
- **Constants alongside aggregates** (`SELECT 1, SUM(v) FROM t`) — `be18eca2`. Both in-loop and after-loop reclassification paths now route column-free expressions to outExpr instead of attempting to look up "1" as a field.
- **Mixed-agg path demoting `SUM(a)+1` to group-by ref** — `0650fa83`. Pre-fix, any computed expression encountered after the first aggregate was treated as a group-by reference; expression-with-aggregates now routes to outExpr.
- **ORDER BY on aggregates not in SELECT list** — `40573855`. New `sortOnly` aggSelectCol flag (alongside `hidden`); orderByClause.rawExpr preserves the AST so the harvest can walk it. sortOnly entries are emitted into the row for sort to find via colIdx, then stripped after sort+limit. countStar fast path is no-op via dedicated guard.
- **ORDER BY expression wrapping aggregate** — `b290bfa6`. Extension of the sortOnly path: when ob.expr contains aggregates (e.g. `ORDER BY SUM(v) * 2`), add a sortOnly outExpr aggCols entry for the wrapping expression itself; the two-pass emit computes its value into the sentinel slot. The "ORDER BY expression only supported in CTE/JOIN" error path no longer fires for aggregate queries.
- **HAVING + ORDER BY referencing the same un-projected aggregate** — `3670cfac`. Dedup branch in the harvest now upgrades hidden → sortOnly when ORDER BY references an aggregate already harvested by HAVING. sortOnly subsumes hidden (still in rowMap for HAVING, also in cols/data for the sort, stripped post-sort).
- **AWF field extraction refactor** — `a584b0a2`. Shared extractAwfFields helper between extractAggFunc + aggColFromAwf. (Round-3 reviewer maintenance note.)

### Bugfixes from extending coverage

- **LEFT OUTER JOIN + WHERE / aggregate on unmatched-left rows** — `427ceed4`. Two real bugs surfaced by extending outer_join.yaml: (1) unmatched-left rows had no right-side keys at all, so WHERE/HAVING couldn't even find the column to compare against NULL; (2) aggregateMapRows's bare-column aggArg fallback silently substituted the other side's column value when the qualified key was explicitly NULL.
- **INSERT INTO t SELECT \<expr\> positional mapping** — `aab6aebb`. SELECT projection text like "id+100" isn't a real table field. SQL-92/Postgres semantics: map by position when no explicit column list given.
- **INSERT INTO bigint_col accepts whole-valued float** — `c5998eb7`. SUM/AVG accumulators produce float64; convertToProtoValue's Int64Kind branch now coerces whole-valued floats. NaN/Inf/fractional still error cleanly.
- **DISTINCT keyed off extraSortFields** — `d528f852`. `SELECT DISTINCT v FROM t ORDER BY id` silently leaked duplicates because the dedup keyed off the full row buffer (projected + extra sort columns). Now keys off the projected width only.
- **CTE ORDER BY on non-projected column** — `9d177ac3`. The CTE sort comparator's colIdx only contained projected colNames; ORDER BY on other CTE columns silently no-op'd. Falls back to the materialised mapRows lookup when colIdx misses.
- **CTE ORDER BY on typo'd column** — `82bd4382`. Round-10 reviewer follow-up: the mapRows fallback returned nil for both sides on a typo, which compared "equal" → stable order, masking the typo. Pre-validation pass now errors 42703.
- **JOIN ORDER BY on non-projected column** — `9500c512`. Same shape as the CTE bug. Sort falls back to filtered[i] map; pre-validation errors 42703 on typo.
- **Aggregate-path ORDER BY typo** — `9c171f4d`. Third in the family — proto aggregate-path sort had the same silent-no-op-on-typo behaviour. Pre-validation against colIdx (visible cols + sortOnly sentinels) errors 42703.
- **SELECT DISTINCT against a CTE** — `3c60f1ad`. execSelectFromCTE had no DISTINCT branch at all (only the proto and JOIN paths had one). Every duplicate row leaked through. Now matches the others' rowKey-on-projected-cols semantic; mapRows trimmed in lockstep.
- **SELECT-list expression alias matches GROUP BY expression** — `5b3191b5`. `SELECT v/10 AS bucket ... GROUP BY v/10` errored 42703 because the HAVING-harvest reclassification used the alias as groupCol; proto path tried to look it up as a field. Now matches projExpr.GetText() against groupBy[] and points groupCol at the GROUP BY entry's text.
- **GROUP BY references SELECT-list alias** — `dd654102`. `GROUP BY bucket` where bucket is `v/10 AS bucket` errored 42703. Postgres / MySQL accept this; we now do too. Added selectAliases/Exprs snapshots captured before reclassification clears them; GROUP BY parser redirects alias hits; new post-GROUP-BY pass rewrites aggCols groupCol to match the GROUP BY text.

## Known issues / tech debt (surfaced this shift)

- **Scalar subqueries** (`SELECT (SELECT MAX(v) FROM t) FROM t2`) — parser rejects 42601. Grammar work; not attempted.
- **Correlated subqueries** (`WHERE EXISTS (SELECT 1 FROM b WHERE b.aid = a.id)`) — subquery evaluator doesn't see the outer row's context; errors 42703.
- **`||` is OR, not concatenation** — matches Java's grammar (logicalOperator includes '||'). Use CONCAT() for string concatenation. Documented in the language reference (none yet).
- **Unary minus before column** (`SELECT -v`, `SELECT ABS(-v)`) — grammar rejects 42601. Workaround: `0 - v`. Grammar work.
- **`INFORMATION_SCHEMA.TABLES` unquoted** — TABLES/COLUMNS/INDEXES/SCHEMATA are reserved keywords; quoting works (`"TABLES"`).
- **CONCAT NULL-skip vs NULL-propagation** — current behaviour (skip) matches MySQL/Postgres CONCAT; Java behaviour unverified. Phase-B harness would settle.
- **OFFSET without LIMIT** — grammar requires LIMIT. Same as Java.
- **CURRENT_TIMESTAMP / CURRENT_DATE** — not implemented; errors 0A000.
