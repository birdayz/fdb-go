## TL;DR

**The cascades planner port advanced from "no RelationalExpression hierarchy" to "31-rule logical optimiser running on a memo with sub-Reference descent and SemanticEquals dedup fallback".** This shift closed B1 (RelationalExpression hierarchy), B3 (Reference / Memo seed), most of B5 (logical-rewrite rules), and the C1 PlanGenerator seed (LogicalOperator → RelationalExpression adapter). The Reference.Insert dedup contract was upgraded from pointer-identity-only to two-tier (pointer-identity fast path + SemanticEquals fallback gated on hash equality), which unblocked the entire push-through-X / pull-above-X rule family. FixpointApply gained sub-Reference descent so rules compose across Quantifier boundaries. End-to-end fuzz coverage on the C1→B5 pipeline (FuzzConvertAndOptimise, 96M execs/120s clean).

## What was done

### B1 — RelationalExpression hierarchy (fully shipped)

New `pkg/recordlayer/query/plan/cascades/expressions/` package with:
- `RelationalExpression` interface (GetResultValue / GetQuantifiers / CanCorrelate / ChildrenAsSet / GetCorrelatedToWithoutChildren / EqualsWithoutChildren / HashCodeWithoutChildren).
- `Quantifier` (ForEach kind + NamedForEach for explicit aliases).
- `Reference` with two-tier dedup (see B3 below).

- 11 concrete expressions: 8 logical (Filter, Projection, Sort, TypeFilter, Distinct, Union, Intersection, Select) + 3 DML (Insert, Update, Delete) + 1 leaf (FullUnorderedScan).
- Correlation walking via `values.GetCorrelatedToOfValue` / `predicates.GetCorrelatedToOfPredicate` free helpers.

Originally sized 4–6 shifts; landed in 1.

### B3 — Reference / Memo / PlanContext seed

- `Reference` with multi-member equivalence-class dedup. **Two-tier contract** (commit 680e664a + 6f6150b7):
  1. **Fast path**: `EqualsWithoutChildren` + pointer-identity on every Quantifier's child Reference. Hits when a rule reuses input Quantifiers (the pattern most seed rules follow). O(1).
  2. **Fallback**: `SemanticEquals` walk gated on `HashCodeWithoutChildren` equality. Catches the case where a rule yields output wrapping a fresh inner Reference whose held expression is structurally equivalent to an existing member's child Reference. Without this fallback, rules like `PushFilterThroughDistinctRule` non-terminate (each fire adds a fresh-Reference duplicate).
- `PlanContext` interface + `EmptyPlanContext` singleton + `PlannerConfiguration` struct + `MatchCandidate` placeholder (cascades root).
- `ExpressionRuleCall` parallel to the existing Predicate/Value RuleCall — Yield inserts into a Reference; dedup absorbs structurally-equivalent re-yields; Yielded() records intent independent of dedup outcome.
- `FixpointApply` with **sub-Reference descent** (commit c4123d15) — each iteration walks the tree from the root through every reachable Reference (collected once per pass via collectReferences, deduped by pointer), firing every rule on every Reference. Rules now compose across Quantifier boundaries; previously they only fired on the top-level Reference's members.

Originally sized 2–3 shifts; landed in 1 with the descent + dedup-fallback in place.

### C1 — PlanGenerator: 8 LogicalOperator types + simple-literal lowering

`pkg/relational/core/query/plangen/Convert(op)` lowers:
- LogicalScan → FullUnorderedScanExpression
- LogicalFilter (with QueryPredicate) → LogicalFilterExpression
- LogicalUnion → LogicalUnionExpression (UNION DISTINCT wraps with LogicalDistinctExpression)
- LogicalDelete → DeleteExpression
- LogicalInsert (Source non-nil) → InsertExpression
- LogicalProject → LogicalProjectionExpression
- LogicalSort → LogicalSortExpression
- LogicalUpdate → UpdateExpression

`lowerSimpleScalarText` recognises in projection / sort-key / SET RHS positions:
- Bare identifier → FieldValue
- Signed integer literal → ConstantValue(int64)
- Simple float literal (`d.d` form) → ConstantValue(float64)
- TRUE / FALSE (any case) → ConstantValue(bool)
- NULL (any case) → NullValue
- Single-quoted string literal (no escape) → ConstantValue(string)

Anything more (arithmetic, function calls, qualified refs, exponent-form numerics, apostrophe-escape strings) returns ErrUnsupported until full text→Value parsing lands.

## Known issues / tech debt

### Discovered this shift

1. **Inverse-direction rules introduce more memo exploration.** With both push and pull rules active, FixpointApply explores more alternatives before convergence. This is correct (cost-driven extraction in B4 will pick the cheaper) but slower. `BenchmarkOptimise_RealisticTree` went from 2.8 µs (pre-shift, 11 rules, no descent) to ~1 ms (31 rules + descent + dedup fallback). The seed's purpose is correctness — B6 task-stack-driven planner will be the perf-tuned version.

2. **Reference.Insert hash recomputation per loop iteration.** For each Insert, the loop calls `m.HashCodeWithoutChildren()` for every existing member. For a Reference with N members and tree size T, that's O(N·T) per Insert. Caching member hashes is sound (HashCodeWithoutChildren is stable for stable expressions) but adds complexity. Defer until a benchmark shows it as a real bottleneck.

3. **`PullCommonFilterAboveUnion/Intersection` only fires when ALL children share the SAME predicate list.** A more powerful version would handle "common subset of predicates" — pull only the common predicates, leave per-branch differences inside. Needs predicate-set algebra; gated on more sophisticated rule infrastructure.

### Pre-existing (carried over)

- **Sub-Reference traversal order in FixpointApply** is unspecified. The current implementation collects all reachable References per pass, then iterates in walker order. Cost-model extraction (B4) will need a defined order for deterministic plan extraction.
- **Cross-Reference equivalence-class merging not yet implemented** — when two Quantifiers in a tree end up pointing at structurally-equivalent References, the seed treats them as separate equivalence classes. Java's full Memo merges them; B3 follow-on.
