### 1. Cascades Java alignment audit (3 parallel research agents)

Compared Go's planner loop, cost model, and plan extraction against Java's CascadesPlanner.java:

- **Planner loop:** Java has task-stack with ExploreGroup → TransformExpression → OptimizeGroup bottom-up. Go has imperative Explore → runPlanningPhase (3 manual passes). Functionally equivalent.
- **Reference architecture:** Java has `exploratoryMembers`/`finalMembers` with `advancePlannerStage`. Go had flat `members`. **Fixed** with `finalMembers`.
- **Cost model:** All 16 criteria aligned. 3 Go-only additions documented (criterion #12 inMemorySort guard, #15b FlatMapVsNLJ, #15c EstimateCost).
- **Plan extraction:** Java prunes `finalMembers` to 1 winner via `OptimizeGroup.pruneWith`. Go stamps winners in a map.

### 3. Aggregate index pipeline (DDL → executor)

Full end-to-end for SUM/COUNT/MIN/MAX:

| Layer | What |
|-------|------|
| DDL registration | `tryAggregateIndexCandidate()` — detects aggregate index types from `GroupingKeyExpression` |
| Metadata builder | `AddAggregateIndex(table, name, groupCols, aggType, aggCol)` |
| Planner rule | `AggregateDataAccessRule` yields `physicalAggregateIndexWrapper` |
| Physical wrapper | `HintCost` with `DistinctSelectivity`, `HintOrdering` by group cols |
| Cost model | `walkExpressionTree` recognizes as covering data access |
| Executor | `executeAggregateIndexScan` → `aggregateIndexCursor` — direct FDB atomic value read |
| EXPLAIN | `AggregateIndex(COUNT, idx_name, [STATUS], T)` |

Fixes:
- `StreamingAggFromIndexRule` skips aggregate candidates
- `MatchesGroupBy`/`MatchesSingleAggregateOf` handle COUNT(*) `ConstantValue` operands
- Empty `aggColumn` uses `aggFunc + "(*)"`
