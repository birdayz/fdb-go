- **`804eb32` api.Transaction + StoreCatalog + SchemaTemplateCatalog interfaces**
  Port Java's 3 catalog-layer interfaces. ### Phase 2 — Java-compliance fixes (5 commits, self-audit + review)

- **`2945635` SaveSchema + LoadSchema**
  Three Java divergences caught during an audit of RecordLayerStoreCatalog:
  1. SaveSchema now asserts the schema's template (name, version) exists in the SchemaTemplateCatalog before persisting — otherwise returns ErrCodeUnknownSchemaTemplate. Previously silently saved.
  2. SaveSchema's missing-database error code: UNDEFINED_DATABASE (not UNKNOWN_DATABASE). Java uses UNKNOWN_DATABASE only on the delete path.
  3. LoadSchema returns UndefinedSchema whether the database or the schema is missing (Java's primary-key tuple lookup can't distinguish the two).
- **`9d506dc` reviewer feedback round 1**
  RepairSchema race fix (re-check existence under second lock), ListSchemasInDatabase early-exit under lock, stale cursor-field comment, unused fmt import, new tests for RepairSchema + ListSchemasAcrossDatabases.
- **`499a38d` port CatalogValidator.validateSchema**
  5-branch validation: nil / empty names / negative version → ErrCodeInvalidParameter. Java exact-text error messages.
- **`6b3a05b` TOCTOU doc on RepairSchema**
  Reviewer observation (non-blocking): between the two lock sections a concurrent SaveSchema could switch templates; same behaviour as Java's FDB-backed impl (row-level locking doesn't close it either). Documented, no code change.

## What to work on next

### HIGH (Phase 2 — catalog)
1. **FDB-backed StoreCatalog** — concrete implementation over `pkg/recordlayer`. Needs catalog proto definition (mirror Java's SystemTableRegistry subspace layout byte-for-byte), RecordMetaData wiring, (de)serialisation between api.Schema and catalog records. Substantial — probably 2-3 shifts.
2. **SchemaTemplate.Builder** — blocked on RecordLayerTable.Builder chain. Java's builder lets the DDL executor construct schemas programmatically; today we require a pre-built RecordMetaData. See handover decision: the chain cost is high (table→column→type dependencies); might be worth deferring to after Phase 3 is further along.

### HIGH (Phase 3 — semantic analysis, massive)
3. **LogicalOperator hierarchy** — 630 LOC in Java + a 104K-LOC cascades dependency graph. Do NOT attempt this in a single shift.
4. **SemanticAnalyzer (partial)** — ANTLR visitor that walks parse tree → resolves identifiers against catalog → produces logical operators. Gated on LogicalOperator.

### MEDIUM
5. **Columns / IndexInfo per-column data** — once api.Table surfaces the PK column list and api.Index surfaces its key-expression columns, widen the ResultSet rows (currently single-row sentinels).
6. **OptIndexFetchMethod embedded fallback** — concrete Connection must coerce to SCAN_AND_FETCH in embedded mode until Phase 9 (gRPC) lands.
7. **SchemaTemplate.isIntermingleTables Builder path** — flag is plumbed (IntermingleTables()), but the DDL-path builder that sets it is pending the SchemaTemplate.Builder work.

### LOW
8. **stringResultSet allocations** — 1020 allocs for a Columns listing is boxing overhead; revisit if SHOW COLUMNS shows up on a real profile.
9. **getSearchStringEscape** in DatabaseMetaData + LIKE-ESCAPE in compileLikePattern.
