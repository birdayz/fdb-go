The dayshift-34 handover left a 5-agent QA audit listing HIGH + MEDIUM + architecture gaps. This shift cleared **every HIGH bug**, **every bounded MEDIUM item**, **one architecture item**, added SQL-layer fuzz coverage + typed error-path assertions, caught three mid-shift Java-conformance bugs (ORDER BY NULL ordering, GROUP BY NULL hash collision, COUNT(DISTINCT) type-tag), and in the **final session** spent ~2h on a systematic Java `CastValue.java` / `VariadicFunctionValue.java` / `ExpressionVisitor.java` audit that surfaced **six more real Java divergences** — all fixed and pinned:

1. `messageTypeFromDescriptor` recursive message stack-overflow (cycle guard added; documented as deliberate divergence — Go now stricter than Java).
2. `GREATEST`/`LEAST` skipped NULL (Postgres behaviour) instead of propagating NULL (Java behaviour).
3. `BETWEEN`/`NOT BETWEEN` collapsed any-NULL-operand to UNKNOWN instead of decomposing into AND/OR of comparisons with Kleene short-circuit.

5. `CAST(string AS INT/FLOAT)` didn't trim whitespace; `CAST(string AS BOOLEAN)` accepted more strings than Java does (`'t'`, `'T'`, `'F'` etc. that Java's trim+lowercase of only `"true"/"false"/"1"/"0"` rejects).

Plus `ROUND(x, decimals)` now validates the `decimals` argument and propagates NULL (previously silently swallowed parse errors and defaulted to 0 decimals).

6. `convertToProtoValue` silently narrowed int64 → int32 / uint32 / uint64 via raw Go conversion, turning `INSERT INTO t (n_int32) VALUES (2147483648)` into a stored value of `-2147483648` (wrap). Now range-checks against target type limits before narrowing, matching Java's `CastValue.LONG_TO_INT`. `TestFDB_IntegerRangeEnforcement` pins max+1 / min-1 rejection; in-range max still succeeds. Four `//nolint:gosec` suppressions dropped once the bounds check proved conversions safe.

The only intentionally-open MEDIUM is the 8 datatype panics — discovered to be Java-conformant during the earlier audit (Java also throws unchecked RuntimeExceptions for the same conditions).

### HIGH — correctness / Java conformance

**1. NOT of UNKNOWN returns UNKNOWN** (`6853cee5` → `35bea829`)

`NotExpression` was `!v` on a bool that had already collapsed NULL→false at the leaves, so `NOT (x = NULL)` returned TRUE and every row matched. Same pattern in `NOT LIKE NULL`, `NOT IN (NULL)`, `NOT BETWEEN NULL`, `NOT (UNKNOWN AND TRUE)`, `NOT (UNKNOWN OR FALSE)`.

Fix: introduced Kleene three-valued `triBool` (triTrue/triFalse/triNull) and `evalExprPredicateTri` / `evalHavingTri` / `evalPredicateOnMapExprTri` + leaf Tri variants for comparison, IN, LIKE, BETWEEN. The existing bool-returning functions become thin wrappers that collapse UNKNOWN→false at the WHERE/HAVING/ON filter boundary — matching SQL semantics (UNKNOWN filters the row out, same outcome as FALSE, but NOT of UNKNOWN stays UNKNOWN internally). AND/OR short-circuit on the dominant concrete value (FALSE for AND, TRUE for OR) but combine through `triAnd`/`triOr` otherwise.

**2. Mixed-type equality via stringification** (`6853cee5`)

`valuesEqual` and `compareValues` fell through to `fmt.Sprintf("%v", ...)` for cross-type pairs, so `'5' = 5`, `n IN ('5', 6)` (n=int), and similar returned TRUE. Leaked into IN, NULLIF, subquery IN.

Both helpers now: preserve int64/float64 numeric coercion (same SQL "number" family), compare strings to strings, bools to bools, []byte to []byte, and return false (valuesEqual) / non-zero by type-name (compareValues) when concrete types are genuinely different.

**3. Arithmetic unification** (`ad249d55`)

`applyMathOp` (proto path) and `applyArithmeticOp` (map/JOIN/CTE path) had diverged:
- Div/0: proto errored (SQL standard), map silently returned NULL.
- `%` operator: map supported, proto missing.
- int64 preservation: proto yes, map always promoted to float64.

Collapsed both to one canonical `applyMathOp`; `applyArithmeticOp` now a one-line delegate so they cannot drift again.

**4. Aggregate NULL semantics** (`c370213e`)

Five sub-bugs, all in the aggregate accumulator loops (proto path in `execSelectByIndex`, map path in `aggregateMapRows`, fixed symmetrically):

- `COUNT(col)` previously counted NULLs — counter was bumped unconditionally. Now only increments on non-null. `COUNT(*)` still counts every row (special-cased by `aggArg == ""`).
- `SUM` of empty-or-all-NULL returned 0. SQL spec is NULL — emit nil when no non-null contribution was observed (tracked via `counts[i]`, which doubles as "non-null seen" after the COUNT fix).
- `SUM`/`AVG` silently treated non-numeric values as 0 (`toFloat64` `_` on ok). Now returns `ErrCodeInvalidParameter` with the offending type.
- Ungrouped aggregate over empty input emitted zero rows. SQL spec is one row (COUNT=0, SUM/MIN/MAX/AVG=NULL). Added a synthetic empty group.
- `COUNT(*)` inside `GROUP BY` was implicitly broken by the NULL-skip fix (empty aggArg means column lookup yields nil). Special-cased ahead of the NULL-skip check.

**5. Catalog subspace Java compatibility** (`04d03398`)

`DefaultCatalogSubspace` packed two string tuple elements (`"__SYS", "CATALOG"`), but Java's `RelationalKeyspaceProvider` declares:

```
KeySpaceDirectory(SYS,     KeyType.NULL)            -> NULL
  KeySpaceDirectory(SYS,   KeyType.NULL)            -> NULL
    KeySpaceDirectory(CATALOG, KeyType.LONG, 0L)    -> int64 0
```

So the actual catalog prefix Java writes is tuple `(NULL, NULL, 0L)`, which packs to `0x00 0x00 0x14`. Our previous two-string prefix meant a Go-written catalog was at a completely different subspace than a Java-written one — **blocking for the cross-language goal**.

Fix: `subspace.Sub(nil, nil, int64(0))` produces the exact prefix Java uses. `TestDefaultCatalogSubspaceBytes` pins `0x000014` so any future drift is caught at unit-test time rather than by a cross-language conformance run.

### MEDIUM

**6. CAST(NULL AS type), ABS overflow, integer-arg validation** (`1707446b`)

- `CAST(NULL AS <type>)` returned "unsupported CAST" error instead of NULL. Early-return `nil, nil` in `castValue`.
- `ABS(math.MinInt64)` silently wrapped back to MinInt64 (two's-complement negation is undefined for the smallest int64). Return `ErrCodeInvalidParameter` matching MySQL/Postgres behaviour.
- `LEFT` / `RIGHT` / `SUBSTRING` length/position args used a blind `int64` type assertion. Any float, string, bool, or NULL silently became 0, truncating to empty string. Replaced with `toIntegerArg` helper: accepts int64, accepts whole-valued float64 as a convenience, errors on fractional floats and non-numeric types.

**7. fmt.Errorf → api.Error sweep** (`d8defe3f`)

~30 production `fmt.Errorf(...)` sites across metadata, catalog, embedded, keyspace, and ddl dropped the SQLSTATE ErrorCode, so `errors.As(&apiErr)` silently missed them. Users couldn't map returned errors to SQLSTATE — violated the project error-handling rule.

Added `api.WrapErrorf(cause, code, format, args...)` mirroring the `fmt.Errorf("ctx: %w", err)` idiom but preserving an explicit code. Every production `fmt.Errorf` swept to `api.WrapErrorf` or `api.NewErrorf` / `api.NewError`. Codes chosen by call-site semantic: InvalidSchemaTemplate, InvalidParameter, UnsupportedOperation, InternalError.

**8. ResetSession clears per-request state** (`fcf64daa`)

Previously ResetSession only restored `schema`. Other per-request state (activeTx, ctes, schemaCache) leaked into the next pooled checkout:

- `activeTx`: if a query panicked mid-transaction, next checkout inherited an open (stale) FDB context.
- `ctes`: usually self-clearing via defer, but panics/errors could leave the map populated → phantom CTE names.
- `schemaCache`: entries survived across checkouts even after the underlying schema evolved → stale descriptor.

ResetSession now rolls back active tx, nils ctes, resets the schemaCache map, and restores `defaultSchema`.

**9. MetaDataEvolutionValidator wired into CreateTemplate** (`bef810c0`)

Validator already existed in recordlayer (exercised by 23 unit tests) but `CreateTemplate` never invoked it — a Go writer could save a schema-template evolution that Java's validator would reject. On `CreateTemplate`, if a prior version exists and the new SQL-layer version strictly exceeds it, run `MetaDataEvolutionValidator.Validate(prior, new)` and wrap any failure as `ErrCodeInvalidSchemaTemplate`. Uses `SetAllowNoVersionChange(true)` because SQL-layer versions are independent from RecordMetaData versions.

**10.a Proto recursion — messageTypeFromDescriptor cycle guard** (end-of-shift audit)

Fuzz-caught crash class + mid-shift Java audit revealed a latent stack-overflow: `messageTypeFromDescriptor → protoFieldToDataType → messageTypeFromDescriptor` on a self-referential proto (e.g. `message Tree { repeated Tree children = 1; }`). Fixed by threading a `visited` set through the descent — on re-entry return `UnresolvedType(fullName)`.

Java audit: `Type.java:464` → `Record.fromDescriptor → Field.fromDescriptor → Type.fromProtoType` has the exact same unbounded mutual recursion, no cycle guard, no depth check. Java avoids the crash in production because SQL DDL can't express self-reference; but a proto-rebuilt RecordMetaData (which Go handles natively) can carry one. **Deliberate divergence: Go is now stricter / safer than Java here.** Noted in TODO.md + persistent memory. Worth an upstream issue.

Pinned by `TestMessageTypeFromDescriptor_RecursiveMessageTerminates` + `FuzzMessageTypeFromDescriptor` (33M execs clean) + `FuzzRecordMetaDataFromProto` (34M execs clean, catches both the older `rebuildFileDescriptor` dep-cycle and the new message-type cycle).

**10.b GREATEST / LEAST propagate NULL** (`97e0c731`)

Go was skipping NULL arguments — Postgres behaviour. Java's `VariadicFunctionValue.PhysicalOperator.GREATEST_{INT,LONG,FLOAT,DOUBLE,STRING,BOOLEAN}` and the LEAST mirror all short-circuit `if (i == null) return null` on the first NULL arg (Oracle / SQL Server match). Fix the Go eval loop to return NULL as soon as any arg is NULL. Test `TestFDB_GreatestLeast` gains a NULL case pinning both functions.

**10.d CAST float→int & string→{int,float,bool}: Java-conformant** (`1acc097b`, `258073ee`, `13f43b58`)

Four distinct CAST divergences found via `fdb-record-layer-core/src/main/java/…/CastValue.java` audit:

- `CAST(float AS INT)`: Java uses `Math.round` (= `floor(x + 0.5)`, ties→+Inf); Go was truncating via `int64(n)`. Note `Math.round(-0.5) = 0` but Go's `math.Round(-0.5) = -1` — they diverge on negative halves. Fixed by computing `math.Floor(n + 0.5)` explicitly.
- NaN / Infinity: Java `INVALID_CAST`; Go silently became 0. Now errors.
- Float→int overflow: Java range-checks; Go's `int64()` was silently wrapping. Now checks against float-representable int64 bounds and errors with `ErrCodeInvalidParameter`.
- `CAST(' 42 ' AS BIGINT)` / `CAST(' 3.14 ' AS DOUBLE)`: Java does `parseInt(in.trim())` / `parseDouble(in.trim())`; Go's `ParseInt`/`ParseFloat` reject leading/trailing spaces. Added explicit `strings.TrimSpace` to match.
- `CAST('t' AS BOOLEAN)`: Java only accepts trim+lowercased `"true"/"false"/"1"/"0"`; Go's `strconv.ParseBool` was wider (`"t"`, `"T"`, `"F"`, …). Replaced with the Java-exact accept list so both engines error on the same inputs.

Four tests in `TestFDB_CastAndSubstring` pin the cases: rounded positive (`CAST(1.6 AS BIGINT) = 2`), Java-style tie to +Inf for negative half (`CAST(-1.5 AS BIGINT) = -1`, not `-2` as `math.Round` would give), rounded negative (`CAST(-2.6 AS BIGINT) = -3`), whitespace trim, padded mixed-case boolean, `'t'` rejection, `1e20` overflow rejection.

**10.c BETWEEN Kleene short-circuit** (`8ee5e98d`)

Go collapsed any NULL operand to UNKNOWN up front, losing short-circuit. Java's `ExpressionVisitor.visitBetweenComparisonPredicate` rewrites:

- `x BETWEEN lo AND hi` → `(lo <= x) AND (x <= hi)`
- `x NOT BETWEEN lo AND hi` → `(x < lo) OR (x > hi)`

then the AND / OR combine via proper Kleene. Matters when one side is definitive: `0 NOT BETWEEN 1 AND NULL` = `TRUE OR UNKNOWN` = TRUE in Java, UNKNOWN (wrongly filtered) in pre-fix Go. Fix: evaluate both comparisons as triBools and combine via `triAnd`/`triOr`. `TestFDB_NotOfUnknownIsUnknown` gains two short-circuit cases.

**11. CTE scope stack** (`c92301ed`)

`c.ctes` was a single flat map. Two bugs:
- Nested `WITH C AS (...)` inside a subquery/derived table overwrote the outer map with `make(map[...])`, losing every outer CTE. Defer then wiped on return.
- Derived-table aliases materialised inside inner queries stayed visible in outer scope until the outermost SELECT unwound.

Added `pushCTEScope()` returning a pop closure. Each entry point that introduces CTE names (`execSelect` with WITH; `execSelectQuery` with derived-table FROM) does `defer c.pushCTEScope()()`. New scope inherits the outer map so inner queries can still reference outer CTEs, but pops back to the exact outer map on exit.

### Review feedback

**11. NULL-in-IN-list correctness hole** (`4570981c`, reviewer-caught)

SQL §8.4: `x IN (1, NULL)` and `x NOT IN (1, NULL)` yield UNKNOWN when no element matches, because the expansion expands into an AND/OR chain with `x = NULL` as UNKNOWN. Both proto and map paths now track `hadNullElement` during the list scan, skip NULL entries from equality, and after the loop emit `triNull` if any NULL was seen and nothing matched.

**12. evalHavingTri NOT error guard** (`4570981c`, reviewer-caught)

When inner expression errored, zero-value `v` was `triFalse` and `v.Not()` returned `triTrue`, burying the error. Fixed to return `triFalse, err` on error, matching `evalExprPredicateTri`'s pattern.

**13. SUM `|| aggDistinct` guard removed** (`4570981c`, reviewer-caught)

Papered over the DISTINCT SUM path not accumulating into `sums[i]`. Guard removed, TODO added marking the deeper gap.
