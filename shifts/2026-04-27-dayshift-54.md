## What was done

### A2 SchemaTemplateCatalog — peeled four onion layers

The blockers I expected to be 1-2 shifts of work. Took ~3 hours.

**Layer 1 (keyspace) — verified MATCHING, no fix needed**:
Java's `RecordLayerStoreCatalog` (via `RelationalKeyspaceProvider.instance().getKeySpace()`) and Go's `catalog.OpenRecordLayerStoreCatalog()` (via `DefaultCatalogSubspace`) both target the FDB subspace `(NULL, NULL, int64(0))` — 3-byte prefix `0x000014`. Confirmed via direct subspace byte inspection. The Go sqldriver's separate `__SYS/__SYS/CATALOG` three-string subspace (3-byte prefix `0x05025f5f...`) is a sqldriver-specific concern, not a catalog-package concern.

**Layer 2 (catalog metadata version)** — `pkg/relational/core/catalog/metadata.go`:
Java's `RecordMetadataSerializer.visit(SchemaTemplate)` calls `setVersion(schemaTemplate.getVersion())` where `CATALOG_TEMPLATE_VERSION = 1`, then 3 `addIndex` calls bump to 4. Go's `BuildCatalogMetaData()` was starting at 0 → ending at 3 → `StaleMetaDataVersionError{Local:3, Stored:4}` blocked any cross-engine catalog read. **Fix**: explicit `SetVersion(1)` in `BuildCatalogMetaData()` before the addIndex calls.

**Layer 3 (proto FileDescriptor rebuild)** — `pkg/recordlayer/metadata_proto.go`:
Java's `FileDescriptorSerializer` (in fdb-relational-core/.../serde/FileDescriptorSerializer.java) emits relative type-name references like `setTypeName("T")` (no leading dot, no package) for `RecordTypeUnion` fields whose type is the user's top-level message `T`. Go's `protodesc.NewFile` resolves the reference relative to `RecordTypeUnion`'s scope first (looking for `RecordTypeUnion.T`) and doesn't fall back to file-scope. The `protodesc.FileOptions{AllowUnresolvable:true}` option does NOT silence this. **Fix**: `absolutizeFieldTypeNames` pre-processes the FileDescriptorProto before `protodesc.NewFile`, rewriting every `field.type_name` to absolute form (`"T"` → `".T"` or `".pkg.T"`). Java's `Descriptors.FileDescriptor.buildFrom` apparently does this internally; Go's `protodesc.NewFile` doesn't.

**Layer 4 (union-descriptor naming convention)** — `pkg/recordlayer/metadata.go` + `metadata_proto.go`:
- `findUnionDescriptorName` previously only knew `UnionDescriptor` (RecordLayer-core default). Java's fdb-relational uses `RecordTypeUnion`. Added it to the lookup chain.
- `setRecordsWithUnionName` previously only handled fields named `_TypeName` (RecordLayer convention with leading underscore). Java's fdb-relational emits `TypeName_N` (per `FileDescriptorSerializer.java:107`: `typeDescriptor + "_" + fieldCounter`). The discovery now ALSO handles message-typed fields by deriving the record type name from the field's TYPE reference rather than the field name.

## Tech debt / known issues

1. **Reverse-direction SchemaTemplateCatalog round-trip not yet pinned**: Go writes via `OpenRecordLayerStoreCatalog`, Java's standard JDBC reads. Mechanical follow-on. The forward direction shipped here gives strong confidence the wire format is genuinely byte-equivalent; reverse just needs the symmetric Java-side reader step.
2. **Go sqldriver three-string subspace remains divergent**: this is a sqldriver-specific concern that doesn't block A2. The catalog package itself is now Java-compat. Migrating sqldriver to use `DefaultCatalogSubspace` is a separate refactor.
3. **`absolutizeFieldTypeNames` walks all FileDescriptorProto messages**: the function is recursive over nested types and walks dependencies too. For deeply nested protos it's O(messages × fields). Today's catalog metadata is small (3 record types + 3 indexes); future schemas with many tables could benefit from caching but this is purely a perf concern.
