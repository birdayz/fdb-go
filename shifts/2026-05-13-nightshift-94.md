# nightshift-94 — Java-aligned streaming cursor architecture

**PR:** #180 (nightshift-94 branch)
**Status:** All review findings addressed, CI pending, awaiting re-review

## What shipped

Replaced the entire SQL execution model with Java-aligned streaming cursors:

### Streaming cursor architecture (replaces CollectAll)
- `aggregateCursor`: true streaming GROUP BY with group-break detection, one group at a time. Partial state serialized into `AggregateCursorContinuation` proto (keyVals, count, sums, mins, maxs)
- `memorySortCursor` / `customSortCursor`: load/emit phases with `MemorySortContinuation` proto carrying buffered records as JSON in `SortedRecord`
- `nljCursor`: incremental outer streaming, inner materialized per page

### Cross-transaction pagination
- `paginatingRows` replaces `cascadesRows`. Each page is a fresh `DB.Run` transaction. Cursor hierarchy recreated from composite continuation per Java's `CascadesPlanner` pattern
- Empty-page loop handles blocking operators (GROUP BY accumulates across pages before emitting)

### Planner fixes
- **PK scan narrowing**: `PrimaryScanMatchCandidate` registered in `GetMatchCandidates()`, `ImplementIndexScanRule` handles `RecordQueryScanPlan` via `extractScanPlan`
- **JOIN predicate pushdown**: WHERE predicates on cross-joins merged into `SelectExpression`
- **EXISTS predicate splitting**: compound `AND(ExistsPredicate, filter)` split into separate predicates
- **Hash aggregation removed**: streaming only (Java has none). Go extension: `InMemorySortPlan` inserted below streaming agg when no index
- **MaxTasks raised** to 100K (Java default)

### FDB client fixes
- `StreamingMode` default fixed: was `SmallMode` (10 rows/batch = 20K RPCs for 100K rows), now `Iterator` (exponential doubling). **This was the root cause of all 100K scan truncation.**
- `nextKV()` error propagation: surfaces `transaction_too_old` instead of silent truncation
- `indexFetchCursor`/`coveringIndexCursor` now propagate `TimeLimitReached` (was hardcoded to `SourceExhausted`)
- `RangeIterator.SetTraceLog` added for debugging

## What's next (for next shift)

1. **Port `FlatMapPipelinedCursor`**: JOINs still brute-force O(N×M) at 30s/10K. Java uses `FlatMapPipelinedCursor` with correlation bindings + pipeline depth 5 for correlated index probes. This would drop JOIN time to O(N×logM). Read Java source at `fdb-record-layer-core/.../cursors/FlatMapPipelinedCursor.java`.

2. **EXISTS + compound WHERE + ORDER BY planner non-convergence**: `NormalizePredicatesRule` has an existential guard that prevents predicate normalization for SelectExpressions with Existential quantifiers. Removing it causes infinite exploration. Needs rule deduplication infrastructure. Documented in TODO.md and DIVERGENCES.md.

3. **Covering index for SQL**: Port `IndexKeyValueToPartialRecord` (826 LOC), `computeIndexEntryToLogicalRecord`, `CollapseRecordConstructorOverFieldsToStar`. Root cause fully mapped in DIVERGENCES.md.

4. **DecorrelateValuesRule**: 25/29 Java tests pass. 4 remaining need push-down-into-child infrastructure.
