### pkg/relational/api — Phase 0 skeleton
- **`errcode.go` + `error.go`**: 70 SQLSTATE error codes matching Java's enum exactly. `Error` struct with code/message/cause/context, immutable `WithContext`, deterministic `Error()` output (sorted context keys), `errors.As` matching.
- **`datatype.go` + `datatype_primitive.go` + `datatype_composite.go`**: full DataType hierarchy.
  - Primitives (singleton-backed, zero-alloc construction): Boolean/Integer/Long/Float/Double/String/Bytes/Version/UUID/Null/Unknown
  - Composite: Vector (precision+dimensions), Array (element), Enum (name+values), Struct (name+fields), Unresolved (forward reference)
  - Named / CompositeType interfaces
  - JDBCType mapping matching java.sql.Types
  - Rune-safe StructType.String truncation (reviewer catch)
  - Matches Java's behaviour on composite types omitting the "∪ ∅" suffix
- **`options.go`**: 30 option names matching Java, IndexFetchMethod enum, 22 defaults pinned to Java's static initializer, parent-chain semantics, immutable With/Builder, reflect.DeepEqual fallback in Equal (reviewer catch).
- **`keyset.go`**: mutable KeySet with immutable EmptyKeySet singleton, explicitly-documented mutation contract.
- **`continuation.go`**: Continuation interface + ContinuationReason enum (4 values) + AtBeginning/AtEnd helpers.
- **`row.go`**: Row + RowIterable interfaces (typed accessors by position).
- **`metadata.go`**: full metadata interface tree (Metadata, Visitor, Column, Index, Table, View, InvokedRoutine, SchemaTemplate, Schema) + VisitTableTree / VisitSchemaTemplateTree helpers (Go has no interface defaults, so Java's default Accept cascades are hoisted to package-level functions).
- **`sqltypes.go`**: SqlTypeNamesSupport port — name↔JDBC↔DataType tri-directional mapping for 11 SQL types.
- **`connection.go`** (Driver + Connection + Statement + PreparedStatement), **`resultset.go`** (ResultSet + ResultSetMetaData + ColumnNullable constants pinned to JDBC values), **`array.go`** (Array + ArrayMetaData), **`struct.go`** (Struct + StructMetaData), **`direct_access.go`** (DirectAccessStatement). Lean Go-idiomatic shape — drop the java.sql.* inheritance and throw-stubs.

### pkg/relational/sqldriver — database/sql driver skeleton

- **`dsn.go`**: net/url-backed DSN parser (`fdbsql:///PATH?…` / `fdbsql://HOST:PORT/PATH`), deterministic String() round-trip, rejects bad DSNs with ErrCodeInvalidPath at sql.Open time (via DriverContext).

- **Don't skip `just gazelle`** after adding Go files — CI flagged a stale BUILD.bazel once during this shift. Fixed, but avoidable.
- **Don't break Java wire-format compat** when porting StructType or EnumType — the reviewer almost had me "fix" Java's quirky toString suffix rules (EnumType/StructType intentionally omit the `∪ ∅` suffix; primitives include it).
- **Phase 4 (cascades) is the elephant.** ~104K LOC. Do NOT start it in a shift that also has other goals. It needs its own multi-shift focus with sub-RFCs per rule family.
- **Memory saved:** `project_relational_layer_port.md` — use it for context on future shifts.
- **yamsql test corpus:** `fdb-record-layer/yaml-tests/src/test/resources/*.yamsql` — 178 files. YAML format with `schema_template:` (CREATE statements) + `test_block:` (query/count or query/result pairs). Phase 1 conformance tests should run Go parser + executor against every file and diff results vs Java. That's the acceptance bar for "SQL compatible with Java."
- **Phase 1 unblocker — Java action in grammar.** Downloaded ANTLR 4.13.1 + ran `-Dlanguage=Go -package parser -visitor` over the vendored `.g4` files. Generation produced 70 881 lines of Go successfully BUT included one Java-specific action block from `RelationalLexer.g4:1381`:
  ```
  ERROR_RECOGNITION
    : . { this.notifyListeners(new LexerNoViableAltException(this, _input, _tokenStartCharIndex, null)); }
    ;
  ```
  That `{…}` is Java syntax and the Go output compiled to `this.notifyListeners(new LexerNoViableAltException(...))` verbatim, which is not valid Go. Options for Phase 1:
  1. **Remove the action.** ANTLR will still surface unrecognised characters through default error listeners; the explicit `notifyListeners` call is a convenience, not required. Safest option, single-line change, doesn't affect Java. Run `grep-l` across Java runtime to confirm the default ListenerDispatch still fires.
  2. **Target-specific action syntax:** ANTLR supports `{<java: ...>}` form. We'd rewrite the action with a Java-only branch. Preserves Java behaviour exactly.
  3. **Keep two grammar files** — a Go-only variant. Maintenance nightmare; do not do this.
  Recommendation: go with (1) unless a yamsql test depends on the exact error text from `notifyListeners`. (2) is fallback.

  **PoC confirmed (nightshift-24, in /tmp):** removed the Java action, regenerated with ANTLR 4.13.1 `-Dlanguage=Go`, compiled cleanly against `antlr4-go/antlr/v4@v4.13.1`. 3/4 test SQL statements (SELECT * FROM, SELECT with WHERE, INSERT INTO VALUES) parsed with zero errors. The 4th (CREATE TABLE) failed with `no viable alternative at input 'CREATETABLE'` — grammar keywords are case-sensitive by default and Java uses a `CaseInsensitiveCharStream` wrapper that we'll need to port too. ANTLR Go runtime provides the hooks; the `CaseInsensitiveCharStream` is ~50 LOC of Go.

  **So Phase 1 in bullet-form:** (a) delete the `notifyListeners` action line, (b) add `antlr4-go/antlr/v4@v4.13.1` to go.mod + `com_github_antlr4_go_antlr_v4` to MODULE.bazel use_repo, (c) generate + commit `pkg/relational/core/parser/gen/*.go` (~71 kLOC of generated code), (d) port `CaseInsensitiveCharStream`, (e) write the `parser.Parse(sql)` wrapper + error listener around it, (f) yamsql corpus smoke-test.

  **End-to-end PoC validated in /tmp/phase1-poc (nightshift-24):** after (a)+(b)+(c)+(d), 10 out of 10 test SQL statements parse cleanly through `p.Root()` with zero errors, including mixed-case (`Select Id From Orders`) and DDL. `CaseInsensitiveCharStream` is **40 LOC** of Go — wraps `antlr.NewInputStream(s)` and overrides `LA(i)` to `unicode.ToUpper(rune(stream.LA(i)))`. Original text is preserved via `GetText` for token text extraction (matches Java semantics).

  **Grammar is otherwise target-agnostic.** `grep` confirms only ONE language-specific action in the whole 2836-line grammar (the `notifyListeners` one at `RelationalLexer.g4:1381`). No `@members`/`@header` blocks, no semantic predicates `{…}?`, only target-agnostic `channels { MYSQLCOMMENT }` and `options { tokenVocab=RelationalLexer; }`. That means option (1) — single-line deletion — is sufficient to fully port the grammar to Go without any Java/Go branching infrastructure. Safe to proceed.

  **Non-obvious dialect gotcha discovered during PoC:** this SQL dialect does NOT have top-level `CREATE TABLE` / `DROP TABLE`. Top-level DDL is strictly `CREATE SCHEMA … WITH TEMPLATE …`, `CREATE SCHEMA TEMPLATE … templateClause+`, or `CREATE DATABASE path`. Table definitions live inside `templateClause` within `CREATE SCHEMA TEMPLATE`. For `DROP`: only `DROP SCHEMA/SCHEMA TEMPLATE/DATABASE` at the top level (verify with `grep -A5 "^dropStatement" RelationalParser.g4`). This differs sharply from MySQL/Postgres users' expectations — document visibly in the Go parser's README when Phase 1 lands. Yamsql tests set up tables via `schema_template:` in YAML, not via top-level DDL, which lines up with this.
