### Cross-engine harness improvements

* **`ExpectErrorMessage` field on `RunQuery`** (stricter than `ExpectErrorContains`): requires `*plandiff.JavaError.Message` and `*api.Error.Message` to literally equal the same string. Catches drift the moment either engine reorders or rephrases its message.
* **Per-entry isolation for negative tests**: when an entry has `ExpectErrorContains` or `ExpectErrorMessage` set, the harness spawns a fresh Java conformance server just for that entry. Eliminates fdb-relational's planner-state-leak from stalling the run (the same issue documented in `yamsql_cross_engine_conformance_test.go` as the reason `error_code` tests are skipped). Cost: ~5s extra Java startup per negative entry. Positive entries still share the outer-It server.
* **Java-verbatim error messages on Go side**: changed Go's "unsupported bitwise operator" to "Unsupported operator", "unsupported scalar function" to "Unsupported operator", and the MIN/MAX runtime-gate message to verbatim Java's VerifyException message. Per project conformance principle: doesn't work in Java → doesn't work in Go, **with identical error wording on every removed feature**.

## Known issues / tech debt

### Discovered this shift

1. **CI keeps cancelling under rapid pushes**. Same pattern as nightshift-60 / swingshift-59. Each commit cancels the in-flight CI run.
2. **Java's NPE messages reference parser-internal class names** (`InnerJoinContext.expression()`, `AggregateWindowedFunctionContext.ALL()`). They can't share clean substrings with Go's user-friendly error messages without aligning Go to mimic Java's panic-style failures (UX regression). Cross-engine ExpectErrorContains works for the Java-clean-error rejections (operator-not-registered: bit-shift, NULLIF) but not for Java-NPE rejections (CROSS JOIN, COUNT(DISTINCT)).
3. **Per-entry Java-server isolation costs ~5s per negative entry** for spawn + ready-check. Acceptable for the current 4 negative entries (~20s overhead) but would balloon if all yamsql `error_code` tests were unblocked under the same pattern.
