## What shipped

### Bug fixes (5 correctness issues + 1 limitation removed)

1. **Correlated NOT EXISTS returned EXISTS results** — translator's `splitNonExistsPredicates` dropped `NOT(ExistsPredicate)` from the SelectExpression's predicate list. The rule's `implementExistentialSelect` couldn't detect negation, defaulting to EXISTS. Added `extractExistsPredicates` to preserve EXISTS predicates in the predicate list.

2. **EXISTS outer-only predicates pushed to inner plan** — `WHERE status='pending' AND NOT EXISTS(...)` ignored the status filter because ALL residual predicates were pushed inside the inner plan (or passed as NLJ join predicates). Fixed by splitting predicates: inner-referencing go to inner/join, outer-only go to outer plan filter. Fixed in both `buildExistsFlatMap` (PK/index FlatMap path) and the NLJ fallback in `implementExistentialSelect`.

3. **Nested NOT EXISTS dropped middle-level correlation** — `NOT EXISTS (products WHERE cat_id=c.id AND NOT EXISTS (reviews WHERE product_id=p.id))` lost the `cat_id=c.id` predicate. The `buildCorrelatedExists` hoisting replaced the middle-level plan with the innermost plan. Fixed by distinguishing: (a) middle has correlation + nested EXISTS → build proper LogicalFilter preserving both levels; (b) middle has only EXISTS (cross-level correlation) → hoist as before.

4. **HAVING EXISTS predicate split** — HAVING path didn't use `splitNonExistsPredicates`/`extractExistsPredicates`, so NOT(ExistsPredicate) buried in an AND was invisible to the rule.

5. **Multi-table inner queries in correlated EXISTS** — `buildCorrelatedExists` only handled single-table inner queries. Added join tree building from `sq.joins` and scope registration for join tables.

### Extended session bug fixes (3 more correctness issues)

6. **IN-list returning 0 rows** — Multi-value IN queries (`WHERE id IN (1,2,3)`) returned empty result sets. Root cause: `InComparisonToExplodeRule` decomposes IN-list into `Explode` + `SelectExpression`. The NLJ rule matched this pattern and produced an NLJ plan. But `mergeRows` can't combine a scalar outer (from Explode) with a map inner (from scan). Fix: add Explode guard in `ImplementNestedLoopJoinRule` so IN patterns are only handled by `InJoinRule` or fall back to filter+scan.

7. **Nested aggregates panic** — `SUM(MAX(v))` reached the executor and panicked with "aggregate must be evaluated over rows by the aggregator." Fix: detect nested aggregates at parse time by walking the ANTLR tree of aggregate arguments for `AggregateFunctionCallContext` nodes. Returns error 0A000 matching Java's `SemanticAnalyzer.validateGroupByAggregates`.

8. **HAVING EXISTS silently wrong** — `HAVING NOT EXISTS(...)` returned 0 rows instead of an error. The correlation references pre-GROUP-BY scope (table columns) but HAVING evaluates post-GROUP-BY. Fix: reject HAVING EXISTS at translation time, returning "could not plan query" instead of silently wrong results. Java doesn't support this either.
