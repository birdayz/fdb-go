## What was done

### B4 — Cost model (4.4): full seed

New `pkg/recordlayer/query/plan/cascades/properties/` package:
- `cost.go`: `Cost{Cardinality, CPU}` Go-native heuristic, `EstimateCost` walking the RelationalExpression hierarchy (12 operator arms covering all seed concrete types), `BestRefCost(ref)` with Reference-keyed memoisation (O(N+K) vs un-memoised O(N×K) for wide References), `CostLess` comparator, `StatisticsProvider` interface (Default / Fixed / Map impls), `CostHinter` for opaque wrappers (physical-wrapper bias multiplier 0.9 makes physical alternatives strictly cheaper than logical), `BestMemberSelector` for planner integration, `WithChildren` interface for opaque-type rebuild support.
- `extract.go`: `ExtractBestPlan(ref)` recursive plan extractor returning a fresh singleton-Reference tree. Switch-on-type with 12 constructor arms; opaque-passthrough fallback for unknown wrappers.
- `cardinality.go` (new this shift): `EstimateCardinality` / `EstimateCardinalityWith` / `BestRefCardinality` / `CardinalityLess`. Thin wrapper over Cost that projects out the CPU axis. Useful for cardinality-aware rules without paying for CPU calc.
- `ordering.go` (new this shift): `EstimateOrdering` / `IsOrdered`. Yes/no + ordering-keys analysis. Sort produces known order; Filter/Projection/TypeFilter/Distinct/Unique/DML inherit from inner; Union/Intersection/Scan return unknown.

Calibration targets pinned in unit tests: Filter beats Sort, Distinct(scan) beats Distinct(Sort(scan)), Sort(Filter(scan)) beats Filter(Sort(scan)), Intersection bounded by min child. `FuzzCostMonotonicity` (8.8M execs/20s clean) + `FuzzExtractBestPlan_SingletonInvariant` (12.6M execs/15s clean).

### B6 — Task-stack planner (4.6): full seed

`cascades.Planner` with EXPLORE + OPTIMIZE phases:
- `ExploreReferenceTask` / `ExploreExpressionTask` / `ApplyRulesTask` (with critical saturation-stamp-only-on-grew==0 fix found by FuzzPlanner_Confluence — premature stamp incorrectly skipped re-fires after growth).
- `OptimizeReferenceTask` picks cheapest member by Cost and stamps it in `bestMember[ref]`.
- `Plan()` runs both phases; `BestMember(ref)` accessor exposes per-Ref winners.
- `PlannerEventHandler` hooks: OnExploreReference / OnExploreExpression / OnApplyRules / OnOptimizeReference.
- MaxTasks hard cap (default 100k), LIFO stack invariant documented.

Real perf win: BenchmarkPlanner_RealisticTree 2.43ms / 54K allocs vs BenchmarkOptimise_RealisticTree (FixpointApply) 4.65ms / 99K allocs — ~48% faster from saturation tracking.

Confluence with FixpointApply pinned via FuzzPlanner_Confluence (4.2M execs/20s clean post-fix). Idempotence + initial-member preservation pinned by 2 more fuzz targets. Plan() entry pinned by FuzzPlanner_PlanFullPipeline (7.3M execs/15s clean). FuzzPlanner_WithBatchA_NoPanic (430K execs/10s, 583K execs/15s) added late-shift to stress the implement rules interacting with logical-rewrite chain.

Filter-over-Union and similar shapes silently couldn't physically implement. Fixed across all 9 rules + `wrapPhysicalPlan`. The bridge wrappers themselves are a SEED workaround for the lack of plan-aware Memo; ~300 lines + every Implement* rule's wrapper-detection switch can collapse when proper Memo lands.

4. **OrderingProperty Distinct + Unique inheritance is conservative**: Java's analysis only preserves Sort ordering through Distinct/Unique when the inner ordering aligns with the dedup grouping keys. The Go seed unconditionally inherits — produces correct optimisations but may miss some "actually unsorted" cases. Sharpen when sort-elimination rules need it.

5. **TableFunctionExpression accepts Value, not StreamingValue interface**: Java has a `StreamingValue` marker interface; the Go port doesn't (Go-idiomatic). Caller is expected to pass a streaming-capable Value (RangeValue today). Mismatched callers surface as a degenerate result type, not a compile-time error. Add a Go interface marker if/when the planner needs to type-assert.
