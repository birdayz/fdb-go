## What shipped

### FlatMapPipelinedCursor + RecordQueryFlatMapPlan (Java-aligned)

Ported Java's correlated join execution pattern. For each outer row, the inner plan is re-executed with the outer row bound as a correlation — enabling targeted PK probes on the inner side (O(N×logM) vs O(N×M)).

**Architecture (1:1 with Java):**
- `CorrelationBinder` interface in values package — allows `QuantifiedObjectValue.Evaluate` to resolve correlations from the EvaluationContext
- `EvaluationContext.GetCorrelationBinding` implements the interface
- `flatMapCursor` — streams outer, re-executes inner per row with correlation binding (Go equivalent of `FlatMapPipelinedCursor` minus async pipelining)
- `RecordQueryFlatMapPlan` — plan type with outer/inner plans + correlation identifiers
- `physicalFlatMapWrapper` — Cascades expression wrapper for the optimizer
- `ImplementNestedLoopJoinRule.tryFlatMapPlan` — detects single-predicate equi-joins on inner PK and emits FlatMap

**Triggers when:**
- Single equi-join predicate on inner table's primary key
- Outer FK column name differs from inner PK name (e.g., `orders.customer_id = customers.id`)
- Inner plan is a full table scan

**Falls back to NLJ when:**
- Multiple predicates (WHERE filters + join condition)
- Same column name on both sides (`a.id = b.id`) — column derivation ambiguity
- Inner plan is not a scan

## Known limitations (for next shift)

1. **Same-column-name joins**: `a.id = b.id` falls back to NLJ (guard in place). The column expansion layer for SELECT * doesn't handle the qualified-key output from JoinMergeResultValue identically to NLJ's mergeRows. Fixing requires adjusting the ResultSet column derivation.

2. **No FlatMapContinuation proto**: cross-transaction pagination through FlatMap results falls back to re-scan. Port `FlatMapContinuation` proto encode/decode (outer + inner continuation + check value).

3. **No async pipelining**: Go is synchronous, no pipeline depth 5. Only matters for latency-hiding — not a correctness issue.

4. **JoinMergeResultValue vs RecordConstructorValue**: the translator should produce a proper RecordConstructorValue (field-by-field from both QOVs) instead of QOV(leftAlias). JoinMergeResultValue is a bridge until the translator is fixed.

### SQL LIMIT/OFFSET (Go extension)

Most-requested usability feature, now shipping. Approach:
- PlanVisitor.visitLimit parses LIMIT/OFFSET from ANTLR into LogicalLimit
- Cascades translator SKIPS LogicalLimit (no Cascades pipeline involvement)
- paginatingRows.Next() applies sqlLimit/sqlOffset post-execution (skip + cap)
- Plan cache skips caching LIMIT queries (limit not stored in cache)

Tests: TestFDB_SelectLimit, TestFDB_LimitOffset, yamsql offset.yaml (7 scenarios), limit_join.yaml (3 scenarios combining LIMIT + FlatMap JOIN).
