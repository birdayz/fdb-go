## What landed (chronological)

### Secondary-index range predicates — single-col (38f3f7a0)

`WHERE col > X` / `col >= X` / `col < X` / `col <= X` on a
single-column VALUE index now hits `ScanIndexRecords` instead of
the full type scan. Wiring mirrors the PK range pattern:
`trySecondaryIndexRangeFromWhere` walks AND leaves into bucketed
`pkRangeBounds`; `secondaryIndexRangeScanCursor` builds a
`TupleRange` with `TreeStart`/`TreeEnd` fallback when one side is
open. SELECT + UPDATE + DELETE. +13 yamsql cases.

### Composite secondary-index — leading-eq + trailing-range (3dc7f279)

`WHERE idx_col_0 = X AND ... AND idx_col_n_range > Y` on a
composite VALUE index. `trySecondaryIndexCompositeRangeFromWhere`
walks AND leaves once into equalities + range bounds,
`secondaryIndexCompositeRangeScanCursor` builds a TupleRange whose
low and high tuples share the prefix and differ only in the last
component. +14 yamsql cases. (Initially implemented as last-col
only; relaxed in 9dcb0bfb.)

### Composite PK range — relax to any column (dd97a817)

Lifted the arbitrary "last PK col only" restriction. The narrowed
FDB key range is contiguous as long as the leading prefix is fixed
and the first varying component carries a bound — true for a range
on any PK col, not just the last. Now:

- PK (a, b, c), `WHERE a = 1 AND b > 5 AND c = 3` → pushdown with
  prefix=[1], range on b; c=3 post-filtered.
- PK (a, b, c), `WHERE a < 5` → pushdown with empty prefix, range
  on a; b and c unconstrained.

Cursor builder `pkPushdownCompositeRangeScanCursor` already
supported arbitrary-length prefixes (even empty), so only the
detection logic changed. +13 yamsql cases on a new 3-col PK
`kvw` table.

### Composite secondary-index range — same relaxation (9dcb0bfb)

Same "range on any idx col" relaxation applied to the composite
secondary-index extractor. Walks AND leaves once into equalities
and range bounds, finds the first idx col with a range, requires
leading equalities on prior cols. +8 yamsql cases.

### BETWEEN pushdown across all four range extractors (1e771813)

`col BETWEEN lo AND hi` now narrows scans across:
- single-col PK range
- composite PK range
- single-col secondary-index range
- composite secondary-index range

`extractColBetweenLiteral` recognises a `PredicatedExpressionContext`
with a non-NOT `BetweenComparisonPredicateContext` inside, a bare
column LHS, and two constant literal bounds. Contributes both an
inclusive low and an inclusive high to the column's range entry.
NOT BETWEEN bails (non-contiguous key range — needs a UNION of
scans). NULL bounds bail (UNKNOWN predicate cannot narrow). +25
yamsql cases. Existing `between.yaml` (scan-path evaluator) still
green — no regression to the non-pushdown path.

### Edge-case tests (70691074, e296b1c7)

Additional defensive coverage:
- Empty table range + BETWEEN scans: no rows, no crash.
- NULL-valued indexed column: scan loop's 3VL evalPredicate drops

  or closed-low (index never visits NULL).
- BETWEEN inside OR: flattenAndPredicates bails on OR, scan path
  handles the full predicate. Guards against any extractor silently
  narrowing under a disjunction.

to mention BETWEEN and to unstale its xref to the composite-range
extractor.

### Post-LGTM bug fix: scannability check in the helper loop (5e28f487)

Caught during a post-LGTM audit. The three `trySecondaryIndex*FromWhere`
helpers picked the first matching index in declared order regardless
of state; the outer call-site `&& store.IsIndexScannable(...)` guard
then rejected the whole pushdown if that first match happened to be
WRITE_ONLY / DISABLED — even when a LATER scannable index would
have matched just as well. Two indexes on the same column with the
WRITE_ONLY one declared first (e.g. during online index rebuild)
would silently fall through to a full scan.

Fix: push the `IsIndexScannable` check into the iteration loop of
each helper. Non-scannable indexes are skipped, so a later scannable
match is found naturally. Signature change — all three `*FromWhere`
helpers and their `*Pushdown` wrappers now take an extra
`store *FDBRecordStore` param. Six call sites updated (three in
`pkPushdownCursor` for UPDATE / DELETE, three in the SELECT path);
outer scannability guards removed.

Reviewer LGTM'd the fix in a targeted re-review pass. No behavioural
change when all indexes are READABLE (today's common case). Gap
closed ahead of online indexing scenarios that produce WRITE_ONLY
indexes during rebuild.

## Next-shift follow-ups

### HIGH — the big remaining piece

- [ ] **Covering-index optimisation** — scoped out of dayshift-43
  because it touches SQL projection / WHERE / ORDER BY column
  analysis and the `*FDBStoredRecord[proto.Message]` synthesis path
  (`dynamicpb.NewMessage(rt.Descriptor)` is likely fine since
  protoreflect access is the only consumer, but the projection
  analysis needs careful scoping). Skip the by-PK record fetch
  when every column referenced by projection + WHERE + ORDER BY
  is either in the index key or in the PK tuple. Real perf win
  (1 round-trip per row instead of 2). Est: 2–3 hours.

- [ ] **Intermingled schema type-filter wrapper** — re-enable PK
  pushdown on `FieldKeyExpression` PKs by wrapping the narrowed
  cursor with a type-filtering predicate (`filterCursor` in
  `pkg/recordlayer/cursor_combinators.go` needs an exported
  constructor). **Blocker**: `INTERMINGLE_TABLES` exists in the
  lexer/parser grammar but isn't wired through DDL. Without
  end-to-end DDL support we can't write yamsql tests. Either
  wire DDL first or test via unit tests with hand-constructed
  metadata.

### MEDIUM

- [ ] **IN-list pushdown** — `WHERE col IN (1, 2, 3)` decomposes
  into a union of point scans. Needs a different cursor shape
  (UnionCursor over per-value range scans) or could be emulated
  by scanning the full range `[min, max]` and post-filtering.
  The union-of-points approach is the correct Java-aligned shape.

- [ ] **LIKE prefix pushdown** — `WHERE col LIKE 'foo%'` maps to
  `col >= 'foo' AND col < 'fop'` (strinc on the prefix), a
  contiguous range. Trickier characters (Unicode, escaped wildcards)
  mean the prefix detection needs to be conservative.

- [ ] **Mutual recursion** — `WITH RECURSIVE a AS (..., b ...), b
  AS (..., a ...)`. Uncommon but a real correctness gap today —
  CTEs are evaluated in declaration order.

- [ ] **Continuation resumption of partial recursive results** —
  for `maxRows: 1` style paginated retrieval. Java's
  `RecursiveStateManager` uses TempTable cursors.

### LOW — niche polish

- [ ] **ORDER BY dedup edge cases** — `ORDER BY b, B`
  (case-folded) and `ORDER BY a.b, b` (qualified-vs-bare resolving
  to the same column). Ride on a future identifier-folding pass.

- [ ] **`GROUP BY (a+b) AS alias`** — expression group keys can't
  be referenced via alias from the SELECT list because the rewrite
  path only handles bare-column group-by entries.

- [ ] **Benchmark comparing pushdown vs full scan** — no direct
  benchmark today. The CI bench reporter shows variance (unrelated
  `SaveRecordBatch +13%` etc.) that's noise; a targeted micro-bench
  would make the pushdown perf win observable.

- [x] ~~Secondary-index pushdown: first-match + unscannable gap~~
  — fixed in 5e28f487 this shift. `IsIndexScannable` check pushed
  into the iteration loop of each `trySecondaryIndex*FromWhere`
  helper, so a WRITE_ONLY / DISABLED index no longer masks a later
  scannable match.
