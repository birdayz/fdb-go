### Highest-value next-shift work

Pick one to fix from the surfaced bugs:

1. **#56 case-folding** is the largest behaviour divergence. Find every place table/column names are looked up against catalog metadata (probably in `select_parser.go`, `select_helpers.go`, `eval_predicate.go`) and apply `strings.ToUpper(...)` (or whatever Java's normalization is — verify empirically). Standard SQL says unquoted identifiers fold to upper case; Java does this; Go doesn't. Many existing corpus entries are workaround-aware (use the canonical UPPER table name in queries) but real users don't. **High impact, ~1 shift.**

2. **#42 compound DISTINCT** is well-isolated. Reproduces with `SELECT DISTINCT a, b FROM t`. The dedup code in `select_query_full.go:1182` looks correct on inspection; bug must be in what gets passed in. ~0.5 shift to root-cause + fix.

3. **#44 UNION ALL ORDER BY** is the second-most-isolated. UNION ALL with trailing ORDER BY on PK doesn't honour the sort. Fix likely in `union.go` or `select_parser.go` extraction of the order-by clause. Note: `union.go::execUnion` looks correct on inspection — the trailing ORDER BY is hoisted from the right simpleTable correctly. Real bug is probably in how the parser routes the trailing clause for some shapes.

4. **#48 signed zero** is small and well-scoped. Fix in `valuesEqual` / `CompareValues` — but on inspection both look correct. Real bug must be in scan-time tuple-key comparison (FDB tuple encoding for `-0.0` vs `+0.0`). ~0.5 shift to investigate.
