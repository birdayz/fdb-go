## What was done

### 1. Bug fix: SetTransactionRetryLimit(0) silently ignored

The `fdb` wrapper layer used `retryLimit != 0` as the sentinel for "not set", but 0 is a valid value meaning "no retries". `db.Options().SetTransactionRetryLimit(0)` silently had no effect — the retry limit was never propagated to transactions.

Root cause: `txDefaults.retryLimit` defaulted to 0 (Go zero value), and `applyTxDefaults` checked `if d.retryLimit != 0` which was false for both "not set" and "set to 0".

Fix: Added `hasRetryLimit bool` to `txDefaults`, matching the client package pattern. `SetTransactionRetryLimit()` now sets the flag, and `applyTxDefaults` checks `hasRetryLimit` instead.

Regression test: Uses `Transact()` callback with invocation counting — with retryLimit=0, `OnError` rejects retry so the callback runs exactly once and the error propagates. Three iterations: (1) initial test used raw `CreateTransaction` (doesn't apply wrapper defaults), (2) reviewer caught weakness, (3) rewrote to use `Transact()` with callback counting, also fixed a data race where FutureNil from OnError was passed to `t.Logf` without `.Get()`.

### 3. Bug fix: data races in read path vs postCommitReset (CRITICAL)

Multiple data races between Watch goroutines and `Commit()→postCommitReset()`:

1. `ensureReadVersion()` read `userSetReadVersion` and `readVersion` outside `readVersionMu`
2. `sendGetValue/sendGetKey/sendGetRange/sendWatch` all read `tx.readVersion`, `tx.lockAware`, `tx.tenantId` inside closures that execute in goroutines (via hedge), while `postCommitReset` clears these fields

Root cause: swingshift-15 added `readVersionMu` for `hasReadVersion` but missed ALL other reads of the protected fields outside the lock.

Fix (two commits):
- `ensureReadVersion`: capture `userSetReadVersion` and `readVersion` under lock
- All 4 send functions: capture `readVersion`, `lockAware`, `tenantId` under lock before building closures. `sendGetValueToServer` signature changed to accept captured values.

Verified: `just race-all` (all 5 targets) + CI race detector both clean.

### 4. Dead code removal in online_indexer

`allTargetIndexesIdempotent()` had a redundant trailing check: `return isIndexTypeIdempotent(oi.primaryIndex().Type)`. Since `primaryIndex()` returns `targetIndexes[0]`, the loop already covered it. Simplified to `return true`.

### 11. Server selection: power-of-two random choices (C++ divergence #20)

Replaced deterministic min-metric server selection with C++'s "power of two random choices" algorithm in both `chooseServer` (Watch) and `chooseTopTwo` (getValue/getKey/getRange). Old behavior: all clients converge on same least-loaded server under uniform load, creating hotspots. New: pick 2 random candidates, select lower metric. Proven to distribute load better. For hedged reads, primary is randomized, backup is deterministic best-remaining.

### 12. Fix: binding stress bazel-bin symlink invalidation

`bazel-bin/` is a symlink recreated on every Bazel invocation. When a concurrent build runs (e.g., pre-commit hook during stress), the Python binding tester can't find the stacktester binary → false positive failure. Fix: `filepath.EvalSymlinks()` resolves to the actual file in the Bazel cache, which survives across builds.

Verified: 20/20 seeds pass even with concurrent `just build`. Previously this was a consistent false positive.

### 13. Fix: govulncheck CI step (broken since addition) + nightly fuzz (broken since April 7)

Both used `go install` / `go test` which fails with `command not found` on the self-hosted runner (no system Go in PATH). Fix: resolve Bazel's hermetic Go SDK via `ls "$OUTPUT_BASE"/external/rules_go++go_sdk+*download*/bin/go`.

Verified: govulncheck passes in CI for the first time. Nightly fuzz fix will be verified at next scheduled run (3:17 AM UTC).
