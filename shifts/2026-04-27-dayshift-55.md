## What was done

### Track A3 starter wiring (single-scenario proof-of-concept)

Initial commit `89a86491`: One scenario (`where_literal_on_left`) inlined as Go code and driven through `plandiff.NewJavaRunnerHTTP(...).RunWithSetup`. Asserts Java result rows match the scenario's declared expected rows. 10 cross-engine tests pass.

Why inline rather than load yamsql/testdata/: the Bazel sandbox doesn't carry that tree, and adding it as a data dep would couple the conformance test to the yamsql package's data layout. Inlining also makes per-scenario adaptations explicit.

### Track A3 expansion: 14 more scenarios + multi-scenario harness shape

Commits `98972adb` through `09241e2f`: Added arithmetic, cast, composite_pk, bytes, between, boolean, like, case_when, aggregate_empty_table, bitwise, avg, derived_table, coalesce_nullif, bare_col_with_agg. Restructured to a per-scenario `Describe` loop driven by a list of constructors; adding a new scenario is now appending to the list. Added multiset comparison for `unordered: true` tests.

Per-scenario adaptations needed for cross-engine compat (each documented inline in the scenario constructor):
- Drop `NOT NULL` on PK columns (fdb-relational restriction; PK is implicitly NOT NULL anyway)
- Uppercase BYTES literals: `X'cafe'` not `x'cafe'`, `B64'yv4='` not `b64'yv4='` (lexer strict)
- Drop untyped-NULL operands: `a + NULL`, `b AND NULL`, `b OR NULL` (planner type-encapsulation error)
- Drop FROM-less SELECTs: `SELECT -10 / 3` (planner rejects)
- Drop multi-col ORDER BY: `ORDER BY a, b` (planner unsupported)
- Drop GROUP BY tests (planner unsupported)
- Drop DISTINCT tests (planner unsupported)
- Drop IS TRUE / IS FALSE / IS NOT TRUE / IS NOT FALSE (planner unsupported)
- Drop `WHERE (b = true)` — bare paren parses as RecordConstructorValue, not a predicate
- Drop `WITH ... ORDER BY` — Java rejects "order by is not supported in subquery"
- Drop NULLIF (function not registered)
- Drop bit-shift `<< / >>` (tokenized but not implemented)
- Drop bare-bool-column-in-projection (`SELECT b AND TRUE`, `SELECT NOT b`) — Go rejects, Java accepts (the asymmetric one — surfaced ONLY by adding Go-side runner)

### Round-1 reviewer-driven major work: WIRE THE GO SIDE

Round-1 review (commit `4326941033` comment) called out that the harness was named `cross-engine` but only ran Java — i.e. it was a Java oracle test, not a cross-engine equivalence test. Address: invoked `plandiff.NewGoSQLSetupRunner(clusterFilePath)` alongside the Java runner (commit `5474f86a`). Each test now does:

```go
javaRes := javaRunner.RunWithSetup(ctx, ...)  // (a)
goRes   := goRunner.RunWithSetup(ctx, ...)    // (b)
assertRowsMatch(javaRes.Rows.Rows, t.Rows, ...)         // Java vs expected
assertRowsMatch(goRes.Rows.Rows,   t.Rows, ...)         // Go vs expected
assertRowSetsCrossEqual(javaRes.Rows.Rows, goRes.Rows.Rows, ...) // Java == Go
```

The Go runner needs a cluster-file path (the fdbsql driver opens it directly) while the Java runner needs the contents (sent over HTTP). `writeClusterFileToTemp` — already in conformance package — bridges this.

Wiring Go surfaced the bare-bool-column-in-projection asymmetry (above) — five boolean-projection tests dropped, gotcha added.

## Tech debt / known issues

1. **Go-engine bare-bool-column-in-projection asymmetry**. `SELECT b AND TRUE`, `SELECT NOT b` rejected by Go ("expected BooleanValue but got FieldValue"); Java accepts. The Go projection-path strictness check was tightened too aggressively at some point. Five tests deferred. Fix is in `pkg/relational/core/embedded/select_query_full.go` (or wherever the projection-eval predicate-only check lives) — relax for projection context, keep strict for WHERE/HAVING context.
2. **Java-vs-Go semantic divergence: `SELECT COUNT(*) FROM t WHERE <none-match> HAVING COUNT(*) >= 0`** — SQL spec says one row [[0]], Java returns 0 rows. Documented in CLAUDE.md; the divergent test dropped from cross-engine. Future: file upstream Java bug.
3. **Scenario coverage is ~10% of the 1587 yamsql statements**. 14 more scenarios shipped this shift; another 100+ scenarios in the testdata tree wait. Each requires: inline transcription, drop NOT NULL on PK, identify which tests trip Java (planner) and Go (engine) limitations, document as gotcha if novel. Mechanical follow-on but tedious.
4. **`crossEngineScenarios()` ALL constructors run at registration time** (Ginkgo `Describe` body). For 16 scenarios this is ~30 µs total — fine. If we hit hundreds, switch to lazy.
5. **Per-test ephemeral schema** — each test does its own runWithSetup, which means each test creates + destroys an FDB schema. 111 tests × 2 engines × ~150ms/setup = ~30s scenario-test wall time today. For 1000+ tests, batch by scenario.

## What I would have done differently

1. **Started with the Go-side wiring.** I shipped 13 scenarios using only the Java runner, then the reviewer correctly pointed out the harness wasn't actually cross-engine. Wiring Go in commit 12 instead of commit 14 would have:
   - Surfaced the bare-bool-projection asymmetry earlier (so I could either fix it or cleanly skip those tests from the start)
   - Given me data on Go-engine flakiness across scenarios as I added them, instead of all at once
   - Saved the round of scenario-comment rewrites that came after wiring Go (the boolean scenario doc had to be rewritten to add the new asymmetry note)
2. **Looked at the existing Go runner pattern from `run_sql_conformance_test.go` first.** I assumed Go-side execution would need new infrastructure; in fact `plandiff.NewGoSQLSetupRunner` was already there. 5-min lookup would've saved 15 min of wrongheaded thinking earlier in the shift.
3. **Used the conformance harness pattern (BeforeEach + ephemeral schema) from day 1.** The `writeClusterFileToTemp` pattern is well-established and I should have copied it directly into the BeforeEach without reinventing.
4. **Counter-intuitively: maybe LESS scenario count, more depth per scenario.** 16 scenarios at ~7 tests/scenario is shallow per shape. A future shift focused on a single hairy area (e.g. CTE or in-list semantics) at 20+ tests/scenario would surface more divergences than I caught here.
