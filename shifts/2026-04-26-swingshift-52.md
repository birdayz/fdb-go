### Engine fixes that came out of running the corpus

- **Identifier-case normalization at JDBC boundary** (`staticRows.Columns()` uppercases unquoted identifiers; `id` ‚Üí `ID`).
- **Qualified column stripping** (`t.id` ‚Üí `ID`, `u.name` ‚Üí `NAME`).
- **Anonymous-projection naming `_N`** parity with Java (was emitting parser's whitespace-stripped expression text ‚Äî `nameISNULL` etc.). Fixed via `$expr_N` internal sentinel that defeats `isSimpleIdentifier` so `jdbcColumnName` falls through to `_<position>` synthesis.
- **JDBC type-name plumbing**: embedded driver implements `RowsColumnTypeDatabaseTypeName`. `staticRows.colTypes` populated from proto FieldDescriptor. Aggregate columns default to BIGINT (covers SUM/MIN/MAX/COUNT over integral types).
- **Empty result-set type inference** ‚Äî JDBC type-name no longer dependent on first-row value inspection.
- **UUID column type ‚Äî full end-to-end**: DDL parser accepts `key UUID`; metadata builder emits `tuple_fields.UUID` proto sub-message; `CAST(string AS UUID)` validates and returns canonical 36-char form; `ConvertToProtoValue` encodes via `most_significant_bits` / `least_significant_bits` (big-endian from 16-byte UUID); `ProtoValueToDriver` reverses on read; `jdbcTypeNameForFD` reports `OTHER` (matches `java.sql.Types.OTHER` for UUID).
- **Engine-side type inference for arithmetic / CAST / CASE / scalar functions**: `inferProjectionJDBCType` walks the projection-expression AST and returns the JDBC type name for `x + y`, `id + 10`, `CAST(... AS UUID)`, `CASE WHEN ... THEN 'low' END`, `COALESCE(a, b)`, `UPPER(s)`, etc. Uses the `jdbcTypeMax` lattice (DOUBLE > FLOAT > BIGINT > INTEGER for numerics, identity for non-numeric same-type pairs). Plumbed through JOIN executor (per-source type map) and CTE-scan executor (`cte.colTypes` propagated).
- **Bare-bool-FieldValue WHERE rejection** in `evalComparisonPredicateTri` to match Java strictness (`WHERE flag` rejected; must be `WHERE flag = TRUE`).
- **UNION colTypes propagation**: was discarded, now `leftColTypes` propagated through.

### Cross-engine state-leak investigation (the 5-whys)

The corpus initially exhibited intermittent `//conformance:conformance_test` hangs (6 unrelated specs in `error_conformance` / `unique_violation` wedged at 120s `Client.Timeout`). Bisection by N (entry count):

- N=125: PASS
- N=126: PASS
- N=127 (`integer_overflow_on_cast` at #127): HANG
- 30√ó `integer_overflow_on_cast` alone: 0‚Äì9 fast, 10 hangs at 30s
- 126 entries (no overflow) + INSERT-MISSING: PASS
- 126 entries + overflow at any position + INSERT-MISSING: HANG

**Root cause**: fdb-relational 4.11.1.0's error-path teardown leaks state across schema/database drops. Setup-time INSERT errors compound this ‚Äî ~11 entries in a single Java-server lifetime is the cliff. Combined with state already accumulated by prior conformance specs (existence_check_*, error_*), per-request Java latency wedges past 30s.

**Fix** (commit f86aa521):
1. Cross-engine corpus spec uses `NewIsolatedJavaInvoker()` ‚Äî its own pristine Java server, deferred Close. Shared singleton stays clean across the conformance suite.
2. Drop three setup-INSERT-error corpus entries: `integer_overflow_on_cast`, `division_by_zero`, `duplicate_pk_insert`. Go-side coverage for these lives in `pkg/relational/sqldriver` integration tests; only the cross-engine pin is missing.
3. `-Xmx2g` on `conformance_server` (already done earlier in the shift) keeps GC pressure from compounding.

**Verification**: //conformance:conformance_test 447/447 green in 96s (down from 814s with 6 timeouts), forced with `--cache_test_results=no`.

## Tech debt / known issues discovered

1. **Three corpus entries can't be cross-engine pinned today**: `integer_overflow_on_cast`, `division_by_zero`, `duplicate_pk_insert` ‚Äî all hit fdb-relational 4.11.1.0's setup-INSERT-error state-leak. Go-side coverage is in `pkg/relational/sqldriver` integration tests. Re-add when fdb-relational fixes the leak (NOT a Go-side fix).
2. **`NOT IN (..., NULL, ...)`**: one-sided divergence (Java rejects, Go propagates UNKNOWN per SQL standard). Not expressible in `ExpectErrorContains` (would need both engines to fail) or as a positive entry (Java rejects). Documented as gotcha; re-add when one side aligns.
3. **`<<` / `>>` bit-shift**: tokenized but unimplemented in fdb-relational. Same one-sided shape; documented.
4. **`IS TRUE` / `IS FALSE` on BOOLEAN**: rejected by fdb-relational planner; Go accepts. Same one-sided shape; documented.
5. **Multi-column ORDER BY, GROUP BY, DISTINCT, LIMIT, scalar functions other than COALESCE/IFNULL/UPPER/LOWER/SUBSTRING/etc.**: unsupported by fdb-relational 4.11.1.0's Cascades planner. Wait until upstream ports the relevant rules (RFC-022 ¬ß4.5 Batch B).
6. **INFORMATION_SCHEMA**: GO-side only. fdb-relational 4.11.1.0's parser rejects the schema-qualified reference. **Track A4** (system-table byte-equivalence) is BLOCKED on upstream ‚Äî Go-side enforcement unilaterally would create divergence.
