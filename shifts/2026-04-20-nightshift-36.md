### 1. The harness — `pkg/relational/conformance/yamsql/`

Four source files, 38 seed scenarios (~160 assertions), all passing against a real FDB container.

- `yamsql.go` — loads a scenario from a YAML file. Fields: `name` (optional, defaults to basename), `schema_template` (CREATE TABLE body), `setup` (list of INSERT statements), `tests` (list of `{query, rows, unordered?, error_code?}`).
- `runner.go` — sets up a fresh database + schema template per scenario, routes queries through `db.QueryContext` (SELECT/WITH/VALUES via `isQuery()` helper) and DDL/DML through `db.ExecContext`, tears down on exit. `runErrorTest` handles `error_code`-tagged cases by asserting the error `errors.As` to `*api.Error` with a matching SQLSTATE.
- `diff.go` — row comparison. Normalises int/uint/float across widths (`int64` / `float64`), handles `[]byte`, supports `unordered: true` via stable sort on type-tagged row keys. Deliberate type-tag keeps `int64(5)` and `string("5")` distinct (matching `rowKey`/`GroupByKey` encoding).
- `runner_test.go` — `TestYamsqlConformance` walks `testdata/*.yaml`, each file runs as a subtest in parallel with a unique DB path + template name.

**Build wiring:** `go_test` target has `tags = ["manual"]` so `bazelisk test //...` does NOT run it. Use `just conformance-sql` (new recipe) to run locally. Reasoning: CLAUDE.md already flags 9-target parallel FDB-container load as a CI risk, and the harness is a targeted tool not a blanket smoke test.

**Scenario format** — deliberately simpler than Apple's yamsql:
- YAML flow sequences (`[1, true]`) for rows instead of yamsql's `{1, true}` flow-map shorthand (which is ambiguous — bare items without keys are treated as `key: null` by standard YAML parsers; adopting it would have needed a custom constructor).
- `~` or `null` for SQL NULL.
- Explicit `unordered: true` (yamsql has both `result` and `unorderedResult`).
- `error_code: "22023"` pins the expected SQLSTATE for error paths.

The format trade-off was worth the simpler loader. Later we can port yamsql files by translating their result blocks.

### 2. Real Java divergences caught and fixed

**(a) `BIGINT / BIGINT` returned FLOAT** — commit `b53afd51`.

The handover claimed int64 preservation, but `applyMathOp` promoted both operands to float64 first, so `10 / 3 = 3.333...` instead of Java's `DIV_LL` integer division (`10 / 3 = 3`). Removed the "preserve int64 if whole-result" post-hoc cast (which produced non-deterministic types across inputs — `10.0 + 3.0` → int64 vs `10.0 + 3.5` → float64) and fast-pathed `int64 op int64` through Go's native operators. Same for MOD_LL.

**(b) Integer overflow silently wrapped** — commit `3c5496d8`.

Java's `ADD_LL/SUB_LL/MUL_LL` are `Math.addExact/subtractExact/multiplyExact` which throw. Our int64 fast-path used `+`/`-`/`*` which wrap. Added `addInt64Checked`/`subInt64Checked`/`mulInt64Checked` with Hacker's-Delight sign-bit and divide-back tests. Also guard `MinInt64 / -1` (abs value overflows). Returns SQLSTATE `22023` on overflow (closest existing Go code; see follow-up below). Unit-tested against boundary values; pinned by `overflow.yaml`.

**(d) INSERT duplicate PRIMARY KEY silently replaced** — commit `e3cb8e86`.

Plain `store.SaveRecord(msg)` overwrites existing records at the same PK — that's record-layer semantics. For SQL INSERT it's wrong: Java raises SQLSTATE 23505. Switched both INSERT paths (`execInsert`, `execInsertSelect`) to `SaveRecordWithOptions(msg, recordlayer.RecordExistenceCheckErrorIfExists)`; the resulting `*RecordAlreadyExistsError` flows through `wrapSaveRecordError` to SQLSTATE 23505. `execUpdate` keeps plain `SaveRecord` since UPDATE legitimately overwrites. `unique_violation.yaml` extended to pin that a failed duplicate INSERT leaves the original row intact.

**(c) `store.SaveRecord` errors not wrapped with SQLSTATE** — commit `71a0baf2`.

`RecordIndexUniquenessViolationError`, `RecordAlreadyExistsError`, `IndexKeySizeError`, `IndexValueSizeError` leaked through the SQL layer unwrapped. Callers using `errors.As` against `*api.Error` missed the class-23 / class-22 codes that Java's `RelationalException` translator provides. Added `wrapSaveRecordError()` — applied at all three INSERT paths (`execInsertValues`, `execInsertSelect`, `execInsertSelectWhere`). Later extended to `execUpdate` too — commit `7e2e583a`, after round-3 review caught the UPDATE path would still leak raw recordlayer types. Already-wrapped `*api.Error` values pass through untouched.

**(e) `SELECT tbl.*` qualified-star unimplemented** — commit `7abddbd0`.

Parser accepted `SELECT lb.*` but evaluation errored with "unsupported SELECT element type SelectQualifierStarElementContext". Common yamsql idiom — blocked porting whole scenario classes. Handled the single-source FROM case by treating qualifier-dot-star as SELECT *. Multi-source FROM (JOIN / comma-join) still needs per-source column filtering to correctly restrict the output — that's TODO'd. Pinned via `boolean.yaml` which now uses `SELECT lb.*` matching Java's boolean.yamsql.

**(f) Explicit `NULLS FIRST` / `NULLS LAST` ignored** — commit `ace2b515`.

Grammar parsed the clause but `extractOrderBy` didn't carry it through — users could type `ORDER BY v DESC NULLS FIRST` and silently get the default `NULLS LAST`. Added `nullsFirst *bool` override to `orderByClause` and a centralised `orderByLess(a, b, ob)` helper so all three sort sites (single-table, aggregate, CTE/JOIN) go through one NULL+direction rule rather than three inlined copies. `order_by_nulls.yaml` grew ASC-NULLS-LAST and DESC-NULLS-FIRST cases.

**(g) Multi-CTE cross-join dropped rhs** — commit `be01891a`.

`execSelectQuery` short-circuited to `execSelectFromCTE` whenever the primary source resolved to a CTE — but that path materialises only one CTE's rows and never looked at `sq.joins`. So `SELECT COUNT(*) FROM lo, hi` with both as CTEs silently dropped `hi` and returned `|lo|` instead of `|lo|*|hi|`. Fixed by gating the short-circuit on `len(sq.joins) == 0`; with joins we fall through to `execSelectJoin`, whose `scanTableToMaps` already resolves CTE names. `cte.yaml` pins the cross-join product now.

**(h) Arithmetic overflow SQLSTATE 22023 → 22003** — commit `f65747a1`.

Added `ErrCodeNumericValueOutOfRange = "22003"` (SQL-standard NUMERIC_VALUE_OUT_OF_RANGE) and switched three overflow sites from 22023 to 22003: `applyMathOp` integer ADD/SUB/MUL/DIV, `ABS(MinInt64)`, `CAST(float_too_big AS BIGINT)`. Division-by-zero stays 22023 since Java's `ErrorCode` enum has no 22012 either. Java's `fdb-relational-core` doesn't catch `ArithmeticException` explicitly — it propagates as an unchecked exception with no SQLSTATE — so we're strictly more informative than Java here (class-22 code on the wire vs RuntimeException leak).

**(i) Parenthesised boolean expressions in WHERE / HAVING / JOIN** — commits `913c4d5b`, `bf039225`, `485436bb`.

Grammar parsed `WHERE NOT (x = y)` as a `RecordConstructorExpressionAtom` with one unnamed field, rejected by both evaluators with "unsupported expression atom". swingshift-35 workaround was to rely on precedence (`WHERE NOT x = y`) but that doesn't nest. Added unwrap case across three evaluators: `evalExprAtom` (proto WHERE), `evalExprAtomOnMap` (JOIN WHERE / ON + CTE WHERE), and `evalHavingTri` (HAVING). Crucially, boolean inners route through the appropriate tri-state predicate evaluator (`evalExprPredicateTri` on the proto path, `evalPredicateOnMapExprTri` on the map path) so NULL comparisons propagate as tri-state NULL (encoded `nil` when returned to the value evaluator) rather than collapsing to FALSE — without that, `NOT (b = NULL)` would wrongly flip UNKNOWN to TRUE. Multi-field and named constructors still error with a clearer message. Closed a TODO tracked across multiple shifts.

**(j) CAST failures use 22F3H INVALID_CAST** — commit `164ba7c0`.

Reviewer note during round-6 audit pointed at Java's `CastValue.java`: it uses `SemanticException.ErrorCode.INVALID_CAST` (SQLSTATE 22F3H) uniformly for ALL CAST failures — range overflow, NaN/Infinity, string-parse errors. My earlier commit `f65747a1` used `22003` (NUMERIC_VALUE_OUT_OF_RANGE) for CAST float-to-int range overflow — wrong for Java-conformance. Fixed four CAST sites to use `ErrCodeInvalidCast` (22F3H): float→int range, NaN/Infinity→int, string→int parse, string→float parse. Arithmetic overflow (`applyMathOp`, `ABS`) keeps 22003 — Java has no dedicated code there, 22003 is the SQL-standard class-22 entry, and semantically we're being more informative than Java (which leaks the raw `ArithmeticException` with no SQLSTATE).

**(k) Map-path paren NULL collapse** — commit `485436bb`.

Round-6 reviewer caught: the map-path `evalExprAtomOnMap` paren unwrap case didn't mirror the proto-path's tri-state routing; it fell through to `evalExprOnMap` which collapses NULL comparisons to FALSE at the value-evaluator boundary. So a JOIN `WHERE NOT (b = NULL)` returned TRUE instead of UNKNOWN — every row wrongly included. Fixed by adding the same `looksBoolean` check + `evalPredicateOnMapExprTri` routing used in the proto path; `triNull` encodes as `nil`. Pinned by a new cross-join test in `join_null_key.yaml`.

**(l) RIGHT OUTER JOIN NULL-padding for qualified columns** — commit `4f5329e2`.

Caught when I extended `outer_join.yaml` to exercise an unmatched right row per a reviewer coverage request. Pre-fix: when b had a row whose join key didn't match anything in a, the merger copied only the right-side keys into the combined map, so `SELECT a.id` fell through to the unqualified `'id'` which was right's id — returning b's id instead of NULL.

Fix: sample a left-side row's keys at materialisation time. For each left key, if that key doesn't also exist on the right, explicitly set `combined[k] = nil`. Skipping right-present keys preserves the intended ambiguous-unqualified behaviour (unqualified `id` takes right's value, which is consistent across the matched and unmatched rows).

### 2a. Feature added (not a bugfix but user-visible)

**BYTES literals** — commit `b90ed93d`.

Grammar parsed `x'deadbeef'` / `X'deadbeef'` / `b64'...'` bytes literals, but `evalConstant` errored with "unsupported constant type BytesConstantContext". Ported from yamsql `bytes.yamsql`. Fixes: pure-Go hex decode (odd-length / non-hex raises `22F03 INVALID_BINARY_REPRESENTATION`), `encoding/base64.StdEncoding.Strict()` for base64 (matches Java's `Base64.getDecoder()`), and `convertToProtoValue` BytesKind now accepts `[]byte` directly (was string-only). Five tests from `bytes.yaml` pin equality, IN, NULL filtering.

## Known issues / tech debt (surfaced this shift, some fixed)

- **Multi-source `SELECT tbl.*` qualified-star** (MEDIUM) — single-source FROM supported this shift; multi-source (JOIN / comma-join) still projects all columns instead of restricting to the aliased table. Needs per-source column-list plumbing. Same item calls out that wrong-alias `foo.*` also silently succeeds — no qualifier validation.
- **RIGHT OUTER JOIN empty-left-table edge case** (LOW) — fixed for the common case this shift, but when both `leftRows` and `joined` are empty (left table entirely empty), qualified `a.*` keys can't be NULLed because there's nothing to sample. Theoretical; no scenario reproduces it yet.
- **INFORMATION_SCHEMA query gaps** (MEDIUM) — tried to pin system-table scenarios this shift but hit two issues: parser rejects `FROM INFORMATION_SCHEMA.TABLES` (reserved keyword), and `WHERE TABLE_NAME = 'EMP'` returns 0 rows due to case-mismatch in the system-tables handler.
- **`CONCAT` NULL semantics** (LOW) — current behaviour skips NULL args (MySQL/Postgres ≥ 9). SQL standard / Oracle / SQL Server propagate NULL. Unclear what Java does; pinned current behaviour as-is, scenario comment flags the follow-up.
- **Diff.go latent sharp edges** (LOW) — `uint64 → int64` wrap and `float64 ↔ int64` precision at values > 2^53. Not reachable from fdbsql today, not gating; add defensive comments if anyone ports the runner.
