### 1. The harness — `pkg/relational/conformance/yamsql/`

Four source files, 20 seed scenarios, all passing against a real FDB container.

- `yamsql.go` — loads a scenario from a YAML file. Fields: `name` (optional, defaults to basename), `schema_template` (CREATE TABLE body), `setup` (list of INSERT statements), `tests` (list of `{query, rows, unordered?, error_code?}`).
- `runner.go` — sets up a fresh database + schema template per scenario, routes queries through `db.QueryContext` (SELECT/WITH/VALUES via `isQuery()` helper) and DDL/DML through `db.ExecContext`, tears down on exit. `runErrorTest` handles `error_code`-tagged cases by asserting the error `errors.As` to `*api.Error` with a matching SQLSTATE.
- `diff.go` — row comparison. Normalises int/uint/float across widths (`int64` / `float64`), handles `[]byte`, supports `unordered: true` via stable sort on type-tagged row keys. Deliberate type-tag keeps `int64(5)` and `string("5")` distinct (matching `rowKey`/`GroupByKey` encoding).
- `runner_test.go` — `TestYamsqlConformance` walks `testdata/*.yaml`, each file runs as a subtest in parallel with a unique DB path + template name.

**Build wiring:** `go_test` target has `tags = ["manual"]` so `bazelisk test //...` does NOT run it. Use `just conformance-sql` (new recipe) to run locally. Reasoning: CLAUDE.md already flags 9-target parallel FDB-container load as a CI risk, and the harness is a targeted tool not a blanket smoke test.

**Scenario format** — deliberately simpler than Apple's yamsql:
- YAML flow sequences (`[1, true]`) for rows instead of yamsql's `{1, true}` flow-map shorthand (which is ambiguous — bare items without keys are treated as `key: null` by standard YAML parsers; adopting it would have needed a custom constructor).
- `~` or `null` for SQL NULL.
- Explicit `unordered: true` (yamsql has both `result` and `unorderedResult`).
- `error_code: "22023"` pins the expected SQLSTATE for error paths.

The format trade-off was worth the simpler loader. Later we can port yamsql files by translating their result blocks.

### 2. Real Java divergences caught and fixed

**(a) `BIGINT / BIGINT` returned FLOAT** — commit `b53afd51`.

The handover claimed int64 preservation, but `applyMathOp` promoted both operands to float64 first, so `10 / 3 = 3.333...` instead of Java's `DIV_LL` integer division (`10 / 3 = 3`). Removed the "preserve int64 if whole-result" post-hoc cast (which produced non-deterministic types across inputs — `10.0 + 3.0` → int64 vs `10.0 + 3.5` → float64) and fast-pathed `int64 op int64` through Go's native operators. Same for MOD_LL.

**(b) Integer overflow silently wrapped** — commit `3c5496d8`.

Java's `ADD_LL/SUB_LL/MUL_LL` are `Math.addExact/subtractExact/multiplyExact` which throw. Our int64 fast-path used `+`/`-`/`*` which wrap. Added `addInt64Checked`/`subInt64Checked`/`mulInt64Checked` with Hacker's-Delight sign-bit and divide-back tests. Also guard `MinInt64 / -1` (abs value overflows). Returns SQLSTATE `22023` on overflow (closest existing Go code; see follow-up below). Unit-tested against boundary values; pinned by `overflow.yaml`.

**(d) INSERT duplicate PRIMARY KEY silently replaced** — commit `e3cb8e86`.

Plain `store.SaveRecord(msg)` overwrites existing records at the same PK — that's record-layer semantics. For SQL INSERT it's wrong: Java raises SQLSTATE 23505. Switched both INSERT paths (`execInsert`, `execInsertSelect`) to `SaveRecordWithOptions(msg, recordlayer.RecordExistenceCheckErrorIfExists)`; the resulting `*RecordAlreadyExistsError` flows through `wrapSaveRecordError` to SQLSTATE 23505. `execUpdate` keeps plain `SaveRecord` since UPDATE legitimately overwrites. `unique_violation.yaml` extended to pin that a failed duplicate INSERT leaves the original row intact.

**(c) `store.SaveRecord` errors not wrapped with SQLSTATE** — commit `71a0baf2`.

`RecordIndexUniquenessViolationError`, `RecordAlreadyExistsError`, `IndexKeySizeError`, `IndexValueSizeError` leaked through the SQL layer unwrapped. Callers using `errors.As` against `*api.Error` missed the class-23 / class-22 codes that Java's `RelationalException` translator provides. Added `wrapSaveRecordError()` — applied at all three INSERT paths (`execInsertValues`, `execInsertSelect`, `execInsertSelectWhere`). Later extended to `execUpdate` too — commit `7e2e583a`, after round-3 review caught the UPDATE path would still leak raw recordlayer types. Already-wrapped `*api.Error` values pass through untouched.

**(e) `SELECT tbl.*` qualified-star unimplemented** — commit `7abddbd0`.

Parser accepted `SELECT lb.*` but evaluation errored with "unsupported SELECT element type SelectQualifierStarElementContext". Common yamsql idiom — blocked porting whole scenario classes. Handled the single-source FROM case by treating qualifier-dot-star as SELECT *. Multi-source FROM (JOIN / comma-join) still needs per-source column filtering to correctly restrict the output — that's TODO'd. Pinned via `boolean.yaml` which now uses `SELECT lb.*` matching Java's boolean.yamsql.

**(f) Explicit `NULLS FIRST` / `NULLS LAST` ignored** — commit `ace2b515`.

Grammar parsed the clause but `extractOrderBy` didn't carry it through — users could type `ORDER BY v DESC NULLS FIRST` and silently get the default `NULLS LAST`. Added `nullsFirst *bool` override to `orderByClause` and a centralised `orderByLess(a, b, ob)` helper so all three sort sites (single-table, aggregate, CTE/JOIN) go through one NULL+direction rule rather than three inlined copies. `order_by_nulls.yaml` grew ASC-NULLS-LAST and DESC-NULLS-FIRST cases.

## Shift working notes

- **Harness ROI**: within the first hour of adding scenarios it had already surfaced two real correctness bugs (integer division, missing SQLSTATE wrap) and confirmed one gap (integer overflow). All three fixable in < 40 lines each. This matches what the swingshift-35 memo predicted. Adding more scenarios is strictly dominant over adding more unit tests.
- **Scenario-format decision**: rejected Apple's yamsql `{v1, v2}` flow-map shorthand because standard YAML parses it as `{v1: null, v2: null}`, making the format ambiguous without a custom constructor. Using YAML flow sequences `[v1, v2]` instead means the format round-trips through any YAML parser cleanly. Translating a yamsql file to ours is mechanical (`{` → `[`, handle `!null` → `~`).
- **One large task at a time** was respected. No parallel subagents on implementation — the scope was tight enough that reviewing my own code between commits was the right cadence.
- **Review feedback was productive**: the @manual tag wasn't merely a nit; if left off, adding more conformance scenarios later would put additional parallel FDB containers in the default test pool, hitting the 9-container parallelism concern documented in CLAUDE.md. The comment on the `mulInt64Checked` redundant case is also worth keeping — made me re-verify which case is actually load-bearing for safety vs defensive.
- **Known handover gap**: review didn't cover the post-second-push commits (unique-violation wrap, AVG pin, review-nits commit). Worth a fresh review round on next shift.
