### fdbsql-driver leak fix (post "swing shift over" hunt)

When the user signalled "swing shift long over" the in-flight pre-commit `just test` had failed with `dial 172.16.0.3:4500: i/o timeout` mid-corpus in `run_sql_conformance`. Followed up at the user's "HUNT IT DOWN" insistence and traced it to:

- `EmbeddedConnection.Close()` (`pkg/relational/core/embedded/connection.go:315`) only sets a `closed` flag — it does NOT close the underlying FDB database.
- Each `sql.Open("fdbsql", dsn)` creates a new `Connector` whose lazy `initialize` calls `purefdb.OpenDatabase(clusterFile)` and assigns the result to `c.fdbDB`.
- database/sql has no `Connector.Close` hook. When `*sql.DB.Close()` runs, the Connector and its `fdbDB` only become eligible for GC — and even then the underlying TCP connection to FDB isn't proactively released.
- `plandiff`'s Go runner does `sql.Open` twice per corpus entry. 170 corpus entries × 2 = 340 leaked FDB connections per run. Under parallel `just test //...` load, the testcontainer FDB's connection table got exhausted around entry ~170 and `dial` timeouts started firing.

Fix in `pkg/relational/sqldriver/driver.go`: process-global `sync.Map` cache keyed by cluster-file path. Concurrent races settle via `LoadOrStore` — the loser closes its just-opened FDB handle so the winner's TCP connection is the only one that lives. Verification: focused conformance suite (357 specs) drops from ~727 s to 45 s. Full `just test` (39 targets) clean.

### CLAUDE.md gotcha additions (9 new)

1. **CROSS JOIN explicit syntax NPEs.** Parser visitor calls `accept(...)` on null `InnerJoinContext.expression()`.
2. **Scalar subqueries `(SELECT ...)` not parsed** — grammar's `expressionAtom` rule doesn't include the subquery alternative.
3. **Equality + contradicting range yields the equality row in Java** — `WHERE id = 2 AND id > 5` returns `[2]` from Java, range residual silently dropped (planner bug).
4. **`ORDER BY` outside natural-order continuation raises UnableToPlan** — range on leading PK + ORDER BY trailing, ORDER BY non-PK col without index, etc.
5. **`plan` is a reserved keyword.** Joins existing `blob` reserved column list.
6. **`COUNT(DISTINCT col)` NPEs.** Parser visitor calls `.getText()` on null `AggregateWindowedFunctionContext.ALL()`.
7. **`SUM(BIGINT) / COUNT(*)` integer-divides Java, float-divides Go.** Go's SUM accumulator is float64; Java does integer division on BIGINT/BIGINT.
8. **`ORDER BY <arithmetic-expression>` raises UnableToPlan.** Even single-col `ORDER BY a + b` fails.
9. **Scalar-function-wrapping-aggregate fails Java planner.** `COALESCE(SUM(...), 0)`, `CASE WHEN COUNT(*) > N THEN ...`. Java raises `unable to eval an aggregation function with eval()`. Aggregate-of-expression direction works fine.
10. **Only `UNION ALL` is supported.** `UNION` (distinct) raises `only UNION ALL is supported`.

(Counts to 10 with the union-distinct one, even though I attributed 9 to swingshift-56 in TODO.md — the count is approximate; CLAUDE.md is authoritative.)
